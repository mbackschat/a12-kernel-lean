# External kernel evidence

This document owns the portable evidence boundary between the external A12 kernel harness and the Lean theory. It describes what is retained, what `lake test` checks, and which correspondence claims the artifacts do and do not support.

## Roles and provenance

[`../evidence/kernel-30.8.1/`](../evidence/kernel-30.8.1/) contains 42 focused, own-domain observations and their standalone DM-JSON models: 36 runtime conformance cases and six kernel-confirmed static authoring observations. They were captured outside this repository through a12-dmkits' adapter boundary in the local `a12-rulekit/` checkout. The original 17 runtime cases use [`CorpusCapture`](../../a12-rulekit/adapter/src/test/kotlin/io/github/mbackschat/a12/dm/adapter/laws/CorpusCapture.kt): it records the Groovy-dynamic kernel result as the anchor, rejects a case if the clean-room interpreter disagrees, executes the static-Java kernel strategy, and records any strategy split. Those cases have no recorded divergence, so both kernel strategies and the interpreter agreed at capture time.

The seven iteration cases use the same adapter engine boundary but were captured directly through [`CorpusEngines`](../../a12-rulekit/adapter/src/test/kotlin/io/github/mbackschat/a12/dm/adapter/laws/CorpusEngines.kt), because one high-value malformed-filter witness correctly exposed an interpreter disagreement that `CorpusCapture` is designed to block. All seven have identical Groovy-dynamic and static-Java kernel results. Six also agree with the a12-dmkits interpreter; [`having-malformed-filter-drops.json`](../evidence/kernel-30.8.1/cases/iteration/having-malformed-filter-drops.json) does not, and the complete three-result split is retained separately in [`triangulation/having-malformed-filter-drops.json`](../evidence/kernel-30.8.1/triangulation/having-malformed-filter-drops.json). The interpreter result is not encoded as a standard `divergences` entry because that field is reserved for kernel execution strategies. Static authoring observations are kernel-confirmed model-validation results and are not runtime-strategy comparisons.

The 12 captured-outer correlation cases use the adapter engine boundary across seven standalone models. Groovy-dynamic, static-Java, and the a12-dmkits interpreter agree on all 12. The earlier seven observations cover clean-filled direction/self-match/self-exclusion and the all-valid plus symmetric malformed-consumer footprints; five additional observations cover empty and malformed local filter operands, numeric inequality, repetition equality, and repetition ordering. The cases retain complete canonically sorted message-signature lists, including formal errors outside the focused authored code, and each referenced model carries the actual executed stored `errorCondition` containing `$`; this is intentional because a formatter/conversion path has previously lost a `$`-bearing conjunct. The canonical list order comes from a12-dmkits' `CorpusEngines` normalization and is not claimed to be kernel emission order. The typed correlation projection describes origins structurally but never substitutes for preserving that raw executed condition.

Four static observations exercise the checked one-star correlation authoring route through `KernelAdapter.diagnose`: all-outer `Having` is rejected with `MVK_NO_ITERATION_FOR_WILDCARD`, unequal-scale numeric `==` is rejected with `MVK_INVALID_COMPARE_DEC_PLACES`, the same unequal-scale operands under `<` are accepted with no diagnostic, and an inner field from sibling repeatable `Supply` while starring `Demand` is rejected with `MVK_INVALID_ITERATION_IN_FILTER_CONDITION`. The exact seeded [`order`](../evidence/kernel-30.8.1/models/lean-correlation-elaboration-order.json) and [`staffing`](../evidence/kernel-30.8.1/models/lean-correlation-elaboration-staffing.json) model snapshots are retained alongside the complete candidate drafts and structured diagnostics, and their full-file SHA-256 identities are pinned in the projection. These are four focused authoring observations, not a general diagnostic matrix.

The runtime cases use a12-dmkits' portable conformance shape: pinned kernel version, model reference, ordered document placements, operation, complete observed message signatures, and optional divergence records. The two earlier diagnostic artifacts retain the pinned version, complete draft, expected code, and full structured kernel diagnostic list. The four correlation-elaboration artifacts deliberately contain no `expectedCode`; replay derives the entire ordered code list from `diagnostics[].code`, including the empty list for the accepted `<` control. The models and full outputs are retained rather than replacing them with Lean-shaped expected values. This repository contains no kernel binary, binding, or runtime dependency.

Capture runs the sibling adapter harness only from an already ignored disposable workspace under `../a12-rulekit/build/`, after checking that path with `git check-ignore` and recording the sibling's visible status. The disposable harness calls [`CorpusEngines`](../../a12-rulekit/adapter/src/test/kotlin/io/github/mbackschat/a12/dm/adapter/laws/CorpusEngines.kt) directly to collect Groovy-dynamic, static-Java, and interpreter results; [`CorpusCapture`](../../a12-rulekit/adapter/src/test/kotlin/io/github/mbackschat/a12/dm/adapter/laws/CorpusCapture.kt) is used when interpreter agreement is a required capture precondition. The live [`CorpusGenerator`](../../a12-rulekit/adapter/src/test/kotlin/io/github/mbackschat/a12/dm/adapter/laws/CorpusGenerator.kt) is not run because it rewrites tracked a12-dmkits corpus files. Only the portable model and observation artifacts are copied into this repository; the disposable source and outputs are removed, and the sibling's tracked and visible-untracked status must be unchanged afterward.

## Why replay uses a focused projection

The portable a12-dmkits case format deliberately starts from full DM-JSON and stored condition text. The current Lean capsule starts later: expanded field declarations, classified scalar cells, and structured parser-independent conditions. Consuming the full corpus directly would force a premature DM-JSON loader and bilingual condition parser into a capsule whose semantics intentionally begin after those boundaries.

[`projection.json`](../evidence/kernel-30.8.1/projection.json) therefore carries only the closed input projection needed by the flat/path/required fragment: field declarations, structured paths/conditions, classified cells, row-content eligibility, and operation kind. [`iteration-projection.json`](../evidence/kernel-30.8.1/iteration-projection.json) is deliberately separate and carries one repeatable group, ordered row IDs, row-indexed numeric cells, an optional row-local equality filter, and the narrow selected-sum comparison. [`correlation-projection.json`](../evidence/kernel-30.8.1/correlation-projection.json) schema v2 carries the second one-group fragment: 1-based ordered row IDs, numeric field IDs and paths, a closed filter with explicit inner/outer origins, explicit guard and starred-consumer field IDs, the guarded any-filled observer, and the outer-row-to-pointer map. [`correlation-elaboration-projection.json`](../evidence/kernel-30.8.1/correlation-elaboration-projection.json) is a fourth closed projection containing only repeatable-group declarations, Number-field metadata, group-qualified relative field and one-star references, inner/outer origins, the admitted `equal`/`lessThan` comparison tags, and the referenced external observation. None contains an expected Lean result or diagnostic code. Its closed JSON boundary is defined by [`CorrelationElaborationSchema.lean`](../A12Kernel/Evidence/CorrelationElaborationSchema.lean), and [`CorrelationElaborationReplay.lean`](../A12Kernel/Evidence/CorrelationElaborationReplay.lean) lowers it through the public checked elaborator.

For runtime correlation, the driver traverses the retained model's own group tree, requires unique projected group and Number-field paths/configurations and a unique rule carrying the focused code, resolves the rule's authored absolute-or-relative condition group/field references to the projected absolute paths, resolves its error entity to the separately projected guard field, canonically renders only the admitted guarded-any-filled/filter subset, and requires byte-exact equality with the actual stored `errorCondition`; this binds the typed filter to the exact authored-path rule the kernel executed without adding a general condition parser or repeatable elaborator. For static correlation elaboration, the driver first checks the retained model's full-file SHA-256 identity, then requires `content.modelConfig.conditionLanguage.code` to be `en_US`, requires the declaring and projected groups uniquely in that seeded model with canonical names and physical parents, checks repeatability and rejects a projected group with a repeatable proper ancestor, requires every projected field to have its canonical name and immediate physical parent plus exact Number kind/scale/signedness, binds the stored valid seed rule's identity/error field/code/severity/base condition, and requires the complete retained candidate draft identity to equal the draft rendered from the projection. Diagnostic provenance, severity, kernel source, and `where.rule` routing are checked before comparing complete code lists. The static renderer admits only conservative unquoted ASCII identifiers outside the full English lexer-keyword set; quoted authoring syntax is outside this transport. [`A12Kernel/EvidenceMain.lean`](../A12Kernel/EvidenceMain.lean) then derives focused firing membership or the complete diagnostic-code list from the external artifact, executes the corresponding typed projection through public Lean semantics/elaboration, and compares the result.

This is an observation-level bridge. In the flat fragment, a fired Lean verdict maps to its VALUE/OMISSION message signature; both `notFired` and `unknown` map to no authored message because the external validation output cannot distinguish them. In the uncorrelated iteration fragment, the current Lean result is truth `K`, so replay compares `K.tru` with focused authored-message presence. In the correlation fragment, replay derives firing membership from the retained canonically sorted focused signatures, reconstructs the expected row/pointer pairs in the projection's document-row order, and compares that list with Lean; it makes no claim about external kernel emission order. Both iteration projections retain the external VALUE/OMISSION signatures without claiming filtered-result polarity is modeled. Consequently the malformed cases support firing or suppression at the message boundary but do not by themselves prove that the kernel's hidden intermediate result is specifically Lean's `unknown`. In particular, the three-row malformed-local correlation witness proves observably that the malformed row is not kept and that healthy equal-key siblings continue to select one another; it does not expose the kernel's hidden truth value for the malformed comparison. That stronger internal treatment remains grounded in source/findings plus the Lean definition and its internal laws. The correlation rule/filter/path binding is gate-checked as described above; the projection's classified cells and row list are still a review-trusted transcription of the retained placements rather than a general DM-instance decoder, and that narrower transport boundary must remain visible in reviews.

The development [`flat-validation-empty-logic-v1` capability descriptor](../reference/flat-validation-empty-logic-v1.capability.json) is the checked projection-to-protocol bridge for eight selected flat cases. [`FlatProtocolBridge.lean`](../A12Kernel/Evidence/FlatProtocolBridge.lean) selects each typed replay input by retained case ID, derives the focused authored observation from the original message signatures, constructs a normalized request, requires that the public decoder returns the same checked replay input, evaluates that request through the public Lean reference, and classifies the resulting response by what the observation can expose. It generates the [evidence-derived request/response fixtures](../examples/reference-cli/flat-evidence/) and [`flat-validation-empty-logic-v1.conformance.json`](../reference/flat-validation-empty-logic-v1.conformance.json); `lake exe syncFlatHandover --check` rejects drift, while `lake exe syncFlatHandover --write` regenerates those derived files after an intentional change to their owned inputs. The same command also checks or writes the adjacent [mutation qualification plan](../reference/flat-validation-empty-logic-v1.mutation-plan.json); that file is source-maintainer testing metadata and contributes no kernel observation.

Four cases have a focused authored message and therefore externally establish the exact final firing and VALUE/OMISSION polarity: `number-empty-equals-zero-content`, `confirm-empty-not-true`, `healthy-or-malformed`, and `number-not-filled-empty-row`. The other four externally establish only authored silence: `number-empty-equals-zero-empty-row`, `boolean-empty-equals-true`, `malformed-number-equals-zero`, and `healthy-and-malformed`. Their exact normalized `NotFired` or `Unknown` verdict is deliberately sourced from the Lean runtime projection, because the retained validation output cannot distinguish those hidden alternatives. The capability and suite preserve this classification case by case instead of presenting all eight exact verdicts as directly kernel-observed.

The static correlation-elaboration replay maps exactly three externally established Lean rejection shapes: `missingInner` to `MVK_NO_ITERATION_FOR_WILDCARD`, `equalityScaleMismatch` to `MVK_INVALID_COMPARE_DEC_PLACES`, and inner-origin `fieldOutsideGroup` to `MVK_INVALID_ITERATION_IN_FILTER_CONDITION`. Outer-origin `fieldOutsideGroup` and every other elaboration result fail closed rather than being guessed. Successful elaboration maps to the empty diagnostic list, which locks the unequal-scale `<` control. In particular, `missingOuter` is not mapped because an all-inner `Having` is valid uncorrelated authoring and belongs on a different route.

The evidence transport and IO driver stay outside [`A12Kernel.lean`](../A12Kernel.lean), [`A12Kernel/Conformance.lean`](../A12Kernel/Conformance.lean), and [`A12Kernel/Proofs.lean`](../A12Kernel/Proofs.lean). The pure replay projection is not part of the trusted semantic theorem root; it is an empirical adequacy check over retained observations.

[`single-group-correlation-v1.conformance.json`](../reference/single-group-correlation-v1.conformance.json) is a downstream language-neutral index over this retained material, not a fifth evidence projection and not a new observation source. It manually cross-references each normalized request/response pair to the retained runtime or static case ID associated with its observable choice. `checkCandidateConformance` uses bounded duplicate-safe JSON parsing, enforces closed suite/case/evidence objects, pins a suite's compatibility identifiers plus finite-scope marker and retained case counts to selected support-manifest members, verifies those case IDs exist, and checks each case's explicit `externalSupports` and `expectedResponseSource` classification. The flat suite's requests, responses, evidence classifications, capability descriptor, and suite are mechanically produced and checked by the bridge above; the correlation suite still uses reviewed manual cross-references and therefore retains its separate projection-to-protocol integrity gap. The correlation suite includes all 12 runtime firing-membership cases and all four static authoring cases. Runtime rows declare external support for `firingRows`; static rejections declare support only for the elaboration rejection class while their public diagnostics are project-defined; the mixed-scale `<` case declares static acceptance only and labels its empty runtime answer as a Lean projection.

The flat descriptor and suite have status `developmentColdHandover`. One isolated Rust implementation has now exercised their finite knowledge-transfer boundary, but that process result creates no new kernel evidence: it does not close the wider flat operation for release or turn unexercised accepted inputs into kernel-backed claims. The exact cold-test outcome, remaining qualification work, and semantic/evidence limits are owned by [`IMPLEMENTER-KIT-FLAT-EMPTY-LOGIC.md`](IMPLEMENTER-KIT-FLAT-EMPTY-LOGIC.md).

All retained correlation candidates are contiguous one-based repetition indices and all retained parsed runtime Number cells are non-negative integers; the projection schema can transport signed integers, but the current cases do not exercise negative values, fractions, or scale-19 rounding dynamically. The normalized protocol therefore restricts candidates to non-empty contiguous `1..n` and machine-reports the row/value evidence boundary. Its general canonical exact-decimal and signed runtime behavior is Lean-account conformance with external evidence pending, not a kernel-backed release claim.

## Current coverage

The retained bundle covers these focused observable witnesses:

- empty Number comparison in a content-bearing row and its suppression by the all-empty row gate;
- empty Boolean and Confirm comparison asymmetry;
- `FieldNotFilled` eligibility on an otherwise empty row;
- malformed Number suppression and the `healthy Or malformed` / `healthy And malformed` branch behaviors;
- matching and mismatching parent-relative Number resolution plus a positive absolute path;
- bare-name declaring-group precedence, unique model-wide fallback, duplicate-name rejection, and flag-disabled rejection;
- empty, filled, and malformed absolute/non-repeatable required Number behavior.
- unfiltered `Sum == 17`, filtered `Sum == 7`, a one-cell mismatch, and no-match filtered-sum identity zero over one repeatable group;
- a malformed filter operand dropping its row, plus a paired malformed consumed cell that is harmless when dropped and suppresses the authored rule when kept;
- captured-outer direction: `[inner Count] < [$outer Count]` over `5, 6, 9` fires on outer rows 2 and 3;
- unexcluded equality self-match fires on every outer row;
- explicit `CurrentRepetition(inner) != CurrentRepetition($outer)` makes distinct values fire nowhere and duplicate `5, 5, 9` fire on outer rows 1 and 2;
- with a separate valid `Count` guard, duplicate `StockQty` keys, and starred `UnitWeight`, all-valid consumers fire on rows 1 and 2;
- a malformed row-1 `UnitWeight` is self-dropped for outer row 1 but kept for outer row 2, so only row 1 fires; moving the malformed consumer to row 2 mirrors the result so only row 2 fires.
- [`correlation-filter-empty-equals-zero.json`](../evidence/kernel-30.8.1/cases/correlation/correlation-filter-empty-equals-zero.json) holds valid guards and consumers fixed while an empty numeric filter key and a zero key compare equal, firing on outer rows 1 and 2;
- [`correlation-filter-malformed-local.json`](../evidence/kernel-30.8.1/cases/correlation/correlation-filter-malformed-local.json) gives row 1 a malformed local numeric key while rows 2 and 3 remain valid and equal, retaining the kernel formal-error signature `zahlHatDezimalTrenner|VALUE_ERROR|/Order[1]/Items[1]/StockQty` and firing on outer rows 2 and 3;
- [`correlation-number-not-equal.json`](../evidence/kernel-30.8.1/cases/correlation/correlation-number-not-equal.json) partitions numeric keys as `5, 5, 9` and fills only row 3's consumer, so inequality fires on outer rows 1 and 2;
- [`correlation-repetition-equal.json`](../evidence/kernel-30.8.1/cases/correlation/correlation-repetition-equal.json) fills only row 2's consumer, so repetition equality exposes self-selection by firing only on outer row 2;
- [`correlation-repetition-less-than.json`](../evidence/kernel-30.8.1/cases/correlation/correlation-repetition-less-than.json) fills every consumer, so inner repetition less than outer repetition exposes predecessor selection by firing on outer rows 2 and 3.
- [`correlation-elaboration-missing-inner-rejected.json`](../evidence/kernel-30.8.1/diagnostics/correlation-elaboration-missing-inner-rejected.json) retains the all-outer rejection `MVK_NO_ITERATION_FOR_WILDCARD`;
- [`correlation-elaboration-equality-scale-mismatch-rejected.json`](../evidence/kernel-30.8.1/diagnostics/correlation-elaboration-equality-scale-mismatch-rejected.json) and [`correlation-elaboration-ordering-mixed-scale-accepted.json`](../evidence/kernel-30.8.1/diagnostics/correlation-elaboration-ordering-mixed-scale-accepted.json) hold the two Number fields fixed and change only `==` to `<`, separating equality scale legality from ordering acceptance;
- [`correlation-elaboration-cross-group-inner-rejected.json`](../evidence/kernel-30.8.1/diagnostics/correlation-elaboration-cross-group-inner-rejected.json) retains `MVK_INVALID_ITERATION_IN_FILTER_CONDITION` when the inner reference comes from sibling repeatable `Supply` while the one-star consumer iterates `Demand`.

This closes the focused external gap for the earlier runtime clauses, the bare-name mechanism, capsule 5a's uncorrelated truth fragment, capsule 5b's clean-filled direction/self-match/self-exclusion shapes, capsule 5c's selected-consumer footprint witnesses, the five previously named correlation runtime gaps, and four narrow one-star authoring outcomes. The path cases corrected a real modeling error: bare names do not walk ancestors; they try the declaring group and then, when enabled, a model-wide unique short name. [`LF6`](LEAN-FINDINGS.md#lf6--bare-name-resolution-is-local-or-global-not-an-ancestor-walk) records that correction. The malformed uncorrelated-filter case exposed an a12-dmkits interpreter disagreement while both kernel strategies and Lean agree; [`LF7`](LEAN-FINDINGS.md#lf7--a-malformed-having-filter-drops-its-row-before-consumption) records its exact scope. The correlation evidence does not establish a universal comparison or authoring-diagnostic matrix, a general bilingual parser, hidden kernel truth states, general repeatable path elaboration, filtered-result polarity, nested or multi-star binding, or general consumers. The sibling-group rejection proves only the retained `Supply`-inside-`Demand*` shape; it does not establish that every `$` expression or every cross-group reference is invalid. Static observations for other rejected surface shapes and the legal arithmetic precision witness remain outside the current fragment.

## Running the gate

```sh
lake test
lake exe syncFlatHandover --check
```

Use `lake exe syncFlatHandover --write` only after intentionally changing an owned projection, typed capability descriptor, protocol mapping, expected Lean verdict, or typed mutation plan, then review the generated [capability descriptor](../reference/flat-validation-empty-logic-v1.capability.json), [suite](../reference/flat-validation-empty-logic-v1.conformance.json), [fixtures](../examples/reference-cli/flat-evidence/), and [mutation qualification plan](../reference/flat-validation-empty-logic-v1.mutation-plan.json) and rerun `--check`. The write mode does not create new kernel evidence; it reprojects retained observations and source-side testing metadata already in this repository.

Static snapshot binding uses `sha256sum` with a `shasum -a 256` fallback; at least one must be available when running `lake test`. This is an IO-only test dependency and is not linked into the Lean library or its semantics.

The test driver rejects an unsupported schema/kernel version, empty or duplicate case identities, zero/duplicate/undeclared correlation rows or cells, undeclared guard/consumer fields, invalid or duplicate projected entity paths, incomplete or duplicate outer-row pointer maps, correlated filters missing either origin, a retained-model digest mismatch, a missing/non-unique/moved/renamed retained group or Number field, nested repeatable ancestry outside the one-group projection, a non-English retained model, a mismatched focused or seed rule, condition binding, error/guard path, stored condition, static draft identity, capture provenance, or diagnostic routing, invalid projected models, unsafe, keyword-colliding, or otherwise non-renderable references, missing retained models, kernel-strategy divergences, malformed focused signatures, unmapped static elaboration errors, and result mismatches. Generic executable binding guards cover moved or renamed projected entities and structural hierarchy errors. Focused mutations of one real retained diagnostic fixture exercise rejection of digest, condition-language, nested-ancestry, complete-draft and seed-identity, capture-provenance, and diagnostic source/routing drift; every retained case independently crosses the positive binding gate. Additional focused examples lock renderer injection and keyword collisions, wrong guard routing, and unmapped `missingOuter`. A successful run reports the number of agreeing projections across all four projection bundles; the current gate reports `42/42`, comprising 36 runtime observations and six static authoring observations.

`lake build` remains the pure internal build and executable-example gate. `lake test` adds retained external correspondence evidence; [`scripts/check-lean-trust.sh`](../scripts/check-lean-trust.sh) independently audits the trusted theorem closure. None substitutes for the others.
