# External kernel evidence

This document owns the portable evidence boundary between the external A12 kernel harness and the Lean theory. It describes what is retained, what `lake test` checks, and which correspondence claims the artifacts do and do not support.

## Roles and provenance

[`../evidence/kernel-30.8.1/`](../evidence/kernel-30.8.1/) contains 70 legacy focused, own-domain observations and their standalone DM-JSON models—64 runtime conformance cases and six kernel-confirmed static authoring observations—plus the first complete source-maintained portable packet for five direct-cascade cases across three runners. They were captured outside this repository through a12-dmkits' adapter boundary in the local `../a12-rulekit/` checkout. The original 17 runtime cases use [`CorpusCapture`](../../a12-rulekit/adapter/src/test/kotlin/io/github/mbackschat/a12/dm/adapter/laws/CorpusCapture.kt): it records the Groovy-dynamic kernel result as the anchor, rejects a case if the clean-room interpreter disagrees, executes the static-Java kernel strategy, and records any strategy split. Those cases have no recorded divergence, so both kernel strategies and the interpreter agreed at capture time.

The seven iteration cases use the same adapter engine boundary but were captured directly through [`CorpusEngines`](../../a12-rulekit/adapter/src/test/kotlin/io/github/mbackschat/a12/dm/adapter/laws/CorpusEngines.kt), because one high-value malformed-filter witness correctly exposed an interpreter disagreement that `CorpusCapture` is designed to block. All seven have identical Groovy-dynamic and static-Java kernel results. Six also agree with the a12-dmkits interpreter; [`having-malformed-filter-drops.json`](../evidence/kernel-30.8.1/cases/iteration/having-malformed-filter-drops.json) does not, and the complete three-result split is retained separately in [`triangulation/having-malformed-filter-drops.json`](../evidence/kernel-30.8.1/triangulation/having-malformed-filter-drops.json). The interpreter result is not encoded as a standard `divergences` entry because that field is reserved for kernel execution strategies. Static authoring observations are kernel-confirmed model-validation results and are not runtime-strategy comparisons.

The 12 captured-outer correlation cases use the adapter engine boundary across seven standalone models. Groovy-dynamic, static-Java, and the a12-dmkits interpreter agree on all 12. The earlier seven observations cover clean-filled direction/self-match/self-exclusion and the all-valid plus symmetric malformed-consumer footprints; five additional observations cover empty and malformed local filter operands, numeric inequality, repetition equality, and repetition ordering. The cases retain complete canonically sorted message-signature lists, including formal errors outside the focused authored code, and each referenced model carries the actual executed stored `errorCondition` containing `$`; this is intentional because a formatter/conversion path has previously lost a `$`-bearing conjunct. The canonical list order comes from a12-dmkits' `CorpusEngines` normalization and is not claimed to be kernel emission order. The typed correlation projection describes origins structurally but never substitutes for preserving that raw executed condition.

The six operator-sensitive empty-value runtime cases use two standalone models to hold field type fixed while varying the consuming operator. The retained [`capture receipt`](../evidence/kernel-30.8.1/captures/operator-sensitive-empty-2026-07-15.json) pins kernel 30.8.1, a12-dmkits revision `699e8619ac1667c861e14b285c5924ac57a705f1`, the two model digests, all six case digests, and the six rule identities and stored conditions. The 13 String-computation runtime cases use three standalone models to separate direct copy, suffix concatenation, and two-field concatenation while independently varying source emptiness, prior target state, and row content. The retained [`capture receipt`](../evidence/kernel-30.8.1/captures/string-computation-2026-07-15.json) binds the disposable harness digest, normalized command and required environment, a12-dmkits revision, three complete model artifacts and consistency results, all 13 case digests, and the complete Groovy-dynamic, static-Java, and interpreter signature lists. ## Why replay uses a focused projection

The portable a12-dmkits case format deliberately starts from full DM-JSON and stored condition text. The current Lean capsule starts later: expanded field declarations, classified scalar cells, and structured parser-independent conditions. Consuming the full corpus directly would force a premature DM-JSON loader and bilingual condition parser into a capsule whose semantics intentionally begin after those boundaries.

[`projection.json`](../evidence/kernel-30.8.1/projection.json) therefore carries only the closed input projection needed by the flat/path/required fragment: field declarations, structured paths/conditions, classified cells, row-content eligibility, and operation kind. [`iteration-projection.json`](../evidence/kernel-30.8.1/iteration-projection.json) is deliberately separate and carries one repeatable group, ordered row IDs, row-indexed numeric cells, an optional row-local equality filter, and the narrow selected-sum comparison. [`correlation-projection.json`](../evidence/kernel-30.8.1/correlation-projection.json) schema v2 carries the second one-group fragment: 1-based ordered row IDs, numeric field IDs and paths, a closed filter with explicit inner/outer origins, explicit guard and starred-consumer field IDs, the guarded any-filled observer, and the outer-row-to-pointer map. [`correlation-elaboration-projection.json`](../evidence/kernel-30.8.1/correlation-elaboration-projection.json) is a fourth closed projection containing only repeatable-group declarations, Number-field metadata, group-qualified relative field and one-star references, inner/outer origins, the admitted `equal`/`lessThan` comparison tags, and the referenced external observation. Its closed JSON boundary is defined by [`CorrelationElaborationSchema.lean`](../A12Kernel/Evidence/CorrelationElaborationSchema.lean), and [`CorrelationElaborationReplay.lean`](../A12Kernel/Evidence/CorrelationElaborationReplay.lean) lowers it through the public checked elaborator.

[`operator-empty-projection.json`](../evidence/kernel-30.8.1/operator-empty-projection.json) is the fifth closed projection. [`OperatorEmptySchema.lean`](../A12Kernel/Evidence/OperatorEmptySchema.lean) admits exactly Number and String declarations, the three directional Number rules, direct String equality, and the two captured `Length` comparisons; [`OperatorEmptyReplay.lean`](../A12Kernel/Evidence/OperatorEmptyReplay.lean) lowers those forms through the shared checked library flat route. That route is also used by the normalized process, but String and `Length` are not public protocol forms. The driver binds the provenance receipt, its source revision, the two complete model-file digests, six case-file digests, complete model and rule inventories, and the unique projected fields with their exact identities, plus exact rule names, codes, stored conditions, error routes, and complete three-strategy signature lists. Each projected case is additionally tied to the retained `GROUP`/`FIELD`/`EMPTY` placements, repetition numbers, the `Acme` content witness that independently sets row eligibility, projected cells, and `hasContent`. The projection contains no separately authored expected Lean verdict; expected signatures come from the retained external result.

[`string-computation-projection.json`](../evidence/kernel-30.8.1/string-computation-projection.json) is the sixth closed projection. [`StringComputationSchema.lean`](../A12Kernel/Evidence/StringComputationSchema.lean) transports exactly three resolved non-repeatable String computation shapes, classified source cells, prior target state, row-content witness, and retained artifact identities. [`StringComputationReplay.lean`](../A12Kernel/Evidence/StringComputationReplay.lean) evaluates those inputs through the library String semantics and derives delta signatures without carrying a second expected table. The IO-only [`StringComputationBinding.lean`](../A12Kernel/Evidence/StringComputationBinding.lean) hash-binds the receipt, all models, and all cases; requires exact receipt and version-owned case inventories; structurally binds language, group, four fields, computation identity, target, rendered operation, and placements; requires Groovy-dynamic and static-Java agreement; and checks the exact three-case interpreter mismatch set as triangulation. The receipt's normalized model-shape digest remains a hash-retained assertion established by the removed disposable harness; this repository does not ship the normalization algorithm needed to recompute it independently.

[`string-target-validation-projection.json`](../evidence/kernel-30.8.1/string-target-validation-projection.json) is the seventh closed projection. [`StringTargetValidationSchema.lean`](../A12Kernel/Evidence/StringTargetValidationSchema.lean) carries only four resolved String-operation and positive single-bound policies, nine source/prior-target inputs, the target pointer, and retained artifact identities; it carries no expected delta or application table. [`StringTargetValidationReplay.lean`](../A12Kernel/Evidence/StringTargetValidationReplay.lean) derives the target outcome, payloadful computation delta, and value-only applied result through the library semantics. The IO-only [`StringTargetValidationBinding.lean`](../A12Kernel/Evidence/StringTargetValidationBinding.lean) binds the exact receipt, models, cases, model structure, operation text, placements, target policy, rich kernel result, delta, exact applied target shape, two kernel strategies, and a12-dmkits triangulation. Exact `absent` versus present `empty` after rejection remains retained and mechanically checked external evidence; Lean currently replays only the value-level application view and does not overstate that stronger document-instance distinction.

For runtime correlation, the driver traverses the retained model's own group tree, requires unique projected group and Number-field paths/configurations and a unique rule carrying the focused code, resolves the rule's authored absolute-or-relative condition group/field references to the projected absolute paths, resolves its error entity to the separately projected guard field, canonically renders only the admitted guarded-any-filled/filter subset, and requires byte-exact equality with the actual stored `errorCondition`; this binds the typed filter to the exact authored-path rule the kernel executed without adding a general condition parser or repeatable elaborator. For static correlation elaboration, the driver first checks the retained model's full-file SHA-256 identity, then requires `content.modelConfig.conditionLanguage.code` to be `en_US`, requires the declaring and projected groups uniquely in that seeded model with canonical names and physical parents, checks repeatability and rejects a projected group with a repeatable proper ancestor, requires every projected field to have its canonical name and immediate physical parent plus exact Number kind/scale/signedness, binds the stored valid seed rule's identity/error field/code/severity/base condition, and requires the complete retained candidate draft identity to equal the draft rendered from the projection. Diagnostic provenance, severity, kernel source, and `where.rule` routing are checked before comparing complete code lists. The static renderer admits only conservative unquoted ASCII identifiers outside the full English lexer-keyword set; quoted authoring syntax is outside this transport. [`A12Kernel/EvidenceMain.lean`](../A12Kernel/EvidenceMain.lean) then derives focused firing membership or the complete diagnostic-code list from the external artifact, executes the corresponding typed projection through library Lean semantics/elaboration, and compares the result.

This is an observation-level bridge. In the flat fragment, a fired Lean verdict maps to its VALUE/OMISSION message signature; both `notFired` and `unknown` map to no authored message because the external validation output cannot distinguish them. In the uncorrelated iteration fragment, the current Lean result is truth `K`, so replay compares `K.tru` with focused authored-message presence. In the correlation fragment, replay derives firing membership from the retained canonically sorted focused signatures, reconstructs the expected row/pointer pairs in the projection's document-row order, and compares that list with Lean; it makes no claim about external kernel emission order. Both iteration projections retain the external VALUE/OMISSION signatures without claiming filtered-result polarity is modeled. Consequently the malformed cases support firing or suppression at the message boundary but do not by themselves prove that the kernel's hidden intermediate result is specifically Lean's `unknown`. In particular, the three-row malformed-local correlation witness proves observably that the malformed row is not kept and that healthy equal-key siblings continue to select one another; it does not expose the kernel's hidden truth value for the malformed comparison. That stronger internal treatment remains grounded in source/findings plus the Lean definition and its internal laws. The correlation rule/filter/path binding is gate-checked as described above; the projection's classified cells and row list are still a review-trusted transcription of the retained placements rather than a general DM-instance decoder, and that narrower transport boundary must remain visible in reviews.

The operator-sensitive lane compares the complete canonically sorted signature list emitted by all three retained rules in the selected model. This externally distinguishes the VALUE/OMISSION polarity of the empty Number and `Length` firings, not merely message presence. Direct empty-String equality is externally silent and therefore does not by itself distinguish Lean's `notFired` account from a hidden `unknown`. The retained ASCII String cases also do not establish UTF-16 astral-character length; that law remains grounded in the stated semantic account and internal Lean conformance rather than this external bundle.

The String-computation lane compares complete target-delta signatures, so it directly establishes changed `VALUE`, unchanged silence, stale-target `CLEARED`, and absent-target silence for the retained clean String inputs. It also establishes that a non-repeatable suffix computation runs on the independently empty row. The external operation does not expose Lean's intermediate `StringTerm` or `StringStore`; their separation is the internal account explaining the observations and supporting proofs. The packet contains no formally invalid source, target-constraint failure, `ERRORED` result, precondition, alternative, repeatable scope, or downstream cascade, so those behaviors remain outside this correspondence claim.

The String target-validation lane compares the complete rich computation result and the projected target delta. It directly establishes boundary acceptance, one-code-unit minimum/maximum failure, preservation of the attempted rejected value and exact `stringZuKurz`/`stringZuLang` cause, unconditional `ERRORED` for absent, stale, and equal prior targets, change-sensitive silence only for an accepted equal value, and acceptance of the padded ASCII value `" ABC "` at maximum five. It separately binds the kernel's exact post-application state: rejection leaves an absent target absent but clears a previously filled target to present-empty. The Lean replay intentionally claims only the common value-level result until document application models instance presence. The packet does not establish zero or negative bounds, simultaneous minimum and maximum, patterns, enumerations, line-break order, no-value or poison target-check bypass, `noValueValidation`, initially present-empty targets, astral UTF-16 behavior, general target checking, dependency cascades, checked lowering, repeatable computation, or the public protocol.
