# External kernel evidence

This document owns the portable evidence boundary between the external A12 kernel harness and the Lean theory. It describes what is retained, what `lake test` checks, and which correspondence claims the artifacts do and do not support.

## Roles and provenance

[`../evidence/kernel-30.8.1/`](../evidence/kernel-30.8.1/) contains 30 focused, own-domain observations and their standalone DM-JSON models: 28 runtime conformance cases and two kernel-confirmed static diagnostics. They were captured outside this repository through a12-dmkits' adapter boundary in the local `a12-rulekit/` checkout. The original 17 runtime cases use [`CorpusCapture`](../../a12-rulekit/adapter/src/test/kotlin/io/github/mbackschat/a12/dm/adapter/laws/CorpusCapture.kt): it records the Groovy-dynamic kernel result as the anchor, rejects a case if the clean-room interpreter disagrees, executes the static-Java kernel strategy, and records any strategy split. Those cases have no recorded divergence, so both kernel strategies and the interpreter agreed at capture time.

The seven iteration cases use the same adapter engine boundary but were captured directly through [`CorpusEngines`](../../a12-rulekit/adapter/src/test/kotlin/io/github/mbackschat/a12/dm/adapter/laws/CorpusEngines.kt), because one high-value malformed-filter witness correctly exposed an interpreter disagreement that `CorpusCapture` is designed to block. All seven have identical Groovy-dynamic and static-Java kernel results. Six also agree with the a12-dmkits interpreter; [`having-malformed-filter-drops.json`](../evidence/kernel-30.8.1/cases/iteration/having-malformed-filter-drops.json) does not, and the complete three-result split is retained separately in [`triangulation/having-malformed-filter-drops.json`](../evidence/kernel-30.8.1/triangulation/having-malformed-filter-drops.json). The interpreter result is not encoded as a standard `divergences` entry because that field is reserved for kernel execution strategies. The two diagnostic artifacts are kernel-confirmed model-validation results and are not runtime-strategy comparisons.

The four captured-outer correlation cases also use the adapter engine boundary. Groovy-dynamic, static-Java, and the a12-dmkits interpreter agree on all four. The cases retain the complete ordered message signatures and reference standalone models whose `errorCondition` is the actual executed stored text containing `$`; this is intentional because a formatter/conversion path has previously lost a `$`-bearing conjunct. The typed correlation projection describes origins structurally but never substitutes for preserving that raw executed condition.

The runtime cases use a12-dmkits' portable conformance shape: pinned kernel version, model reference, ordered document placements, operation, complete observed message signatures, and optional divergence records. The diagnostic artifacts retain the pinned version, complete draft, expected code, and full structured kernel diagnostic list. The models and full outputs are retained rather than replacing them with Lean-shaped expected values. This repository contains no kernel binary, binding, or runtime dependency.

## Why replay uses a focused projection

The portable a12-dmkits case format deliberately starts from full DM-JSON and stored condition text. The current Lean capsule starts later: expanded field declarations, classified scalar cells, and structured parser-independent conditions. Consuming the full corpus directly would force a premature DM-JSON loader and bilingual condition parser into a capsule whose semantics intentionally begin after those boundaries.

[`projection.json`](../evidence/kernel-30.8.1/projection.json) therefore carries only the closed input projection needed by the flat/path/required fragment: field declarations, structured paths/conditions, classified cells, row-content eligibility, and operation kind. [`iteration-projection.json`](../evidence/kernel-30.8.1/iteration-projection.json) is deliberately separate and carries one repeatable group, ordered row IDs, row-indexed numeric cells, an optional row-local equality filter, and the narrow selected-sum comparison. [`correlation-projection.json`](../evidence/kernel-30.8.1/correlation-projection.json) carries the second one-group fragment: 1-based ordered row IDs, numeric field IDs and paths, a closed filter with explicit inner/outer origins, the guarded any-filled observer, and the outer-row-to-pointer map. None contains an expected Lean result. For correlation, the driver traverses the retained model's own group tree, requires unique projected group and Number-field paths and a unique rule carrying the focused code, resolves that rule's error entity to the projected value field, canonically renders only the admitted guarded-any-filled/filter subset, and requires byte-exact equality with the actual stored `errorCondition`; this binds the typed filter to the rule the kernel executed without adding a general condition parser. [`A12Kernel/EvidenceMain.lean`](../A12Kernel/EvidenceMain.lean) then derives the focused expected emission, raw signature order, or diagnostic from the complete external artifact, executes the typed projection through public Lean semantics/elaboration, and compares the result.

This is an observation-level bridge. In the flat fragment, a fired Lean verdict maps to its VALUE/OMISSION message signature; both `notFired` and `unknown` map to no authored message because the external validation output cannot distinguish them. In the uncorrelated iteration fragment, the current Lean result is truth `K`, so replay compares `K.tru` with focused authored-message presence. In the correlation fragment, replay preserves the raw focused message order and compares the ordered Lean firing rows and mapped pointers with every focused authored message for that rule. Both iteration projections retain the external VALUE/OMISSION signatures without claiming filtered-result polarity is modeled. Consequently the malformed cases support firing or suppression at the message boundary but do not by themselves prove that the kernel's hidden intermediate result is specifically Lean's `unknown`. That stronger internal treatment remains grounded in source/findings plus the Lean definition and its internal laws. The correlation rule/filter/path binding is gate-checked as described above; the projection's classified cells and row list are still a review-trusted transcription of the retained placements rather than a general DM-instance decoder, and that narrower transport boundary must remain visible in reviews.

The evidence transport and IO driver stay outside [`A12Kernel.lean`](../A12Kernel.lean), [`A12Kernel/Conformance.lean`](../A12Kernel/Conformance.lean), and [`A12Kernel/Proofs.lean`](../A12Kernel/Proofs.lean). The pure replay projection is not part of the trusted semantic theorem root; it is an empirical adequacy check over retained observations.

## Current coverage

The retained bundle covers these focused observable witnesses:

- empty Number comparison in a content-bearing row and its suppression by the all-empty row gate;
- empty Boolean and Confirm comparison asymmetry;
- `FieldNotFilled` eligibility on an otherwise empty row;
- malformed Number suppression and the `healthy Or malformed` / `healthy And malformed` branch behaviors;
- matching and mismatching parent-relative Number resolution plus a positive absolute path;
- bare-name declaring-group precedence, unique model-wide fallback, duplicate-name rejection, and flag-disabled rejection;
- empty, filled, and malformed absolute/non-repeatable required Number behavior.
- unfiltered `Sum == 17`, filtered `Sum == 7`, a one-cell mismatch, and no-match filtered-sum identity zero over one repeatable group;
- a malformed filter operand dropping its row, plus a paired malformed consumed cell that is harmless when dropped and suppresses the authored rule when kept;
- captured-outer direction: `[inner Count] < [$outer Count]` over `5, 6, 9` fires on outer rows 2 and 3;
- unexcluded equality self-match fires on every outer row;
- explicit `CurrentRepetition(inner) != CurrentRepetition($outer)` makes distinct values fire nowhere and duplicate `5, 5, 9` fire on outer rows 1 and 2.

This closes the focused external gap for the earlier runtime clauses, the bare-name mechanism, capsule 5a's uncorrelated truth fragment, and capsule 5b's four clean-filled direction/self-match/self-exclusion shapes. The path cases corrected a real modeling error: bare names do not walk ancestors; they try the declaring group and then, when enabled, a model-wide unique short name. [`LF6`](LEAN-FINDINGS.md#lf6--bare-name-resolution-is-local-or-global-not-an-ancestor-walk) records that correction. The malformed-filter case exposed an a12-dmkits interpreter disagreement while both kernel strategies and Lean agree; [`LF7`](LEAN-FINDINGS.md#lf7--a-malformed-having-filter-drops-its-row-before-consumption) records its exact scope. The correlation evidence does not establish correlated empty/malformed behavior, numeric `notEqual`, repetition `equal`/`lessThan`, repeatable path elaboration, `$` authoring diagnostics, field/path/equality-scale legality, filtered-result polarity, phase-aware consumer footprints, nested/multi-star or cross-group binding, or general consumers. Static observations for other rejected surface shapes and the legal arithmetic precision witness also remain outside the current fragment. These are explicit boundaries rather than reasons to delay the next capsule.

## Running the gate

```sh
lake test
```

The test driver rejects an unsupported schema/kernel version, empty or duplicate case identities, zero/duplicate/undeclared correlation rows or cells, invalid or duplicate projected entity paths, incomplete or duplicate outer-row pointer maps, correlated filters missing either origin, a missing or non-unique retained group/Number field/focused rule, a mismatched error path or stored condition, invalid projected models, unsafe references, missing retained models, kernel-strategy divergences, malformed focused signatures, elaboration failures, and result mismatches. A successful run reports the number of agreeing projections across all three projection bundles; the current gate reports `30/30`.

`lake build` remains the pure internal build and executable-example gate. `lake test` adds retained external correspondence evidence; [`scripts/check-lean-trust.sh`](../scripts/check-lean-trust.sh) independently audits the trusted theorem closure. None substitutes for the others.
