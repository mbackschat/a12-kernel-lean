## Architecture boundary

The executable semantics, elaboration, and theorem roots remain pure Lean library layers. Conformance examples are independent checked witnesses; evidence and reference-process adapters are untrusted transport layers outside the theorem root. This dependency direction is the architectural consequence of the project doctrine: IO, external observations, and a downstream candidate may test the semantic center but never define it.

Every public semantic meaning is version-scoped. [`A12Kernel/Basic.lean`](../A12Kernel/Basic.lean)'s `kernelVersion` records the one modeled external behavior version, while [`A12Kernel/Reference/Lineage.lean`](../A12Kernel/Reference/Lineage.lean) separates the historical 0.2.0 executable account from current 0.3.0 without pretending that the kernel behavior version changed. The source tree evaluates only the current account; historical behavior remains reproducible through its pinned source revision and immutable artifacts rather than a second namespace in the trusted semantics. Migrate the semantic namespace to `A12Kernel.V30_8_1` before a second kernel behavior version or stable library API makes the unversioned meaning ambiguous.

The 2026-07-12 external PL-semantics spike is no longer a live input. Where it conflicted with A12 evidenceŌĆömost visibly an intrinsically typed AST and a `{coefficient, scale}` decimal representationŌĆöthe A12 boundary won; the resulting adopted and rejected choices are recorded below.

## Core encoding decisions

### Extrinsic (untyped) AST, not an intrinsically-typed one

The AST is indexed only by nothing ŌĆö a closed inductive with an exhaustive evaluator (per [`../spec/13`](../spec/13-lean-encoding-guide.md)). We **rejected** an intrinsically-typed `Expr (Žā : Schema) : Ty ŌåÆ Type`: A12's type system does not encode cheaply into indices (scale goes `unknown` after `/`/`^`; `TypeDefinition` legality is only known post-expansion; enum comparability is config-dependent; `Custom` delegates), and the hard semantic content lives in the *value domain and cell observations*, not in type indices ŌĆö where intrinsic typing would make proofs fight `Eq.mpr`/transport. Static type/scale/scope checking is done by an elaboration pass that produces a core rule with an explicit `WellFormed` boundary; elaboration soundness and the absence of impossible evaluator failures for accepted core rules are proof obligations. Intrinsic typing stays available as a later hardening only if a concrete proof goal demands it. Derived scale is modelled as `ScaleInfo (exact | unknown)` plain data ([`A12Kernel/Core.lean`](../A12Kernel/Core.lean)).

### Checked elaboration resolves names and owns field-policy coherence

[`A12Kernel/Elaboration/Flat.lean`](../A12Kernel/Elaboration/Flat.lean) follows the parserŌĆōelaboratorŌĆōcore split used by Lean/Lean4Lean and the explicit surface-to-core boundary exemplified by `do` Unchained. Its input is structured and parser-independent; its output is a `CheckedFlatCondition` carrying proofs that the expanded model validated and every core field matches one unique non-repeatable declaration. The supported path subset is intentionally narrow: absolute paths, parent-relative paths without named turning-point labels, and bare lookup in the kernel order declaring group ŌåÆ flag-gated model-wide unique. There is no implicit ancestor tier; parent lookup requires explicit `..` ([`LF6`](LEAN-FINDINGS.md#lf6--bare-name-resolution-is-local-or-global-not-an-ancestor-walk)). The admitted scalar surface now includes Number/Boolean/Confirm equality and inequality, direct String equality, and String `Length <`/`>=`. String inequality, String presence/requiredness, other Length comparisons, ambiguity, invalid models, repeatable references, other unsupported operators, kind mismatches, and illegal Confirm literals fail closed.

The same `FlatModel` compiles raw cells into the runtime `FlatContext`, so successful surface evaluation cannot pair a resolved numeric field with a caller-invented Boolean policy. Trusted theorems connect every admitted field to its unique matching declaration and prove that context construction applies exactly that declarationŌĆÖs `formalCheck` policy; missing or ambiguous IDs become malformed/unknown. The low-level `FlatCondition.evalFull` remains available for isolated semantic laws and can still receive an arbitrary `FlatContext`, so policy coherence is claimed only for the checked surface route.

The flat elaboration slice does not implement the complete ┬¦10 path language. Quoted concrete syntax, named-ancestor labels, `RuleGroup`, stars, `$`, semantic indices, and parser/renderer preservation remain outside it. The checked one-star correlation elaborator below reuses the same expanded-model and path mechanisms for one deliberately closed repeatable shape; it does not turn either slice into a general authored-condition parser. [`IMPLEMENTATION-MAP.md`](IMPLEMENTATION-MAP.md) owns the live assurance and external-evidence status.

### `Value.num` is `Rat` + a separate rendered stored-form ŌĆö not a `{coefficient, scale}` decimal

The value domain carries exact `Rat` for numeric value. A plain `Rat` loses the *representation* A12 needs (`7` vs `7.00` is a reportable compute delta; `minFractionalDigits` rendering). We resolve that by keeping the **numeric value** as `Rat` and modelling the **stored form** as a separately-rendered string (which is literally what A12 stores ŌĆö a value in the target's declared format), so representation-equality is stored-form string equality while numeric comparison is on `Rat`. Faithfulness to the engine's `BigDecimal` requires applying explicit rounding at the arithmetic points: [`A12Kernel/Semantics/NumericComparison.lean`](../A12Kernel/Semantics/NumericComparison.lean) owns the shared scale-19 `HALF_UP` normalization used by flat, iteration, and correlation comparisons, while `MathContext(50)` intermediates and the remaining arithmetic/target-rounding operations are future number-stage work. Its `NumericComparisonOp` intentionally contains only `equal`, `notEqual`, `less`, and `greaterEqual`: these are the union required by the currently closed Number and String-Length consumers, not a claim that the other comparison surfaces are implemented. This keeps the value domain simple and reversible; switching to an explicit `{coefficient, scale}` decimal remains an option if the rendered-string split proves awkward.

### Numeric polarity carries directions, not one substituted bit

A numeric value reaching a comparison carries `NumericFillability {canGrow, canShrink}` separately from its exact amount. A filled scalar is fixed in both directions; an empty unsigned Number becomes zero and can only grow; an empty signed Number becomes zero and can grow or shrink; an empty String consumed by `Length` becomes grow-only zero. `NumericComparisonOp.leftFillCanBreak` then asks whether an available direction could falsify the currently true error condition against the fixed literal. This makes empty unsigned `0 != -1` VALUE, because filling cannot move downward to `-1`, but empty unsigned `0 != 1` OMISSION, because filling could reach `1`. It also makes `Length(empty) < 5` OMISSION and `Length(empty) >= 0` VALUE. A single `given := false` bit cannot express these distinctions; it remains appropriate only inside the current symmetric Boolean/Confirm comparison helper.

The current function is deliberately fixed-right rather than a speculative two-operand polarity algebra. [`A12Kernel/Proofs/NumericComparison.lean`](../A12Kernel/Proofs/NumericComparison.lean) proves the fixed and grow-only clauses, including the generic non-below inequality result; [`A12Kernel/Conformance/FlatValidation.lean`](../A12Kernel/Conformance/FlatValidation.lean) locks two different negative literals as witnesses of the repaired mechanism. General expression propagation, comparisons in which both sides are fillable, arithmetic, aggregate tails, and the distinct symmetric date/string rules remain later capsules. [`LF10`](LEAN-FINDINGS.md#lf10--numeric-polarity-needs-directional-fillability-not-a-given-bit) records the semantic correction.

### Unified `Verdict`, not a bare fired/notFired outcome

A condition evaluates to `Verdict = notFired | fired (p : Polarity) | unknown` with explicit `conj`/`disj` tables (strong-Kleene on truth; omission-wins-`And`, value-wins-`Or`), merging truth and the spec's `Polarity` into one type. A bare `firedValue | firedOmission | notFired` outcome cannot represent an `unknown` rule result, which is essential because a formal error can leave a whole condition unknown. `K` is retained for expression/predicate-level truth and for stating the exact information-order laws: unknown lies below both definite values, which are incomparable; `And`/`Or` are monotone under that order, so unknown may refine to true or false while an already definite result is stable. The `conj`/`disj` tables have executable `example ŌĆ” := rfl` guards; the algebraic theory still requires quantified proofs ([`A12Kernel/Core.lean`](../A12Kernel/Core.lean)).

### Two-level cell model: `CheckedCell` ŌåÆ `observeCell(Phase)` ŌåÆ `CellObservation`

The earlier three-state sketch (empty ŌēĀ invalid) is refined into an invariant `CheckedCell {rawPresent, parsed, findings}` and a **phase-indexed read** producing a `CellObservation` (`empty | value | unknown cause | poison cause`) ([`A12Kernel/Cell.lean`](../A12Kernel/Cell.lean)). Base `formalCheck` handles ordinary local findings such as malformed values and declared constraints; generated and structural passes annotate the same checked-cell representation later. Requiredness is deliberately staged: evaluate the generated mandatory rule against base checked cells, retain its hit/message, and only on a hit add `.required` to the empty target for authored validation rules. Installing `.required` first would make the mandatory rule's own `FieldNotFilled` read unknown and suppress itself. Ordinary findings surface as `unknown` in validation and `poison` in computation, while `.required` is validation-scoped and observes as ordinary empty during computation. **Empty-substitution never happens in the read** ŌĆö the consuming operator decides what an empty operand means (numberŌåÆ`0` in `<`, skipped by `Max`/`Min`; stringŌåÆ`""` in concat, ignored by `==`). This is the single most important defence against the most common reimplementation bug.

`CellObservation.empty` is intentionally unresolved. Field kind supplies a default tier, but the semantic clause consuming the observation may override it by operator and operand position, and the enclosing consumer may still distinguish comparison, concatenation, or assignment. Model-derived field roles act earlier by adding staged findings, while `Phase` decides whether those findings expose unknown, poison, orŌĆöin the required-only exceptionŌĆöordinary empty. Consequently the evaluator must not expose a global `FieldKind ŌåÆ Value` normalization. Each operator family owns named operand-resolution functions that preserve the provenance its consumer needs: simple symmetric clauses may retain a given/substituted bit, while numeric clauses retain directional `canGrow`/`canShrink`. Aggregate semantics state both per-element empty treatment and the all-empty selection identity; computation retains empty/no-value until the store boundary can choose VALUE, CLEARED, or ERRORED. [`LF5`](LEAN-FINDINGS.md#lf5--empty-handling-is-a-layered-consuming-clause-policy-not-a-field-kind-function) records the consuming-clause boundary, and [`LF10`](LEAN-FINDINGS.md#lf10--numeric-polarity-needs-directional-fillability-not-a-given-bit) records the directional refinement.

### String support begins at two named consuming clauses

The first String capsule does not add a generic String value policy. [`formalCheck`](../A12Kernel/Semantics/Observation.lean) applies [`normalizeEvaluatedString`](../A12Kernel/Semantics/String.lean) once when an admitted parsed String enters the checked evaluation cache: each existing non-overlapping CRLF pair becomes LF, existing LF and lone CR remain unchanged, and an empty normalized result becomes the same clean-empty cell as absent input. This is evaluation-side content, not a mutation of stored document text. The transformation is deliberately not globally idempotentŌĆöone pass maps `"\r\r\n"` to `"\r\n"`, while a second pass would map that result to `"\n"`ŌĆöso direct comparison, `Length`, and computation reads consume the cached checked value and never normalize independently. [`A12Kernel/Proofs/StringIngestion.lean`](../A12Kernel/Proofs/StringIngestion.lean) states that shared-consumer boundary and the checked non-law; [`A12Kernel/Conformance/StringIngestion.lean`](../A12Kernel/Conformance/StringIngestion.lean) locks the separating concrete cases.

Direct String equality consumes a clean-empty observation as not evaluated and also suppresses an empty literal; `Length` instead consumes it as grow-only numeric zero. Filled String length is the number of UTF-16 code units in the cached evaluated String, matching the Java/JavaScript engine boundary rather than Unicode scalar values or grapheme clusters. The current elaborator admits only direct equality plus `Length <` and `Length >=`; direct inequality, patterns, other functions, String presence/requiredness, line-break permission, enumeration ordering, and raw-type rule elimination are explicitly rejected, fail closed, or remain unimplemented rather than inferred. [`A12Kernel/Conformance/StringLength.lean`](../A12Kernel/Conformance/StringLength.lean) locks empty, filled, malformed, wrong-kind, row-gate, UTF-16, and rejection cases, while [`A12Kernel/Proofs/StringLength.lean`](../A12Kernel/Proofs/StringLength.lean) proves the same-empty operator distinction.

This capsule remains internal to the library and evidence replay. The normalized reference protocol and existing consumer capabilities do not accept String or Length, so no public transport or release claim grows merely because the core can execute these clauses.

### String computation separates expression, root store, target check, application, and delta

[`A12Kernel/Semantics/StringComputation.lean`](../A12Kernel/Semantics/StringComputation.lean) begins ┬¦11 with one unconditional, non-repeatable, already-resolved String computation. `StringExpr.eval` produces `StringTerm`: a clean field absence is `noValue`, a literal or concatenation is evaluated `text`, and a formal error is `poison`. Concatenation is the consuming clause that turns a clean missing operand into an empty contribution. It may therefore produce `text ""` when every contribution is empty; that intermediate is deliberately not collapsed back to `noValue`.

The root store is a separate total function. It maps both bare `noValue` and final `text ""` to `StringStore.noValue`, maps nonempty text to `produced StoredString`, and preserves poison. `StoredString` proves only that its rendered stored form is nonempty; target legality is intentionally not part of that type because the same value must survive as the attempted payload of `ERRORED`. A low-level wrong-kind cell is an explicit `StringComputationFault`, not another quiet empty case. This evaluator starts after model/path checking; a checked computation elaborator remains later work.

`StringTargetLengthPolicy.check` is the next boundary. The first closed subset admits either one positive minimum or one positive maximum; zero/nonpositive declarations, combined bounds, patterns, enumerations, line-break permission, and `noValueValidation` remain outside it until separately observed. Input-ingestion normalization does not grant permission for a computed target to contain a line break. A root no-value or poison bypasses target checking unchanged. A produced String with no CR or LF is either `accepted` or `errored attempted cause`; target rejection is an ordinary semantic result, not a `StringComputationFault`, poison, or payload-free flag. A produced String containing CR or LF instead yields `StringTargetCheckResult.unsupported`, so this length-only capsule cannot silently infer the preceding target-permission decision. That result is fail-closed fragment routing, not an external computation outcome. The admitted check uses the shared [`utf16CodeUnitLength`](../A12Kernel/Semantics/String.lean), while the retained conservative-ASCII packet does not establish broader Unicode behavior externally.

`StringTargetOutcome.projectDelta` owns report granularity. Accepted values are change-sensitive, but the single outcome-first `errored` branch always emits `ERRORED(attempted, cause)` for absent, stale, and equal prior targets. It and the pre-target `StringStore.projectDelta` compatibility API share only the lower-level value/no-value delta primitives. The pre-target API never constructs a `StringTargetOutcome`, so an unchecked write attempt cannot masquerade as `accepted`; the two lanes still cannot acquire competing change/clear implementations. `StringTargetOutcome.appliedValue` is deliberately only a value-level projection: accepted exposes its checked value; errored, no-value, and poison expose no stored value. It does not conflate or claim the document-level distinction between an absent target and a present-empty target after application; the retained packet preserves that stronger shape for a later application-state capsule.

This decomposition turns the apparent contradictions ŌĆ£empty String concatenates as `""`ŌĆØ versus ŌĆ£a final empty String is not stored,ŌĆØ and ŌĆ£equal values are silentŌĆØ versus ŌĆ£equal invalid attempts still report,ŌĆØ into explicit boundaries. It supplies two refactoring rules: adding a clean missing contribution is store- and delta-preserving although not term equality, and target-check preservation must be stated at the target-outcome or delta boundary rather than only over expression text. [`A12Kernel/Proofs/StringComputation.lean`](../A12Kernel/Proofs/StringComputation.lean) states the useful laws and nearest stronger non-laws; [`LF12`](LEAN-FINDINGS.md#lf12--string-computation-has-expression-store-and-delta-boundaries) and [`LF13`](LEAN-FINDINGS.md#lf13--string-target-outcome-delta-and-application-are-separate-boundaries) record the evidence-backed rationale.

[`A12Kernel/Semantics/StringCascade.lean`](../A12Kernel/Semantics/StringCascade.lean) composes exactly two resolved steps through an explicit outcome overlay. The producer's checked `StringTargetOutcome`, not its delta or applied-value projection, determines the dependent cell: accepted supplies the stored value, clean no-value supplies an empty cell, target error supplies declared-constraint poison without the attempted payload, and ordinary inherited poison preserves its cause. Validation-scoped `.required` is rejected at the outer `StringDependencyFault` boundary because computation observation intentionally ignores it. The nested `Except StringDependencyFault (Except StringComputationFault StringTerm)` is therefore an intentional separation of dependency binding from the later read, not an accidental API shape. This remains an evaluation overlay rather than document application or a scheduler.

The retained direct-cascade evidence enters only through the nontrusted executable boundary. [`StringCascadeSchema.lean`](../A12Kernel/Evidence/StringCascadeSchema.lean) is the closed operation decoder, [`StringCascadeReplay.lean`](../A12Kernel/Evidence/StringCascadeReplay.lean) projects the library evaluator to the public result vocabulary, and [`StringCascadeBinding.lean`](../A12Kernel/Evidence/StringCascadeBinding.lean) verifies the packet, qualification, route relations, and replay. The binder preserves raw channel order and multiplicity for route equality; canonical signature sorting is used only for the declared set-valued semantic comparison. Exact absent, present-empty, and indeterminate states remain retained and route-checked, but the Lean bridge intentionally projects all no-value states to one value-only result. This one-way evidence dependency cannot enter the library or proof roots and does not strengthen the packet into a claim about hidden dependency representation or scheduling.

### `Document` = instantiated rows independent of cell values

`Document {instantiatedRows : List RowAddr, rawCells : CellAddr ŌåÆ Option String}` ([`A12Kernel/Document.lean`](../A12Kernel/Document.lean)). Row existence is tracked separately from cell content because a blank-but-instantiated repeat row is observable ŌĆö inferring rows from non-empty cells breaks `GroupFilled`, requiredness, the row-gate, and repeatable-group quantifiers. We use `List`, **not `Finset`**, because order is observable (`FirstFilledValue`, computation scheduling, poison reads) ŌĆö and `Finset` would pull Mathlib and hurt `#eval`.

### Single-level iteration starts from an ordered candidate list

[`A12Kernel/Semantics/Iteration.lean`](../A12Kernel/Semantics/Iteration.lean) models the first repeatable core capsule as `SingleGroupValidationContext`: one repeatable group, an ordered `List RowIndex` of explicitly instantiated candidates, and a row-local validation read. It never infers candidates from filled cells and never invents a declared-but-uninstantiated tail. `envAt` exposes the corresponding one-binding `Env`. `SingleGroupValidationContext.WellFormed` states the kernel-correspondence boundary that row indices are 1-based and unique while the list itself carries document order; the low-level evaluator remains total on other lists without claiming they are kernel documents.

`SingleStar` contains one numeric value field and an optional uncorrelated row-local `Having` condition. Selection keeps a candidate exactly when the condition evaluates to `Verdict.fired`ŌĆöwith either polarityŌĆöand drops `notFired` and validation `unknown`; no filter keeps every candidate. Selection is completed before `NumberFold.sumRows` observes the numeric consumer. Empty selected cells are skipped, so a selection with no numeric values sums to zero; an invalid selected consumer yields `NumberFold.unknown`, while a consumer in a dropped row is irrelevant.

The executable selector preserves order and multiplicity with `List.filter`. [`A12Kernel/Proofs/Iteration.lean`](../A12Kernel/Proofs/Iteration.lean) relates it exactlyŌĆönot merely by membershipŌĆöto the independent ordered `SelectRows` keep/drop relation, and proves `sumSelected_filter_before_consumer`: once two contexts select the same ordered rows, agreement on consumer cells in those rows is sufficient for equal sums. [`A12Kernel/Conformance/Iteration.lean`](../A12Kernel/Conformance/Iteration.lean) checks the neighboring false claim that a same-group star collapses to the current row, plus empty, malformed-filter, malformed-dropped-consumer, and malformed-kept-consumer cases.

This is a validation-only selection capsule, not general aggregation support. Numeric `Sum` is deliberately the narrow observer of which rows were selected, and its comparison returns truth `K` only. Filtered-result polarity, nested or multiple stars, cross-group joins, static legality and elaboration, adaptation from `Document`, computation, partial validation, and general aggregate operators remain separate capsules. Retained evidence replays only the truth/firing projection while preserving richer external signatures for later work; [`EVIDENCE.md`](EVIDENCE.md) owns the exact witness inventory and claim boundary.

### Captured outer correlation is a closed filter-only core

[`A12Kernel/Semantics/Correlation.lean`](../A12Kernel/Semantics/Correlation.lean) adds the next one-group validation capsule without widening ordinary rule expressions. `CapturedSingleGroupContext` holds the explicit outer row, while each candidate creates a `SingleGroupFilterFrame` with independently observable `innerEnv` and `outerEnv`. `HavingOrigin.inner` routes ordinary references and `CurrentRepetition(G)` to the candidate row; `HavingOrigin.outer` routes `$` references and `CurrentRepetition($G)` to the captured rule row. Origins exist only inside the closed `CorrelatedHaving` AST, so an outer reference cannot escape into `FlatCondition`.

`OriginCheckedCorrelatedHaving` proves only that both `usesInner` and `usesOuter` hold: an all-`$` filter fails with `missingInner`, while an uncorrelated condition remains in the earlier iteration fragment rather than masquerading as a correlation test. The origin wrapper itself does not establish field scope, path legality, or equality-scale legality; the checked one-star lowering below establishes those properties for its admitted surface fragment. The admitted comparison subset is Number `equal`, `notEqual`, and `lessThan`, plus the same operators over structural repetition indices and conjunction. Numeric empty is resolved as zero locally to this comparison clause; a formally invalid operand yields `K.unknown`, and selection keeps only `K.tru`, so invalid/unknown filter rows drop. `CurrentRepetition` reads the structural `RowIndex` directly and does not observe a cell.

`SingleCorrelatedStar.select` is deliberately the straightforward ordered nested loop: for each captured outer row it scans every same-group candidate in document order, including the outer row itself. Self-match is therefore real, and only an authored `CurrentRepetition(inner) != CurrentRepetition(outer)` excludes it. After that selection, `evalSelectedAnyFilled` applies the validation-presence consumer only to selected rows; the filter may independently have referenced the same field while selecting. `evalGuardedAnyFilledOn` then combines that truth with a separately supplied outer guard field, matching `FieldFilled(outer Guard) And AtLeastOneFieldFilled(G*/Value Having filter)`. The API permits the guard and consumer to be the same field; the 5c evidence uses distinct fields to expose the read footprint. `firingRowsOn` applies that observer per outer row; the earlier same-field methods remain convenience wrappers. The result remains `K`, not a general quantifier tally or filtered-result polarity.

[`A12Kernel/Proofs/Correlation.lean`](../A12Kernel/Proofs/Correlation.lean) proves outer numeric-reference stability, inner numeric-reference locality, executable truth equivalence with the independently stated `CorrelatedHaving.Holds`, exact ordered selector equivalence with `SelectCorrelatedRows`, selection-level explicit self-exclusion, selection-level usable same-field self-match, and `evalGuardedAnyFilledOn_filter_before_consumer`. The last theorem is stated at the validation-observation boundary: once selection agrees, equal outer-guard observations plus equal consumer observations in selected rows are sufficient for equal results, so every dropped consumer cell is outside the footprint. [`A12Kernel/Conformance/Correlation.lean`](../A12Kernel/Conformance/Correlation.lean) locks the neighboring non-laws and the row-correct malformed-consumer pair. With two duplicate-key rows, a malformed first consumer is dropped from outer row 1 and kept from outer row 2, yielding `.tru` then `.unknown`; swapping which row is malformed mirrors the firing row. The all-valid control fires on both. Separate checks show that selection agreement and guard-observation agreement are both load-bearing theorem hypotheses, and that a selected valid cell still satisfies this existential consumer alongside another selected malformed cell.

The retained runtime projection carries explicit guard and consumer roles and gate-checks the exact authored absolute-or-relative condition group/field binding, resolved error field, and stored condition; this is evidence transport, not a general repeatable elaborator. [`EVIDENCE.md`](EVIDENCE.md) owns the exact witness inventory, agreement state, and external claim limits.

### Checked one-star correlation lowering owns the admitted static boundary

[`A12Kernel/Elaboration/Correlation.lean`](../A12Kernel/Elaboration/Correlation.lean) adds a parser-independent surface and checked lowering for exactly one validation form: a Number `FieldFilled` outer guard and matching error field conjoined with one direct-child starred Number `AtLeastOneFieldFilled`, filtered by a correlated Number or `CurrentRepetition` comparison tree. `FlatModel` declares repeatable group paths explicitly, rejects a field path that would also have to be a group path, and derives each field's ordered outer-to-inner repeatable ancestry from those paths; structurally impossible field/group hierarchies and a nested repeatable field masquerading as a singleton are rejected at model validation. Star binding resolves the exact authored group path, reuses the flat declaration resolver and shared parent-walk mechanism, requires every admitted field to be a direct child with exactly the selected singleton repeatable scope, checks equality/inequality scales through one core operator predicate while exempting ordering, rejects unsupported operators and origin routing failures, and emits a `CheckedSingleCorrelatedRule` only after model and core well-formedness checks succeed. Sibling repeatable groups remain admissible because ancestry is path-prefix based rather than a global one-group restriction.

The checked wrapper carries structural certificates; [`A12Kernel/Proofs/CorrelationElaboration.lean`](../A12Kernel/Proofs/CorrelationElaboration.lean) provides certificate elimination, unique-declaration and raw-policy coherence, fail-closed unknown/cross-group/wrong-scope lookup results, the derived equality-scale law, and a bridge from successful raw candidate validation to `SingleGroupValidationContext.WellFormed`. These are not yet a source-to-core semantic-preservation theorem: the surface has no independent dynamic semantics against which lowering is related. The reusable checked runtime route validates that candidate rows are 1-based and duplicate-free before compiling raw cells through the same model declarations used during lowering. The normalized public protocol deliberately narrows this to non-empty contiguous `1..n`, because every retained kernel case uses actual repetition indices and repetition comparisons observe their numeric values. Low-level core evaluators remain total outside that checked route for isolated semantic proofs without receiving the stronger correspondence claim.

The retained static authoring evidence anchors the sharp neighboring boundaries, while its closed projection maps only externally established rejection classes and fails closed on `missingOuter` and every other unclassified result. All-inner `Having` is kernel-valid uncorrelated syntax routed to the earlier capsule, not a correlation diagnostic; [`EVIDENCE.md`](EVIDENCE.md) owns the exact observations, codes, provenance checks, and limitations.

The combined runtime and lowering capsule still does not adapt the general `Document`. Its surface is not the full bilingual DSL and does not admit general Boolean guards, non-Number fields, arbitrary consumers, nested or multiple stars, cross-group execution, semantic indices, named-ancestor/quoted paths, or general error-field routing. The retained authoring observations establish only their exact neighboring cases, not that every cross-group `$` reference is invalid or that scale legality has been exhausted. Further deferrals are filtered-result polarity, hash-join optimization and refinement to the nested loop, computation, partial validation, and general filtered consumers.

### Injected `World`, custom hooks as pure/total oracles

`Today`/`Now`/custom conditions/label lookups read from an explicit `World`, never `IO`, so `eval`/`compute` are reproducible given their explicit inputs ([`A12Kernel/Document.lean`](../A12Kernel/Document.lean)). Custom conditions and field-type validators are external oracles required to be pure and total; a host exception becomes an explicit infrastructure failure, never a silent "did not fire". Purity and totality do **not** imply locality, fill-monotonicity, or stability under partial validation, so theorems needing those properties either exclude custom oracles or quantify over an explicit oracle contract with a read footprint and the required laws. **Open:** timezone/DST semantics (spring-gap, autumn-fold, pinned tz-rule version) ŌĆö the review flagged this as "not mathematically closed", and it is the exact still-open divergence in the a12-dmkits Kotlin interpreter (IG62 in the local `a12-rulekit/` checkout).

### Polarity and partial validation are one-sided-sound abstract interpretations ŌĆö verify, don't assume the `iff`

Partial validation (relevant cells concrete, non-relevant ŌåÆ `unknown`, `true Or unknown` fires while `true And unknown` is suppressed) is a definite-truth abstract interpretation with a one-directional guarantee. Message polarity (VALUE/OMISSION) is likewise an approximation of how the expression could vary under future fills ŌĆö and because `Having` escalates to OMISSION conservatively, the intended reading is **VALUE = proven-not-repairable-by-fill; OMISSION = possibly-repairable**, a one-sided soundness property, *not* an exact classification. Formalize the exact kernel algorithm, then prove the one-sided property only after `InformationRefines`, `FillExtends`, `AgreesOn`, row/world stability, and supported-fragment assumptions are explicit. The canonical theorem ladder and known exclusions live in [`LEAN-FORMALIZATION.md`](LEAN-FORMALIZATION.md).

### The reference process is a transport adapter, not a second evaluator

The `a12-kernel-reference` executable exposes two checked surfaces through the closed normalized contract in [`PROTOCOL.md`](PROTOCOL.md). [`A12Kernel/Reference/Protocol.lean`](../A12Kernel/Reference/Protocol.lean) decodes the shared expanded model plus operation-specific flat or correlation requests. [`A12Kernel/Reference/Evaluator.lean`](../A12Kernel/Reference/Evaluator.lean) dispatches by a finite operation enum: flat validation calls the existing `elaborate`/`CheckedFlatCondition.core.evalFull` route, while captured-outer evaluation calls `elaborateSingleCorrelatedRule` and `CheckedSingleCorrelatedRule.firingRows`. Each route validates only transport invariants around the existing checked evaluator. There is no parallel condition, selection, or firing semantics, and the process never calls the kernel or a12-dmkits.

The transport is deliberately normalized and closed. The process caps stdin at the advertised byte limit before decoding, maps invalid UTF-8 to a domain response, and runs a flat lexical nesting/canonical-number preflight before Lean's ordinary JSON parser can allocate from an exponent or recurse through hostile depth. JSON object shapes reject missing, unknown, and duplicate members; structural naturals are canonical and bounded to the JavaScript-safe integer maximum; exact canonical A12 decimal strings are converted directly to `Rat` without floating point; rejected raw states remain distinct from absence; and sparse omission is operation-specific. Flat cells are addressed by field ID with an explicit `hasContent` row gate. Correlation cells are addressed by `(row, field ID)`, candidates are protocol-narrowed to non-empty contiguous `1..n`, and every supplied cell must belong to a candidate and the resolved repeatable group/scope. Recognized but unsupported operators and surface forms receive stable structured diagnostics, while every internal elaboration constructor is exhaustively classified before it can cross the public boundary. A checked elaborator's `incoherentCore` case remains an infrastructure failure rather than a domain response.

The shipped runtime modules under `A12Kernel/Reference/` import only the JSON parser and basic conversion/printer surface they use rather than the `Lean.Data.Json` umbrella. This narrows the source and elaboration dependency but does not change the native runtime boundary: any remaining dependency on modules under `Lean`, including Lean's JSON API, makes the generated executable initialize the full `Lean` library. The measured release-size consequence and the safe experiment boundary are recorded in [`PRODUCTION-RELEASE.md`](PRODUCTION-RELEASE.md#experiment-2026-07-13-narrow-runtime-json-imports).

[`A12Kernel/Reference/Support.lean`](../A12Kernel/Reference/Support.lean) is the finite Lean source for operation tags, accepted and recognized wire classifiers, operation-specific comparison matrices, verdict tags, diagnostic category/code pairs, resource limits, the current version identity, evidence-boundary metadata, and exclusions. Decoder and encoder decisions consume those same classifiers; conceptual sparse omission is separated from explicit raw-cell tags, and relative path admission is operation-specific. [`Lineage.lean`](../A12Kernel/Reference/Lineage.lean) separately owns historical 0.2.0, current 0.3.0, their manifest and suite identities, the separating observation, and the digest of the selected 152-entry historical evidence/example/principal-artifact [`lock`](../reference/reference-semantics-0.2.0.lock.json); the post-revision separating request and replay receipt are bound separately. The executable generates its current schema-2 `--manifest` response from those declarations, and the process gate compares it semantically with [`supported-fragment-v2.json`](../reference/supported-fragment-v2.json), checks the readable [`reference-semantics-lineage-v1.json`](../reference/reference-semantics-lineage-v1.json) mirror, verifies and traverses the artifact lock, checks the principal frozen digests, and exercises the nearest manifest/runtime mismatches. This follows Cedar's separation of executable specification from validation and process integration: the protocol and IO lane are build-tested product surfaces but are not imported by [`A12Kernel.lean`](../A12Kernel.lean), [`A12Kernel/Conformance.lean`](../A12Kernel/Conformance.lean), or [`A12Kernel/Proofs.lean`](../A12Kernel/Proofs.lean), so they do not enlarge the trusted semantic or theorem roots.

Protocol 1 requests and responses do not carry `referenceSemanticsVersion`; no per-request selector dispatches between accounts. The current binary serves only 0.3.0 and identifies that account through `--manifest`. A consumer therefore binds semantics by checking the exact binary's manifest and pinning the binary or release digest. Reproducing 0.2.0 means using the source revision recorded in the lineage, where the directional Number witness returns OMISSION; the current binary returns VALUE. The v2 flat suite preserves the eight v1 cases and adds that witness, while correlation v2 changes only identity and manifest association. This avoids a parallel legacy evaluator in current source and prevents an unchanged response envelope from masquerading as a semantic version selector.

Protocol v1 handles one request per process rather than a stream, service, or FFI. That is the smallest deterministic language-neutral boundary for Kotlin, TypeScript, Rust, shell, and CI consumers; batching or a long-running service remains a measured deployment decision. The process has no general `Document` decoder and exposes only the non-repeatable flat intersection plus the named one-group correlation intersection in its manifest.

[`A12Kernel/CandidateConformanceMain.lean`](../A12Kernel/CandidateConformanceMain.lean) is a separate downstream test product, not another semantics path. It reads a selected language-neutral suite, enforces closed suite/case/evidence objects, validates that suite's complete compatibility identity, finite evidence scope and case composition against the committed support manifest it names, checks fixture metadata, per-case external-support/response-source classification and retained evidence identifiers, invokes any candidate binary directly, requires deterministic normalized JSON, and compares JSON structurally. It deliberately does not call the candidate's `--manifest`; a qualification record must pin the candidate bytes and claimed suite identity separately. `checkReferenceProcess` runs canonical validation and negative guards for historical v1 and current v2 suites, but runs candidate controls against the compiled current Lean reference only for v2. Keeping the runner separate from the reference evaluator lets a partial independent interpreter implement only its selected operation without also exposing the reference manifest or Lean-specific response key order.

The flat handover adds a separate checked production bridge rather than another evaluator. [`A12Kernel/Evidence/FlatProtocolBridge.lean`](../A12Kernel/Evidence/FlatProtocolBridge.lean) owns the frozen 0.2.0 typed capability descriptor, selects its retained flat replay inputs, projects each into a normalized protocol request, combines it with the explicit frozen verdict table, and derives the suite and fixtures with external-support/response-source classifications. It deliberately does not import today's decoder or evaluator. The same module owns the separate source-maintainer mutation plan: canonical baselines and unchanged case identities come from the capability, reviewed mutant case verdicts remain explicit typed data, and complete connective deltas are exhaustively derived from the matching historical verdict algebra. `lake exe syncFlatHandover --check` rejects drift; `--write` rejects the historical v1 identity, so a changed shipment must receive a new capability and generator rather than rewrite qualified inputs. This bridge is not a second current semantics authority and its generated testing metadata creates neither kernel nor candidate-execution evidence. The owning frozen [`flat`](IMPLEMENTER-KIT-FLAT-EMPTY-LOGIC.md) and [`correlation`](IMPLEMENTER-KIT-CORRELATION.md) kits record exact capability integrity and qualification gaps; [`PROTOCOL.md`](PROTOCOL.md) owns runner limits.

### Generated differentials use a separate bounded process lane

[`A12Kernel/Differential/Profile.lean`](../A12Kernel/Differential/Profile.lean) defines the closed generated-profile vocabulary, strict decoder, resource and shape bounds, and four-verdict response projection. [`Generated.lean`](../A12Kernel/Differential/Generated.lean) defines the deterministic 52-request flat matrix and checks its frozen 0.2.0 vocabulary with a dedicated structural validator, including exact family sizes, uniqueness, determinism, and structural metrics. A separate explicitly current backward-compatibility audit sends the generated requests through today's public decoder and evaluator and checks their reference distribution; it is not a reconstruction of the historical campaign and may evolve independently from the frozen profile and receipt. These modules introduce neither a parallel historical evaluator nor retained kernel observations.

[`A12Kernel/Differential/Runner.lean`](../A12Kernel/Differential/Runner.lean) is the outer IO driver. It requires clean repository roots at the profile's exact reference and candidate revisions, contains each executable path within its declared repository, hashes the profile, runner, relay, reference, and candidate, and repeats those identity checks after execution. It invokes the two black boxes sequentially for every generated request, applies the same closed response projection to both, and retains all disagreements plus deterministic minimal witnesses. Its non-writing receipt-consistency gate bounded-reads a green agreement receipt and checks its exact profile identity, finite counts, distributions, budgets, claim, admitted platform, and artifact-digest syntax. It does not authenticate the historical execution, rehash the historical platform-specific executables, or reconstruct omitted per-case green transcripts. The resulting claim is deliberately `finiteLeanAccountDifferential`: it is agreement with the pinned Lean account over the generated profile, with no new external-kernel evidence or proof transfer.

The process boundary is project-owned and language-neutral. [`A12Kernel/Process/Bounded.lean`](../A12Kernel/Process/Bounded.lean) starts [`A12Kernel/Process/RelayMain.lean`](../A12Kernel/Process/RelayMain.lean) as a new POSIX process group with null stdin, while the relay feeds the capped request to the candidate and streams both output channels. This preserves the pinned Lean 4.31 process-group kill metadata while avoiding a blocked parent stdin write. Input, each output channel, execution time, cleanup time, aggregate campaign input/output/time, and result bytes are explicit profile bounds. Completion is split into candidate status written through a strict bounded relay record and raw candidate stdout/stderr, so a candidate's nonzero status or diagnostic cannot be confused with relay failure. [`A12Kernel/ProcessTestMain.lean`](../A12Kernel/ProcessTestMain.lean) exercises the lifecycle independently from A12 semantics.

This helper is a cooperative resource boundary for macOS and Linux, not a security boundary. The candidate inherits credentials, working directory, and environment and can deliberately evade or attack the caller's context. The process group is killed before its relay leader is waited on, preserving the leader PID as a reservation against stale process-group-ID reuse; [`LF9`](LEAN-FINDINGS.md#lf9--the-bounded-process-contract-depends-on-pinned-lean-runtime-details) owns that toolchain-sensitive invariant and its re-audit trigger.

The dependency direction is one-way: `A12Kernel/Differential/` may import the reference adapter and `A12Kernel/Process/`, and the IO mains may import all three, but the library, conformance, and trusted proof roots must never import differential or process drivers. The trust audit enforces that negative edge mechanically. This follows Cedar's separation of specification/theorems from differential integration while retaining this project's clean-room and language-neutral subprocess boundary instead of adopting Cedar's in-process FFI.

### Mutation qualification is a separate replayable process lane

The flat Rust mutation qualification turns the typed mutation plan into a closed packet and result protocol without adding another semantic evaluator. The pure shared [`Process/Artifact.lean`](../A12Kernel/Process/Artifact.lean) module owns validated portable paths and SHA-256 file identities because both qualification records and retained capture receipts consume them. [`Packet.lean`](../A12Kernel/Qualification/Packet.lean) and [`MutationResult.lean`](../A12Kernel/Qualification/MutationResult.lean) provide the qualification-specific closed codecs, compatibility and execution-profile identities, command records, captured observations, mutation outcomes, and structural invariants. These declarations perform no filesystem or candidate execution, although the lane consumes the typed capability from the IO-bearing flat artifact bridge. Expected case changes remain sourced from the typed capability and mutation plan; actual observer output is parsed into a distinct `Observation` before it is admitted to a result record.

The IO boundary is deliberately explicit. [`Process/ArtifactTree.lean`](../A12Kernel/Process/ArtifactTree.lean) supplies the shared bounded, portable, symlink-free exact-tree and digest mechanism without knowing capture or qualification schemas. [`RustPacket.lean`](../A12Kernel/Qualification/RustPacket.lean) reads the frozen Rust Git objects without modifying the sibling checkout and exports its complete build-input closure, exact file modes and bytes, an observer-only baseline patch, seven single-mechanism patches, expected observations, command policy, toolchain profile, payload manifest, and source-owned instructions. [`Checker.lean`](../A12Kernel/Qualification/Checker.lean) validates the closed packet and result trees through that shared mechanism, recomputes every digest, checks packet-owned bytes against this source revision, reconstructs the reviewed patches from the frozen natural library plus tracked observer and edit definitions, validates patch application and reversal in a temporary copy, binds actual observer logs to the recorded observations, and compares each restoration inventory with the frozen path-and-byte projection. [`Runner.lean`](../A12Kernel/Qualification/Runner.lean) executes those commands in a disposable packet-baseline copy and captures the actual streams; [`SelfTest.lean`](../A12Kernel/Qualification/SelfTest.lean) exercises that complete replay plus independent rejection guards; [`MutationQualificationMain.lean`](../A12Kernel/MutationQualificationMain.lean) is the command-line boundary. [`Process/Sha256.lean`](../A12Kernel/Process/Sha256.lean) is shared IO-only process support and introduces no new package dependency.

The packet-owned Rust observer is fixture-driven rather than a second hand-written request suite. It reads the frozen canonical conformance index and its eight normalized request fixtures, rejects a missing, additional, reordered, or unexpectedly named case, invokes the candidate's public evaluation entry point, and records the complete verdict algebra where an exercise needs it. The natural gate adds and removes only that observer; each mutant patch combines the observer with one reviewed semantic edit; every phase must finish with the exact frozen path-and-byte inventory after ignoring only the declared build-output directory. This makes the checked unit the shipment, candidate build-input closure, mutation, command stream, captured output, and restoration state togetherŌĆönot a narrative success report.

`sourceExecutedReplay` and `isolatedSessionAttestation` are intentionally different assurance classes. A source-executed replay records commands that this runner actually invoked in its disposable copy. An isolated-session result is a digest-bound, internally consistent attestation returned by another session: the checker establishes that its identities, logs, outputs, and restored inventories agree with the packet, but it cannot reconstruct the external process history from files alone. Neither class is a proof of the Rust implementation, a release approval, external kernel evidence, or a claim beyond the seven mutations and finite capability observations.

This separation follows Cedar's audited architecture: executable specification, definitional validation, and theorem roots live under `Cedar/Spec`, `Cedar/Validation`, and `Cedar/Thm`, while `DiffTest`, unit tests, and `CedarFFI` are separate integration roots. We preserve that dependency direction but do not copy Cedar's in-process differential FFI. A direct kernel FFI would violate this project's clean-room rule, while an FFI to one candidate language would couple the supposedly language-neutral shipment to that toolchain. A subprocess packet keeps the candidate independent and replaceable, and keeps every IO, native execution, and A12-specific attestation claim outside the trusted semantic and theorem roots.

## Proof discipline

- Trusted core: pure definitions accepted as total by Lean's kernel; prefer structural recursion, but use well-founded recursion or pure local `do`/mutation when clearer. No `partial`, `unsafe`, or `IO` in the trusted semantics.
- Maintain one trusted theorem root that imports every trusted proof module. Generate and review its named axiom report in CI, then audit the complete elaborated trusted project environment rather than treating source tokens as the authority. Reject active `sorry`, used or unused project axioms, `unsafe`, unclassified opaque/source-partial declarations, `implemented_by`, `extern`, and every axiom dependency except `propext`, `Classical.choice`, and `Quot.sound`. Lean's generated `_unsafe_rec` helper is admitted only when its parent is a kernel-visible safe total definition with the same type. `native_decide` remains acceptable in explicitly nontrusted executable-check lanes such as conformance and qualification only because those are regression surfaces: it admits success through the native-decision axiom and therefore adds the compiler/runtime to that example's trust base.
- Every project Lean source has one explicit zone: library root, logical, conformance, evidence, reference, qualification, differential, process, trust driver, or IO driver. The logical theorem closure admits only logical sources; the conformance closure only logical plus conformance sources; and [`A12Kernel.lean`](../A12Kernel.lean) only its root plus logical and conformance sources. Evidence, packet binding, registry, differential, qualification, process, reference, trust, and IO support therefore remain outside trusted roots by positive admission rather than a blacklist. An unfamiliar future directory fails classification before it can enter a root closure.
- A root theorem's review surface includes its exact statement, hypotheses, direction, result domain, supported fragment, counterexamples just outside the fragment, and axiom report. Theorem counts and `0 sorry` are not assurance metrics by themselves.
- Keep the executable core free of external Lake package dependencies while that remains practical. If serious proof work needs mature finite-map, permutation, order, rational, or calendar theory, prefer a separate Mathlib-backed proof target over reimplementing a theorem library or burdening `#eval`.
- Current executable target: pinned `leanprover/lean4:v4.31.0` ([`../lean-toolchain`](../lean-toolchain)) with no external Lake package dependencies. Native artifacts still incorporate Lean runtime/toolchain components, including GMP in the official toolchain; [`PRODUCTION-RELEASE.md`](PRODUCTION-RELEASE.md#dependency-approval-and-bundled-runtime-licensing) owns their distribution consequences. A future separate proof or documentation target may add version-pinned dependencies without changing that core contract.
