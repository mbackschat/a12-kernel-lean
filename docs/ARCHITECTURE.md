# a12-kernel-lean — encoding architecture & decisions

This document owns the current concrete Lean representations, dependency direction, module boundaries, and adopted or rejected encoding choices. The project-owned language-neutral semantics remain in [`../spec/`](../spec/); [`PROJECT-DESIGN.md`](PROJECT-DESIGN.md) owns the project and evidence doctrine, [`LEAN-FORMALIZATION.md`](LEAN-FORMALIZATION.md) owns proof and trust practice, [`SOURCES.md`](SOURCES.md) owns source navigation and provenance, and [`A12-DMKITS-SPEC-SYNC-LEDGER.md`](A12-DMKITS-SPEC-SYNC-LEDGER.md) owns outbound reconciliation. Those contracts are assumed here rather than restated.

## Architecture boundary

The executable semantics, elaboration, and theorem roots remain pure Lean library layers. Conformance examples are independent checked witnesses; evidence and reference-process adapters are untrusted transport layers outside the theorem root. This dependency direction is the architectural consequence of the project doctrine: IO, external observations, and a downstream candidate may test the semantic center but never define it.

Every public semantic meaning is version-scoped. [`A12Kernel/Basic.lean`](../A12Kernel/Basic.lean)'s `kernelVersion` records the one modeled external behavior version, while [`A12Kernel/Reference/Support.lean`](../A12Kernel/Reference/Support.lean) owns the current 0.3.0 reference-semantics identity. The source tree evaluates only the current account; the retired 0.2.0 behavior remains recoverable through its [archived identity and exact revisions](archived/REFERENCE-SEMANTICS-0.2.0-AND-RUST-EXPERIMENT.md) rather than a second namespace or live lineage layer. Migrate the semantic namespace to `A12Kernel.V30_8_1` before a second kernel behavior version or stable library API makes the unversioned meaning ambiguous.

The 2026-07-12 external PL-semantics spike is no longer a live input. Where it conflicted with A12 evidence—most visibly an intrinsically typed AST and a proposal to make every numeric value a `{coefficient, scale}` decimal—the A12 boundary won; expression values remain `Rat`, while exact decimal form exists only at the stored-target boundary. The resulting adopted and rejected choices are recorded below.

## Core encoding decisions

### Extrinsic (untyped) AST, not an intrinsically-typed one

The ASTs remain extrinsic: closed inductives with exhaustive evaluators, parametric in an atom representation rather than indexed by schema or A12 value type (per [`../spec/13`](../spec/13-lean-encoding-guide.md)). We **rejected** an intrinsically-typed `Expr (σ : Schema) : Ty → Type`: A12's type system does not encode cheaply into indices (scale can become unknown; literal expandability and the narrow power exception make it richer than one index; `TypeDefinition` legality is only known post-expansion; enum comparability is config-dependent; `Custom` delegates), and the hard semantic content lives in the *value domain and cell observations*, not in type indices — where intrinsic typing would make proofs fight `Eq.mpr`/transport. Static type/scale/scope checking is done by an elaboration pass that produces a core rule with an explicit `WellFormed` boundary; elaboration soundness and the absence of impossible evaluator failures for accepted core rules are proof obligations. Intrinsic typing stays available as a later hardening only if a concrete proof goal demands it. [`ScaleInfo`](../A12Kernel/Core.lean) is `exact Int | unknown`, so stripped integer-literal scales are representable without a parallel type. [`NumericScaleSummary`](../A12Kernel/Elaboration/NumericScale.lean) pairs that scale with the independent constant-expandability capability and owns pure `+`/`−`/`×`/`÷`, power, and unsuppressed exact-comparison checking. [`AuthoredNumericExpr`](../A12Kernel/Elaboration/NumericExpression.lean) retains each decoded literal's scale and explicit grouping instead of inferring either from `Rat`; [`NumericSource.lean`](../A12Kernel/Elaboration/NumericSource.lean) supplies one shared checked atom algebra for Number fields, fixed Base-Year sources, and declaration-admitted temporal field components; [`NumericValidation.lean`](../A12Kernel/Elaboration/NumericValidation.lean) carries the one supported scale-warning suppression explicitly rather than weakening that shared summary. The older flat `SurfaceLiteral.number Rat` remains a deliberately reduced, scale-erasing route rather than being patched with guessed syntax.

### Checked elaboration resolves names and owns field-policy coherence

[`A12Kernel/Elaboration/Flat.lean`](../A12Kernel/Elaboration/Flat.lean) follows the parser–elaborator–core split used by Lean/Lean4Lean and the explicit surface-to-core boundary exemplified by `do` Unchained. Its input is structured and parser-independent; its output is a `CheckedFlatCondition` carrying proofs that the expanded model validated and every core field matches one unique non-repeatable declaration. The supported path subset is intentionally narrow: absolute paths, parent-relative paths without named turning-point labels, and bare lookup in the kernel order declaring group → flag-gated model-wide unique. There is no implicit ancestor tier; parent lookup requires explicit `..` ([`LF6`](LEAN-FINDINGS.md#lf6--bare-name-resolution-is-local-or-global-not-an-ancestor-walk)). The admitted scalar surface includes all six Number comparisons, Boolean/Confirm/String equality/inequality, all four String `Length` ordering operators, presence predicates for Number, Boolean, Confirm, String, Date, Time, and DateTime, six-way temporal field-to-field comparisons, field-to-Date-literal comparisons over an already-classified and decoded literal, and field/`Now` comparison in either authored order. Temporal declarations and Date literals retain their own six-component format policy; direct comparison applies the coarse Base-Year-aware component gate before lowering field, literal, and dynamic `Now` operands to one shared exact-instant evaluator. An omitted-year Date literal requires a configured Base Year, while `Now` additionally requires the opposite format to carry time. Every field operand must pass model admission and partial relevance, and every clock-dependent checked whole-rule evaluation receives an explicit `World`; the low-level core fails closed without one. The core also rejects the kernel-forbidden constant/constant shape by requiring at least one field. Concrete token classification, literal decoding, ISO-String distinction, `Today`, String `Length` equality/inequality, ambiguity, invalid models, repeatable references, other unsupported operators, kind mismatches, and illegal Confirm literals fail closed or remain earlier boundaries.

The same `FlatModel` compiles raw cells into the runtime `FlatContext`, so successful surface evaluation cannot pair a resolved field with caller-invented policy. `CheckedFlatCondition` retains the exact declaring row group as well as model/core legality; surface elaboration stores its input group, while generated-computation desugaring derives the group from the resolved target declaration. Trusted theorems connect every field read by an admitted comparison to its own unique matching declaration, prove that context construction applies exactly that declaration’s `formalCheck` policy, and preserve the declaring group through elaboration; missing or ambiguous IDs become malformed/unknown. The low-level `FlatCondition.evalFull` remains available for isolated semantic laws and can still receive an arbitrary `FlatContext`, so policy coherence is claimed only for the checked surface route.

[`A12Kernel/Elaboration/NumericValidation.lean`](../A12Kernel/Elaboration/NumericValidation.lean) is the first integrated numeric-expression consumer. It accepts two expressions with at least one actual field across them, requires every field atom to be nonrepeatable and in the evaluated row group, admits plain arithmetic including power plus separately audited root Number-field value functions, and rejects general operation-wrapper traversal. Its shared source algebra also admits fixed numeric/Base-Year-date sources, direct Date/DateTime Day/Month/Quarter/Year or Time/DateTime Hour/Minute/Second field components, and Date-only month/year differences over field or Base-Year operands after component/year checking; every such source is fixed scale 0, while only actual temporal fields satisfy validation's field-presence requirement. Rounding and `Abs` remain direct Number-field-only; canonical operand-list `Min`/`Max` folds admit direct Number fields plus at most one direct constant. A one-bit authoring/runtime state enforces that constant limit through lowering, while the literal retains its authored scale and evaluates as a fixed operand. The consumer independently runs authoring and static-scale checks on both authored trees before lowering each exactly once. One closed `NumericValidationOp` dispatch admits all six ordinary operators and the four fixed tolerance ranges without duplicating the checked pipeline: equality and inequality use the exact-scale gate unless the structured input explicitly carries the one supported warning suppression, ordering is scale-exempt, and tolerance deliberately bypasses the gate. Suppression admits differing or unknown derived scales but cannot bypass model, scope, kind, expression-shape, or authoring checks and is absent by default. Runtime reads reuse the flat context's Number and temporal projections: empty becomes zero with the source's fillability, clean values are fixed, and formal invalidity remains distinct. Date difference gives formal causes precedence over empty substitution, maps any remaining empty operand to symmetric zero, and refuses a present legacy-hybrid payload before reaching the decoded-parts core. `Except FormalCause NumericArithmeticOutcome` composes those distinctions with arithmetic domain failure without a new general effect framework; validation maps formal invalidity to `unknown`, pure domain failure to `notFired`, and two values to the selected ordinary or tolerance seam. Number computation consumes the same resolved source enum at its own phase boundary, where empty is zero and the exact formal cause becomes inherited poison. The full-validation content gate runs before either expression. Direct-atom/literal, temporal-component, and date-difference laws prove agreement with the shared lower-level evaluators, while the checked wrapper carries model and core legality rather than claiming parser correctness or unrestricted surface acceptance.

[`A12Kernel/Semantics/Condition.lean`](../A12Kernel/Semantics/Condition.lean) owns the leaf-parametric `ConditionTree`, its shape-preserving map, any/all leaf predicates, Boolean `And`/`Or` fold, and the sole verdict-aware short-circuit evaluator. [`FlatValidation.lean`](../A12Kernel/Semantics/FlatValidation.lean) retains only `FlatConditionLeaf` and aliases `FlatCondition` to that tree, preserving its established construction surface. [`ValidationCondition.lean`](../A12Kernel/Elaboration/ValidationCondition.lean) embeds a complete flat tree without nesting and adds resolved `NumericComparison` leaves with expression-wide reference and relevance traversal. Its checked wrapper accepts only `CheckedFlatCondition`/`CheckedNumericComparison`, retains one exact common row group, rejects mismatched groups before composition, and rechecks the flattened core against the same validated model. Universal laws prove flat verdict/reference preservation and checked model/core coherence. Checked whole-rule assembly remains the next consumer; it must specialize the existing message-emission boundary rather than creating a parallel rule type.

The flat elaboration slice does not implement the complete §10 path language. Quoted concrete syntax, named-ancestor labels, `RuleGroup`, stars, `$`, semantic indices, and parser/renderer preservation remain outside it. The checked one-star correlation elaborator below reuses the same expanded-model and path mechanisms for one deliberately closed repeatable shape; it does not turn either slice into a general authored-condition parser. [`IMPLEMENTATION-MAP.md`](IMPLEMENTATION-MAP.md) owns the live assurance and external-evidence status.

### `Value.num` remains `Rat`; Number targets use a separate exact decimal

The heterogeneous value domain carries exact `Rat` for numeric values and one closed `TemporalValue` payload under `Value.temporal`. Its Date, Time, and DateTime constructors retain exact `Instant` identity together with exactly the decoded date and clock components that kind exposes; date-bearing constructors also retain stored-Gregorian versus constructed legacy-hybrid calendar provenance. Kind is derived from the constructor, so equal instants with different temporal shapes remain different values and no consumer reconstructs a wall label from an instant. `TemporalComponents` remains the shared six-bit static declaration fact stored by `FieldKind.temporal`. Formal checking accepts exactly a matching payload kind after parsing, declared-format admission, and source-specific consistency checks; `FlatTemporalField` carries those static facts through the existing presence/requiredness/model-context path. `FullDate` and `LocalDateTime` remain proof-bearing parser/resolver-stage refinements over the shared decoded parts. This keeps the existing single field-ID-indexed context and excludes per-kind contexts or a parallel temporal document store.

A plain `Rat` loses the *representation* A12 needs (`7` vs `7.00` is a reportable compute delta; `minFractionalDigits` rendering), while `NumericScaleSummary` is authoring metadata rather than a substitute decimal representation. We therefore keep expression values as `Rat` and use an explicit `{unscaled : Int, scale : Nat}` decimal only at the stored-target boundary, so representation equality is stored-form equality while numeric comparison uses rational values at explicit rounding boundaries. [`A12Kernel/Semantics/NumericRounding.lean`](../A12Kernel/Semantics/NumericRounding.lean) owns scale-19 comparison pre-rounding, authored places `0..14`, and target-scale `FLOOR`/`CEILING`/`HALF_UP`; [`A12Kernel/Semantics/NumericArithmetic.lean`](../A12Kernel/Semantics/NumericArithmetic.lean) owns per-node precision-50 `HALF_UP`, division domain failure, and the staged OpenJDK 21 power account. [`A12Kernel/Elaboration/NumericExpression.lean`](../A12Kernel/Elaboration/NumericExpression.lean) checks the plain-arithmetic authoring regions, owns the separately audited direct-field `Round`/`Abs` and direct-field/one-constant `Min`/`Max` shapes shared by checked validation and computation, and performs the single order-sensitive division rewrite into a distinct evaluation tree; the checked `3 * {1 / 3}` counterexample proves this is semantic staging rather than a value-preserving optimizer. [`A12Kernel/Semantics/NumericFillability.lean`](../A12Kernel/Semantics/NumericFillability.lean) carries directional movement through `+`, `−`, `×`, `÷`, and resolved power without letting reciprocal rounding alter a direct quotient. Power reuses the staged value evaluator and adds the kernel's conservative fixedness/base-region/parity table; negative exponents transform the precision-50 reciprocal base and swap exponent directions before that table. Invalid power stays `notEvaluated`, so metadata cannot turn a domain failure into a value. [`A12Kernel/Semantics/NumericComparison.lean`](../A12Kernel/Semantics/NumericComparison.lean) independently normalizes both operands, defines all six direct ordinary operators, and determines omission polarity from movement on either side; its reduced fixed-right entry point remains a specialization for the older flat Number and String-Length consumers. [`A12Kernel/Semantics/NumericTolerance.lean`](../A12Kernel/Semantics/NumericTolerance.lean) remains a distinct fixed-band operation rather than a misleading arithmetic desugaring and owns the closed dispatch used by the checked consumer. General operation-valued wrapper traversal remains future number-stage work.

### Numeric polarity carries directions, not one substituted bit

A numeric value reaching a comparison carries `NumericFillability {canGrow, canShrink}` separately from its exact amount. A filled scalar is fixed in both directions; an empty unsigned Number becomes zero and can only grow; an empty signed Number becomes zero and can grow or shrink; an empty String consumed by `Length` becomes grow-only zero. `NumericComparisonOp.fillCanBreak` asks whether movement on either operand could falsify the currently true condition; `evalFixedRight` specializes the general evaluator with a fixed literal for reduced consumers. This makes empty unsigned `0 != -1` VALUE, because filling cannot move downward to repair the inequality, but empty unsigned `0 != 1` OMISSION, because filling could reach `1`. It also makes empty unsigned `0 >= 0` VALUE while `0 >= emptyUnsigned` is OMISSION: only the right operand can grow and clear the latter firing. A single `given := false` bit cannot express these distinctions; it remains appropriate only inside the current symmetric Boolean/Confirm comparison helper.

The checked numeric seam is now exactly as general as its consumers require: two numeric operands and one closed dispatch over six direct ordinary operators or four fixed tolerance ranges, with no generic expression framework or alternate AST. Plain arithmetic includes power and consumes the existing exponent-scale gate, direct-left nesting rule, staged value evaluator, and conservative fillability table; runtime-invalid integral power reaches the same quiet validation-domain result as zero division. Equality and inequality verdicts are symmetric; their one warning suppression changes admission only and is proved runtime-irrelevant. Ordering keeps authored operand direction; tolerance retains its strict closed band and scale-gate bypass. Repeated references are evaluated independently, so empty `[U] == [U]` is omission-typed in a content-bearing, row-selected instance under the source-established operand-local fillability account rather than being dependency-deduplicated; an entirely empty full-validation row is still stopped by the row gate. That alias consequence remains external evidence pending. Ordinary inequality and tolerance share `numericDifferenceFillCanClose`, preventing their gap-closing formulas from drifting. The flat fixed-right comparison boundary and generated computation-validation mismatches both consume `NumericValidationOp`, so strict and tolerance branches cannot drift at their common runtime seam. [`A12Kernel/Proofs/NumericComparison.lean`](../A12Kernel/Proofs/NumericComparison.lean) proves fixed-right specialization, equality/inequality symmetry, and selected right-direction laws; [`A12Kernel/Proofs/NumericValidation.lean`](../A12Kernel/Proofs/NumericValidation.lean) proves both checked expression evaluations are reachable, reached power values delegate to the shared outcome, the shared domain/formal projection, suppression and unsuppressed scale rules, runtime suppression irrelevance, the row gate, ordinary atom/literal agreement, and tolerance atom/literal delegation; and [`A12Kernel/Proofs/NumericTolerance.lean`](../A12Kernel/Proofs/NumericTolerance.lean) proves tolerance symmetry and boundary laws. Aggregate tails, expression-valued generated validation, and the distinct date/string rules remain later capsules. [`LF10`](LEAN-FINDINGS.md#lf10--numeric-polarity-needs-directional-fillability-not-a-given-bit) records the semantic correction, [`LF24`](LEAN-FINDINGS.md#lf24--direct-number-ordering-uses-the-same-directional-fixed-right-comparison) records the reduced direct-ordering route, [`LF28`](LEAN-FINDINGS.md#lf28--multiplication-fillability-needs-joint-movement-not-only-current-signs), [`LF29`](LEAN-FINDINGS.md#lf29--division-uses-a-reciprocal-only-for-fillability-and-absorbs-domain-failures), and [`LF52`](LEAN-FINDINGS.md#lf52--power-fillability-is-a-conservative-branch-table-not-parity-inference) record arithmetic propagation, [`LF30`](LEAN-FINDINGS.md#lf30--tolerance-normalizes-operands-independently-and-closes-gaps-directionally) records tolerance, and [`LF34`](LEAN-FINDINGS.md#lf34--checked-numeric-validation-is-an-explicit-staged-consumer) records the integrated checked route.

[`A12Kernel/Semantics/NumericComputationResult.lean`](../A12Kernel/Semantics/NumericComputationResult.lean) owns the operation-neutral `value Rat | domainFailure | poison FormalCause` boundary, while [`A12Kernel/Elaboration/NumericComputation.lean`](../A12Kernel/Elaboration/NumericComputation.lean) is its computation-phase producer. Its checked entry point resolves one nonrepeatable Number target and every nonrepeatable Number operand from the shared authored expression tree, rejects a direct target reference at any nesting depth, applies the existing plain division/power authoring scan, and certifies the resolved operation against the model. The exact result-scale gate is default-on; the one explicit warning flag bypasses only that predicate and remains in the checked core. Constant-only operations remain legal. The generic expression traversal is also consumed by numeric validation, and nonrepeatable lookup by stable ID is shared with generated computation validation; no second numeric AST or resolver was introduced. Each resolved atom retains its `FlatFieldDecl`, so the complete lowered tree still defensively rejects a non-Number declaration before reading any cell; an empty wrong-kind cell cannot masquerade as numeric zero, and an earlier poison cannot hide a later structural fault. Empty and required-only Number reads are the real value zero; arithmetic invalidity remains distinct until a later consumer; and ordinary formal invalidity preserves its cause. There is no numeric `noValue`, validation `Verdict`, fillability, or row gate in this expression result. The evaluator lowers once, preflights the whole lowered tree, then evaluates that tree left-to-right. A left poison aborts; a domain-invalid arithmetic value continues far enough for a later poisoned read to remain observable; and a division rewrite can change which poison is reached first, matching the generated Java receiver/argument order over the lowered form. Reached power operands delegate to the shared staged value algorithm, and runtime-invalid integral power maps to the same domain-failure target/dependency route as zero division. After evaluation, a proof that the supplied target policy has the resolved target's scale and signedness prevents cross-target dispatch; the retained flag selects the ordinary or warning-suppressed target entry point. Target-policy construction remains separate because the current flat declaration omits minimum fractional digits. `NumericScaleBinaryOp.evalValues` remains the shared ordinary binary factor while power has its own shared partial evaluator. Operation-valued wrappers, richer target declarations, repeatable scope, table/generated-rule integration, and concrete parsing remain preceding or later boundaries. String and Number do not receive a generic computation-result abstraction because final empty String and empty Number have different laws. [`LF35`](LEAN-FINDINGS.md#lf35--numeric-computation-preserves-domain-failure-and-read-poison-separately) records the proofs, source basis, and remaining authoring/evidence boundary.

[`A12Kernel/Semantics/NumericStoredNumber.lean`](../A12Kernel/Semantics/NumericStoredNumber.lean) owns `StoredNumber { unscaled, scale }`, making `7` and `7.00` structurally distinct while exposing the same rational amount. Ordinary conversion pre-rounds to scale 19, strips only fractional trailing zeros, and pads to the target minimum; its universal coherence theorem proves that representation work preserves exactly the pre-rounded amount. The separate bounded conversion consumes that canonical pre-rounded value, keeps it unchanged at 16 or fewer significant digits, and otherwise `HALF_UP` rounds it to the greatest nonnegative fractional scale allowed by the 16-digit budget. Rendering is normalized dot-decimal text, while model-configured separator materialization remains a document-boundary concern. [`A12Kernel/Semantics/NumericTarget.lean`](../A12Kernel/Semantics/NumericTarget.lean) is the separate pure consumer after static assignment-scale admission. Its ordinary entry point remains fail-closed when natural scale exceeds the target maximum; its explicit warning-suppressed entry point preserves the ordinary fit branch and sends only no-fit values through bounded conversion. Both branches reuse one reduced target check with digit length before signedness. A no-fit result is rejected unconditionally: a reduced target failure supplies the more specific cause, otherwise the suppressed scale mismatch does. The policy still cannot silently claim range, zero, integer-digit, or other target constraints. The target outcome keeps clean no-result, accepted value, payloadful target rejection, no-attempt calculation invalidity, and inherited poison distinct. Delta projection is deliberately later and lossy: exact stored-form equality controls VALUE, rejection is unconditionally ERRORED, while clean no-result, calculation invalidity, and poison all become fresh silence or stale CLEARED. [`A12Kernel/Semantics/NumericApplication.lean`](../A12Kernel/Semantics/NumericApplication.lean) owns the separate exact one-address final state: accepted output yields the exact stored decimal; every no-applied-value outcome empties a present cell in place without creating an absent cell. Its state distinguishes absent, present-empty, and present-value while its delta-prior projection intentionally merges the first two. Typed-equal accepted output has the same abstract final state without claiming a physical rewrite or raw-lexical preservation. [`A12Kernel/Semantics/NumericDependency.lean`](../A12Kernel/Semantics/NumericDependency.lean) is the distinct pre-consumer projection from the full target outcome to `empty | value StoredNumber | poisoned`. It preserves exact accepted form, never exposes a rejected attempt, and intentionally erases poison subclasses because the downstream invalid-field gate exposes only invalid membership. It is not reconstructed from delta or applied state and does not claim a context overlay, expression evaluator, document mutator, or scheduler. [`LF36`](LEAN-FINDINGS.md#lf36--numeric-target-storage-separates-decimal-form-invalidity-and-delta) records target classification, [`LF37`](LEAN-FINDINGS.md#lf37--numeric-application-preserves-placement-but-erases-cause-provenance) records application, and [`LF38`](LEAN-FINDINGS.md#lf38--numeric-dependency-meaning-survives-lossy-application-and-delta) records the dependency distinction.

### Full Date uses separate component, calendar, and value-admission types

[`A12Kernel/Semantics/FullDate.lean`](../A12Kernel/Semantics/FullDate.lean) starts after textual decoding and refines the shared decoded `DateParts` carried by the generic runtime payload. `DateParts` can represent any decoded component triple, `CivilDate` carries proof that those parts form one day in an unbounded positive-era proleptic-Gregorian account, and `FullDate` additionally carries the inclusive A12 value-floor certificate `≥ 1583-10-16`. This prevents an impossible date from reaching chronology and prevents the separate field-value policy from being mistaken for calendar reality: `1500-01-01` is a real civil date but not an admitted `FullDate`, while `1899-12-31` is admitted because the optional 1900 policy is not universal. The admitted `TemporalValue.date` projection retains those decoded parts, their exact resolved instant, and calendar provenance; parser reachability, resolution consistency, and correspondence outside the reachable four-digit full-Date fragment remain upstream claims rather than consequences of the payload type alone.

The first Date operation is only strict `before`, defined as transparent year/month/day lexicographic chronology over admitted values. The Full Date capsule itself does not introduce an ordinal/epoch conversion, a six-operator temporal dispatch, a generic temporal framework, timezone data, or `World`. The separate Date-comparison owner now supplies the six-operator truth and classified validation projection because correct VALUE/OMISSION polarity requires no-value, formal-invalid, and not-given operand provenance. Parsing/format checks, general Date cells, `Date(...)`, Time, zones/DST, broader arithmetic, and checked rule/protocol consumers remain separate semantic boundaries. [`LF41`](LEAN-FINDINGS.md#lf41--calendar-reality-and-a12-date-value-admission-are-different-types) records the durable rationale and exact limits.

[`A12Kernel/Semantics/DateConstruction.lean`](../A12Kernel/Semantics/DateConstruction.lean) is the separate reason-and-verdict boundary for three-part `Date(day, month, year)`. It combines already-checked component availability with calendar reality supplied by a separate resolver and retains `real DateParts | incomplete | unreal | unknown` rather than collapsing every non-value into `Option.none`. `Valid` and `Invalid` are separate verdict projections: their truth projections are strong-Kleene complements, but only `Invalid` distinguishes incomplete OMISSION from present-but-unreal VALUE. Formal unavailability dominates emptiness, and the 1583 value floor remains a later stored/computed-value check. This layer deliberately does not choose between the kernel's model-zone-sensitive hybrid calendar and the reusable zone-free proleptic `CivilDate` account. The separate [`A12Kernel/Semantics/DateConstructionNumeric.lean`](../A12Kernel/Semantics/DateConstructionNumeric.lean) consumer projects direct day/month/year extraction into `value amount notGiven | unavailable`: real components are fixed, incomplete becomes symmetric not-given zero, unreal becomes fixed zero, and unavailable stays cause-free UNKNOWN at the fixed-literal comparison seam. It deliberately does not invent a `FormalCause` that the construction result no longer retains. The theory still does not project date differences, carry reason through legal DateTime operations, or retain legacy-calendar identity through month/year operations. Two- and four-part forms, Base Year, exact formal causes, row orchestration, component lowering, concrete calendar resolution, target admission, and protocol exposure remain outside it. [`LF46`](LEAN-FINDINGS.md#lf46--constructed-date-validity-needs-a-reason-bearing-non-value) records the proof payoff and evidence limit.

[`A12Kernel/Semantics/CivilDateCoordinate.lean`](../A12Kernel/Semantics/CivilDateCoordinate.lean) owns the proleptic-Gregorian whole-day coordinate and checked successor shared by stored-Date arithmetic and DateTime resolution. [`A12Kernel/Semantics/DateComparison.lean`](../A12Kernel/Semantics/DateComparison.lean) closes the six ordinary truth operators over two resolved full Dates by reusing structural identity and the existing strict chronology, then reuses the generic symmetric scalar classification for present values with missing provenance, no value, and formal unavailability. [`A12Kernel/Semantics/DateAggregate.lean`](../A12Kernel/Semantics/DateAggregate.lean) owns the temporal `Min`/`Max` fold after its second real consumer: it skips empty inputs without a synthetic identity, propagates every reached formal unavailability, retains symmetric empty/tail/`Having` provenance, and lets Date, Time, and DateTime supply only their exact selectors. The Date specialization returns the existing full-Date comparison operand. [`A12Kernel/Semantics/DateShift.lean`](../A12Kernel/Semantics/DateShift.lean) starts after integer offset conversion: `AddDays` inverts the coordinate with a bounded 400-year/12-month search, while `AddMonths` and `AddYears` share landing-day primitives but preserve their clamp-versus-February-end distinction; every result re-enters through `FullDate` admission. [`A12Kernel/Semantics/DateDifference.lean`](../A12Kernel/Semantics/DateDifference.lean) uses the same landing conventions to count signed completed months or years, then classifies stored-Date/Base-Year reads as fixed values, symmetric empty zero, exact formal unavailability, or unsupported legacy calendar before the shared numeric consumers project validation and computation outcomes. Parsing, partially-known Date metadata, constructed-Date legacy execution, DateTime wall-time, temporal target formatting, and Date cell effects remain separate.

### Time uses a day coordinate; DateTime keeps wall labels separate from instant identity

[`A12Kernel/Semantics/TimeComparison.lean`](../A12Kernel/Semantics/TimeComparison.lean) compares already-decoded valid `TimeOfDay` values by elapsed whole seconds since midnight. Kernel checking admits only matching temporal component sets, and time-only parsing gives both operands the same implicit date and model-zone anchor, so this coordinate exactly preserves comparison order as a resolved proof boundary. The checked storage path instead consumes the already-parsed exact instant, matching the kernel's common `VkDate` runtime comparison without reconstructing a wall time. [`A12Kernel/Semantics/TimeAggregate.lean`](../A12Kernel/Semantics/TimeAggregate.lean) supplies only the coordinate-based selector and specialization needed to reuse the shared temporal fold for Time `Min`/`Max`; empty identity, unavailable propagation, and symmetric missing provenance remain owned by that fold. Declared-format decoding, AM/PM, concrete raw-text classification, and computation target application remain upstream.

[`A12Kernel/Semantics/DateTime.lean`](../A12Kernel/Semantics/DateTime.lean) reuses `FullDate` and the shared civil coordinate for one selected post-parse whole-second local DateTime boundary. [`A12Kernel/Core.lean`](../A12Kernel/Core.lean) owns decoded `TimeOfDay`, the scalar `{ epochMillis : Int }` `Instant` identity shared with `World.now`, and the admitted `TemporalValue` payload that keeps those representations together without conflating them. Authored temporal values embed through `Instant.ofEpochSecond`, while an injected `Now` may retain a sub-second remainder. `TimeOfDay` proves `hour < 24`, `minute < 60`, and `second < 60`; `LocalDateTime` is an admitted full Date plus that time. UTC resolution deterministically maps the label to an instant through the coordinate. The total-successor theorem proves that every real civil date advances it by exactly one; strict civil chronology raises it, and strict local wall-label chronology is consequently preserved by UTC resolution. The bounded Gregorian coordinate inverse now belongs to [`CivilDateCoordinate.lean`](../A12Kernel/Semantics/CivilDateCoordinate.lean) at its second real consumer, shared by Date day shifting and timezone recurrence selection. These are internal laws of the selected positive-era Gregorian account, not a claim about parser reachability or universal kernel correspondence.

[`A12Kernel/Semantics/DateTimeComparison.lean`](../A12Kernel/Semantics/DateTimeComparison.lean) owns the operation-neutral six-way temporal enum and the exact-`Instant` comparator. Equality and order observe `epochMillis`, so a chained arithmetic result never becomes equal merely because it renders as the same ambiguous wall label as a freshly parsed value, and a whole-second field does not become spuriously equal to a same-rendered `Now` carrying milliseconds. Concrete parsing and zone resolution remain upstream.

[`A12Kernel/Semantics/DateTimeAggregate.lean`](../A12Kernel/Semantics/DateTimeAggregate.lean) supplies the second selector for the shared temporal fold. Declared-format comparability, operand expansion, parsing, zone resolution, computation targets, and checked lowering remain outside this resolved boundary.

### Unified `Verdict`, not a bare fired/notFired outcome

A condition evaluates to `Verdict = notFired | fired (p : Polarity) | unknown` with explicit `conj`/`disj` tables (strong-Kleene on truth; omission-wins-`And`, value-wins-`Or`), merging truth and the spec's `Polarity` into one type. A bare `firedValue | firedOmission | notFired` outcome cannot represent an `unknown` rule result, which is essential because a formal error can leave a whole condition unknown. `K` is retained for expression/predicate-level truth and for stating the exact information-order laws: unknown lies below both definite values, which are incomparable; `And`/`Or` are monotone under that order, so unknown may refine to true or false while an already definite result is stable. The `conj`/`disj` tables have executable `example … := rfl` guards; the algebraic theory still requires quantified proofs ([`A12Kernel/Core.lean`](../A12Kernel/Core.lean)).

### Two-level cell model: `CheckedCell` → `observeCell(Phase)` → `CellObservation`

The earlier three-state sketch (empty ≠ invalid) is refined into `RawCell α`, an invariant `CheckedCell α {rawPresent, parsed, findings}`, and a **phase-indexed read** producing a `CellObservation α` (`empty | value α | unknown cause | poison cause`) ([`A12Kernel/Cell.lean`](../A12Kernel/Cell.lean)). All three types default `α` to the heterogeneous scalar `Value`, so ordinary formal checking, checked temporal presence, and every established caller share one context, while resolved Date, Time, and DateTime consumers can still carry their proof-bearing values through the same generic boundary. `checkRawCellWith` owns absent, present-empty, admitted, and rejected projection once; scalar `formalCheck` supplies kind checking and String normalization, while `checkAdmittedRawCell` is identity admission only for a typed value whose parser and declaration checks have already succeeded. `rawPresent` records physical placement independently of semantic value: absent is `false`, while a present cell with no raw value and a parsed empty String both become `rawPresent = true, parsed = none`; all three empty inputs still read as `CellObservation.empty`. Registered custom field validation adds a distinct project-bearing cause rather than overloading the fixed declarative fallback. [`Semantics/CustomFieldType.lean`](../A12Kernel/Semantics/CustomFieldType.lean) resolves one named pure validator from `World`, derives effective bounds and stored-value context, excludes nonrelevant/empty/prerejected inputs before sampling, and returns the same ordinary checked cell for every consumer. Generated and structural passes annotate that representation later. Requiredness is deliberately staged: evaluate the generated mandatory rule against base checked cells, retain its hit/message, and only on a hit add `.required` to the empty target for authored validation rules. Installing `.required` first would make the mandatory rule's own `FieldNotFilled` read unknown and suppress itself. Ordinary findings surface as `unknown` in validation and `poison` in computation, while `.required` is validation-scoped and observes as ordinary empty during computation. **Empty-substitution never happens in the read** — the consuming operator decides what an empty operand means (number→`0` in `<`, skipped by `Max`/`Min`; string→`""` in concat, ignored by `==`).

`CellObservation.empty` is intentionally unresolved. Field kind supplies a default tier, but the semantic clause consuming the observation may override it by operator and operand position, and the enclosing consumer may still distinguish comparison, concatenation, or assignment. Model-derived field roles act earlier by adding staged findings, while `Phase` decides whether those findings expose unknown, poison, or—in the required-only exception—ordinary empty. Consequently the evaluator must not expose a global `FieldKind → Value` normalization. Each operator family owns named operand-resolution functions that preserve the provenance its consumer needs: simple symmetric clauses may retain a given/substituted bit, while numeric clauses retain directional `canGrow`/`canShrink`. Aggregate semantics state both per-element empty treatment and the all-empty selection identity; computation retains empty/no-value until the store boundary can choose VALUE, CLEARED, or ERRORED. [`LF5`](LEAN-FINDINGS.md#lf5--empty-handling-is-a-layered-consuming-clause-policy-not-a-field-kind-function) records the consuming-clause boundary, and [`LF10`](LEAN-FINDINGS.md#lf10--numeric-polarity-needs-directional-fillability-not-a-given-bit) records the directional refinement.

### Partial validation separates rule gating from reference masking

[`A12Kernel/Semantics/PartialValidation.lean`](../A12Kernel/Semantics/PartialValidation.lean) admits one already-resolved nonrepeatable rule instance plus an explicit `FlatRuleFilterPresence`. Kernel 30.8.1's `.filtered` branch returns `FlatPartialResult.skipped` before error-field relevance or condition evaluation; it records only that the elaborated rule contains a `Having` filter and does not represent or execute that filter. For `.unfiltered`, error-field relevance remains an independent gate. A relevant rule passes its relevance predicate to `FlatCondition.evalSelected`, where an out-of-set comparison or presence leaf returns `Verdict.unknown` before reading the checked cell. Existing `Verdict.conj`/`disj` supply the strong-Kleene behavior.

Call-local relevance is not a formal defect in the stored cell. The evaluator therefore does not append a fake `FormalCause`, manufacture a malformed `CheckedCell`, or widen computation poison. `FlatCondition.evalSelected` owns the one recursive evaluator and defaults to all fields relevant; partial validation supplies its predicate explicitly, so full and partial connectives cannot drift. The distinct `skipped` result covers both the earlier filter gate and the error-field gate without flattening either into evaluated `notFired` or `unknown`. The universal filter law proves condition independence; the `Or` case with only the non-error operand relevant proves why masking alone is insufficient.

`FlatRelevance = FieldId → Bool` is intentionally only a flat field-instance predicate. It cannot represent group coverage, global augmentation, wildcard repetitions, aggregate-specific completeness, uniqueness partners, or phantom rows, and no such framework is inferred. `FlatContext.AgreesOn` states equality only at relevant field IDs. The trusted one-sided theorem proves that an unfiltered partial firing survives in every content-bearing completion satisfying that relation, while existentially quantifying the full firing polarity because hidden values can change VALUE/OMISSION precedence. A relevant unfiltered rule bypasses the full-validation content gate through selected-context evaluation. [`LF23`](LEAN-FINDINGS.md#lf23--partial-validation-skips-filtered-rules-before-two-relevance-decisions) records the mechanism, laws, counterexamples, and evidence limit.

### String support begins at two named consuming clauses

The first String capsule does not add a generic String value policy. [`formalCheck`](../A12Kernel/Semantics/Observation.lean) applies [`normalizeEvaluatedString`](../A12Kernel/Semantics/String.lean) once when an admitted parsed String enters the checked evaluation cache: each existing non-overlapping CRLF pair becomes LF, existing LF and lone CR remain unchanged, and an empty normalized result becomes the same clean-empty cell as absent input. This is evaluation-side content, not a mutation of stored document text. The transformation is deliberately not globally idempotent—one pass maps `"\r\r\n"` to `"\r\n"`, while a second pass would map that result to `"\n"`—so direct comparison, `Length`, and computation reads consume the cached checked value and never normalize independently. The current elaborator admits direct equality/inequality, all four `Length` ordering operators, String presence, and String presence guards in that bounded generated-computation fragment. `Length ==`/`!=` remain rejected because the reduced surface erases authored literal scale; patterns, other functions, repeatable/parent-gated requiredness, line-break permission, enumeration ordering, and raw-type rule elimination are explicitly rejected, fail closed, or remain unimplemented rather than inferred. This capsule remains internal to the library and evidence replay. The normalized reference protocol and existing consumer capabilities do not accept String or Length, so no public transport or release claim grows merely because the core can execute these clauses.

### String computation separates expression, root store, target check, application, and delta

[`A12Kernel/Semantics/StringComputation.lean`](../A12Kernel/Semantics/StringComputation.lean) begins §11 with one unconditional, non-repeatable String computation. `StringExpr` parameterizes only its leaf type so [`Elaboration/StringComputation.lean`](../A12Kernel/Elaboration/StringComputation.lean) can resolve structured field paths into the same runtime tree rather than define a parallel AST. The checked wrapper certifies one validated flat model and nonrepeatable String kind for every resolved field ID, and its raw-input evaluator constructs checked cells through that same model before delegating. `StringExpr.eval` then produces `StringTerm`: a clean field absence is `noValue`, a literal or concatenation is evaluated `text`, and a formal error is `poison`. Concatenation is the consuming clause that turns a clean missing operand into the empty contribution. It may therefore produce `text ""` when every contribution is empty; that intermediate is deliberately not collapsed back to `noValue`.

The root store is a separate total function. It maps both bare `noValue` and final `text ""` to `StringStore.noValue`, maps nonempty text to `produced StoredString`, and preserves poison. `StoredString` proves only that its rendered stored form is nonempty; target legality is intentionally not part of that type because the same value must survive as the attempted payload of `ERRORED`. A low-level wrong-kind cell is an explicit `StringComputationFault`, not another quiet empty case. Checked copy/literal/concatenation lowering now precedes this evaluator; target declaration/policy checking, whole-computation legality, alternatives, and scheduling remain later work.

`StringTargetLengthPolicy.check` is the next boundary. The first closed subset admits either one positive minimum or one positive maximum; zero/nonpositive declarations, combined bounds, patterns, enumerations, line-break permission, and `noValueValidation` remain outside it until separately observed. Input-ingestion normalization does not grant permission for a computed target to contain a line break. A root no-value or poison bypasses target checking unchanged. A produced String with no CR or LF is either `accepted` or `errored attempted cause`; target rejection is an ordinary semantic result, not a `StringComputationFault`, poison, or payload-free flag. A produced String containing CR or LF instead yields `StringTargetCheckResult.unsupported`, so this length-only capsule cannot silently infer the preceding target-permission decision. That result is fail-closed fragment routing, not an external computation outcome. The admitted check uses the shared [`utf16CodeUnitLength`](../A12Kernel/Semantics/String.lean), while the retained conservative-ASCII packet does not establish broader Unicode behavior externally.

`StringTargetOutcome.projectDelta` owns report granularity. Accepted values are change-sensitive, but the single outcome-first `errored` branch always emits `ERRORED(attempted, cause)` for absent, stale, and equal prior targets. It and the pre-target `StringStore.projectDelta` compatibility API share only the lower-level value/no-value delta primitives. The pre-target API never constructs a `StringTargetOutcome`, so an unchecked write attempt cannot masquerade as `accepted`; the two lanes still cannot acquire competing change/clear implementations. `StringTargetOutcome.appliedValue` remains a deliberately lossy value-level projection: accepted exposes its checked value; errored, no-value, and poison expose no stored value.

[`A12Kernel/Semantics/StringApplication.lean`](../A12Kernel/Semantics/StringApplication.lean) adds the separate exact one-address boundary. `StringTargetState` distinguishes `absent`, `presentEmpty`, and `presentValue StoredString`; `applyTo` writes an accepted value, clears an existing target in place for every outcome without an applied value, and never creates an absent target merely to clear it. This is not a general document mutation API: the target is already resolved, ancestor/group existence is fixed, and traversal, repeatable addressing, scheduling, multi-target updates, and raw-text preservation remain outside the capsule. `PriorStringTarget` stays intentionally lossy because delta reporting needs only empty versus filled; the `toDeltaPrior` projection cannot drive exact application. [`A12Kernel/Proofs/StringApplication.lean`](../A12Kernel/Proofs/StringApplication.lean) states an independent transition relation and proves it equivalent to the executable function and deterministic, proves the exact clearing rule and its placement-preservation consequence for outcomes without an applied value, and connects exact stored state to `appliedValue`. The checked non-laws show both directions of lost information: equal deltas need not yield equal exact states, while equal final empty states need not imply equal deltas or outcome provenance.

This decomposition turns the apparent contradictions “empty String concatenates as `""`” versus “a final empty String is not stored,” “equal values are silent” versus “equal invalid attempts still report,” and “no delta” versus “different exact placement” into explicit boundaries. It supplies two refactoring rules: adding a clean missing contribution is store- and delta-preserving although not term equality, and target-check preservation must be stated at the target-outcome, exact-application, or delta boundary rather than only over expression text. [`A12Kernel/Proofs/StringComputation.lean`](../A12Kernel/Proofs/StringComputation.lean) and [`A12Kernel/Proofs/StringApplication.lean`](../A12Kernel/Proofs/StringApplication.lean) state the useful laws and nearest stronger non-laws; [`LF12`](LEAN-FINDINGS.md#lf12--string-computation-has-expression-store-and-delta-boundaries), [`LF13`](LEAN-FINDINGS.md#lf13--string-target-outcome-delta-and-application-are-separate-boundaries), and [`LF18`](LEAN-FINDINGS.md#lf18--exact-target-placement-is-separate-from-value-and-delta) record the rationale and evidence limits.

[`A12Kernel/Semantics/StringCascade.lean`](../A12Kernel/Semantics/StringCascade.lean) composes exactly two resolved steps through an explicit outcome overlay. The producer's checked `StringTargetOutcome`, not its delta or applied-value projection, determines the dependent cell: accepted supplies the stored value, clean no-value supplies an empty cell, target error supplies declared-constraint poison without the attempted payload, and ordinary inherited poison preserves its cause. Validation-scoped `.required` is rejected at the outer `StringDependencyFault` boundary because computation observation intentionally ignores it. The nested `Except StringDependencyFault (Except StringComputationFault StringTerm)` is therefore an intentional separation of dependency binding from the later read, not an accidental API shape. This remains an evaluation overlay rather than document application or a scheduler.

[`A12Kernel/Semantics/ComputationCondition.lean`](../A12Kernel/Semantics/ComputationCondition.lean) introduces the common checked scalar-computation context and the first condition and selection capsule. `ComputationConditionResult` distinguishes `holds`, clean `notTrue`, and exact-cause `poison`; it is neither validation `Verdict` nor `Bool`. Direct `FieldFilled` and `FieldNotFilled` consume `observeCell .computation`: empty and value reverse clean truth, while both predicates preserve poison. This makes required-only empty cleanly not filled and keeps an ordinary finding poisonous even when requiredness is also present. ### Polarity and partial validation are one-sided-sound abstract interpretations — verify, don't assume the `iff`

Partial validation (relevant cells concrete, non-relevant → `unknown`, `true Or unknown` fires while `true And unknown` is suppressed) is a definite-truth abstract interpretation with a one-directional guarantee. This avoids a parallel legacy evaluator in current source and prevents an unchanged response envelope from masquerading as a semantic version selector.

Protocol v1 handles one request per process rather than a stream, service, or FFI. That is the smallest deterministic language-neutral boundary for Kotlin, TypeScript, Rust, shell, and CI consumers; batching or a long-running service remains a measured deployment decision. The process has no general `Document` decoder and exposes only the non-repeatable flat intersection plus the named one-group correlation intersection in its manifest.

[`A12Kernel/CandidateConformanceMain.lean`](../A12Kernel/CandidateConformanceMain.lean) is a separate downstream test product, not another semantics path. It reads a selected language-neutral suite, enforces closed suite/case/evidence objects, validates that suite's complete compatibility identity, finite evidence scope and case composition against the committed support manifest it names, checks fixture metadata, per-case external-support/response-source classification and retained evidence identifiers, invokes any candidate through the shared bounded macOS/Linux relay, requires deterministic normalized JSON, and compares JSON structurally. It deliberately does not call the candidate's `--manifest`; a qualification record must pin the candidate bytes and claimed suite identity separately. `checkReferenceProcess` runs canonical validation, negative guards, and compiled-reference controls for the current V2 suites only. Keeping the runner separate from the reference evaluator lets a partial independent interpreter implement only its selected operation without also exposing the reference manifest or Lean-specific response key order.

The completed 0.2.0 flat handover and Rust qualification are immutable history, not current architecture. Their exact identities, deleted artifact hashes, executed candidate revisions, 52/52 outcome, limits, and recovery revisions are consolidated in the [archived record](archived/REFERENCE-SEMANTICS-0.2.0-AND-RUST-EXPERIMENT.md); the one-off projection, packet, checker, and differential sources remain inspectable at Git revision `03186c1`. Current implementer kits and runnable suites target 0.3.0/V2.

### Candidate conformance uses the shared bounded process lane

The process boundary is project-owned and language-neutral. [`A12Kernel/Process/Bounded.lean`](../A12Kernel/Process/Bounded.lean) starts [`A12Kernel/Process/RelayMain.lean`](../A12Kernel/Process/RelayMain.lean) as a new POSIX process group with null stdin, while the relay feeds the capped request to the candidate and streams both output channels. This preserves the pinned Lean 4.31 process-group kill metadata while avoiding a blocked parent stdin write. The current candidate runner applies a 10-second execution deadline, 1-second cleanup deadline, 1 MiB stdin and stdout caps, and a 64 KiB stderr cap to every invocation. Completion is split into candidate status written through a strict bounded relay record and raw candidate stdout/stderr, so a candidate's nonzero status or diagnostic cannot be confused with relay failure. [`A12Kernel/ProcessTestMain.lean`](../A12Kernel/ProcessTestMain.lean) exercises the lifecycle independently from A12 semantics.

This helper is a cooperative resource boundary for macOS and Linux, not a security boundary. The candidate inherits credentials, working directory, and environment and can deliberately evade or attack the caller's context. The process group is killed before its relay leader is waited on, preserving the leader PID as a reservation against stale process-group-ID reuse; [`LF9`](LEAN-FINDINGS.md#lf9--the-bounded-process-contract-depends-on-pinned-lean-runtime-details) owns that toolchain-sensitive invariant and its re-audit trigger.

The dependency direction is one-way: candidate and process drivers may import the reference adapter and `A12Kernel/Process/`, but the library, conformance, and trusted proof roots must never import process drivers. The trust audit enforces that negative edge mechanically. This follows Cedar's separation of specification/theorems from differential integration while retaining this project's clean-room and language-neutral subprocess boundary instead of adopting Cedar's in-process FFI.

## Proof discipline

- Trusted core: pure definitions accepted as total by Lean's kernel; prefer structural recursion, but use well-founded recursion or pure local `do`/mutation when clearer. No `partial`, `unsafe`, or `IO` in the trusted semantics.
- Maintain one trusted theorem root that imports every trusted proof module. Generate and review its named axiom report in CI, then audit the complete elaborated trusted project environment rather than treating source tokens as the authority. Reject active `sorry`, used or unused project axioms, `unsafe`, unclassified opaque/source-partial declarations, `implemented_by`, `extern`, and every axiom dependency except `propext`, `Classical.choice`, and `Quot.sound`. Lean's generated `_unsafe_rec` helper is admitted only when its parent is a kernel-visible safe total definition with the same type. `native_decide` remains acceptable in explicitly nontrusted executable-check lanes such as conformance because it admits success through the native-decision axiom and therefore adds the compiler/runtime to that example's trust base.
- Every project Lean source has one explicit zone: library root, logical, conformance, evidence, reference, process, trust driver, or IO driver. The logical theorem closure admits only logical sources; the conformance closure only logical plus conformance sources; and [`A12Kernel.lean`](../A12Kernel.lean) only its root plus logical and conformance sources. Evidence, process, reference, trust, and IO support therefore remain outside trusted roots by positive admission rather than a blacklist. An unfamiliar future directory fails classification before it can enter a root closure.
- A root theorem's review surface includes its exact statement, hypotheses, direction, result domain, supported fragment, counterexamples just outside the fragment, and axiom report. Theorem counts and `0 sorry` are not assurance metrics by themselves.
- Keep the executable core free of external Lake package dependencies while that remains practical. If serious proof work needs mature finite-map, permutation, order, rational, or calendar theory, prefer a separate Mathlib-backed proof target over reimplementing a theorem library or burdening `#eval`.
- Current executable target: pinned `leanprover/lean4:v4.31.0` ([`../lean-toolchain`](../lean-toolchain)) with no external Lake package dependencies. Native artifacts still incorporate Lean runtime/toolchain components, including GMP in the official toolchain; [`PRODUCTION-RELEASE.md`](PRODUCTION-RELEASE.md#dependency-approval-and-bundled-runtime-licensing) owns their distribution consequences. A future separate proof or documentation target may add version-pinned dependencies without changing that core contract.
