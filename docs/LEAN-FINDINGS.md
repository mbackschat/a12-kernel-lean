# Lean formalization findings

This is the durable findings ledger for `a12-kernel-lean`, adapted from a12-dmkits' separation between kernel truth, implementation treatment, and live gaps. It records settled conclusions about how this project should formalize, prove, and empirically anchor the read-only semantics. It does not redefine kernel behavior and it is not a backlog.

Each finding has a stable `LF<n>` identifier, date, semantic section, evidence basis, Lean treatment, and explicit limit. Findings are not deleted or renumbered; later corrections amend the record visibly.

## LF1 — differential evidence and proofs answer different questions

> Date: 2026-07-13. Sections: cross-cutting. Basis: project decision; Cedar-style executable-spec/production-engine separation; the local a12-dmkits (`a12-rulekit/`) corpus and differential architecture.

Kernel differential testing is the empirical backbone. Each executable Lean capsule must ultimately be checked against retained portable observations from the real kernel. Proofs establish universal consequences of the chosen Lean definitions; they do not establish that the external kernel implements those definitions. Fuzzing broadens the empirical search but proves neither internal laws nor universal correspondence.

The evidence topology has four distinct roles: the real kernel is behavioral authority; the external a12-dmkits adapter is the kernel-facing probe/export harness; explicitly versioned portable observations retained in this repository are the replay boundary, with sibling corpus cases available as source material; the a12-dmkits interpreter is a clean-room peer for triangulation and divergence discovery, never the oracle. Contrary kernel evidence overrides agreement between Lean and the interpreter.

Lean treatment: every live fragment in [`IMPLEMENTATION-MAP.md`](IMPLEMENTATION-MAP.md) separates internal assurance from external adequacy. A capsule without retained kernel observations is marked `external evidence pending`, even when it builds, executes, and has trusted proofs.

Limit: finite retained observations cannot prove universal kernel equivalence. A wrong Lean primitive can support flawless theorems, and a finite differential suite can miss a divergence; confidence comes from keeping both forms of evidence connected without collapsing them.

## LF2 — checked elaboration must own model and runtime-policy coherence

> Date: 2026-07-13. Sections: §2, §10. Basis: [`../spec/08-paths-and-references.md`](../spec/08-paths-and-references.md), read-only path differentials, Lean/Lean4Lean parser–elaborator separation, and `do` Unchained's explicit surface-to-core transformation pattern.

The flat evaluator originally accepted resolved IDs, typed field metadata, and already checked cells independently. That was a real semantic admission: a caller could pair one identifier with inconsistent kind/scale metadata or check its raw cell under a different policy.

Lean treatment: [`A12Kernel/Elaboration/Flat.lean`](../A12Kernel/Elaboration/Flat.lean) introduces a parser-independent surface subset and an expanded `FlatModel`. Resolution validates the model first, never uses first-match ambiguity, rejects repeatable scope, and lowers only legal scalar conditions into `CheckedFlatCondition`. The wrapper carries both model-validity and core-field well-formedness proofs. `FlatModel.checkContext` applies the same unique declaration policy to raw cells; unresolved or ambiguous IDs fail closed as malformed.

[`A12Kernel/Proofs/Elaboration.lean`](../A12Kernel/Proofs/Elaboration.lean) proves that an admitted field has one matching non-repeatable declaration, that raw-to-checked context construction uses exactly its `formalCheck` policy, and that lookup failure observes as validation `unknown .malformed`. This closes policy coherence for the checked surface route. The low-level core evaluator remains intentionally usable with an arbitrary `FlatContext` for isolated semantic laws, so the stronger claim does not apply to arbitrary direct calls.

Supported flat path treatment, corrected by LF6: structured absolute paths, plain parent-relative paths, and bare lookup in the kernel order declaring group → flag-gated model-wide unique name. Parent/ancestor lookup requires explicit `..`; it is not a hidden bare-name tier. A later narrow amendment reuses the declaration resolver and parent walker for one group-qualified direct-child star and correlated `Having`: [`A12Kernel/Elaboration/Correlation.lean`](../A12Kernel/Elaboration/Correlation.lean) requires an explicitly declared repeatable group path, exact singleton scope, Number kind, supported comparison, both origins, and operator-specific scale legality before producing `CheckedSingleCorrelatedRule`. This is one closed lowering route, not general star or `$` syntax.

External limit, amended three times on 2026-07-13: [`ShortNameRefDiffTest`](../../a12-rulekit/adapter/src/test/kotlin/io/github/mbackschat/a12/dm/adapter/laws/ShortNameRefDiffTest.kt) and [`NamedAncestorPathDiffTest`](../../a12-rulekit/adapter/src/test/kotlin/io/github/mbackschat/a12/dm/adapter/laws/NamedAncestorPathDiffTest.kt) establish broad kernel-backed provenance. The separating flat bundle covers explicit parent match/mismatch, absolute lookup, declaring-group precedence, unique model-wide fallback, duplicate rejection, and flag-disabled rejection. The correlation-elaboration bundle adds one exact star path and three static rejection classes plus an accepted mixed-scale ordering control. LF6 records the bare-name correction. Concrete parsing/quoting, named turning points, `RuleGroup`, semantic indices, nested or multiple stars, and general repeatable paths remain outside the admitted fragment rather than untested claims within it.

## LF3 — `$` correlation requires two explicit environments

> Date: 2026-07-13. Sections: §9, §10, §12. Basis: [`../spec/07-repetition-and-iteration.md`](../spec/07-repetition-and-iteration.md), [`../spec/01-data-model.md`](../spec/01-data-model.md), read-only kernel codegen/runtime source, a12-dmkits findings and interpreter treatment from the `a12-rulekit/` checkout, and retained kernel differentials.

The semantic mechanism is an explicit two-environment filter. A rule begins with a captured repetition environment `outerEnv`. A starred operand enumerates candidate rows and constructs an `innerEnv` for each candidate. Inside `Having`, ordinary references resolve against `innerEnv`, while `$`-marked references resolve against `outerEnv`; correspondingly, `CurrentRepetition(G)` reads the candidate row and `CurrentRepetition($G)` reads the captured outer row. `$` is correlation, not positional selection.

The kernel source exposes the same mechanism at its data-flow boundary. [`IterationState.java`](../../a12-kernel/kernel-rt/kernel-core-runtime/src/main/java/com/mgmtp/a12/kernel/core/rt/_30_8/internal/util/IterationState.java) carries the captured outer state, [`EbenenIterator.java`](../../a12-kernel/kernel-rt/kernel-core-runtime/src/main/java/com/mgmtp/a12/kernel/core/rt/_30_8/internal/util/EbenenIterator.java) pairs it with each candidate while evaluating the filter, and [`RtInternalIdentifier.java`](../../a12-kernel/kernel-rt/kernel-core-runtime/src/main/java/com/mgmtp/a12/kernel/core/rt/_30_8/internal/util/RtInternalIdentifier.java) chooses captured or candidate indices from the reference's outer marker. [`RuntimeController.java`](../../a12-kernel/kernel-rt/kernel-core-runtime/src/main/java/com/mgmtp/a12/kernel/core/rt/_30_8/internal/core/RuntimeController.java) resolves an empty Number to zero with fillability metadata and obtains `CurrentRepetition` from the resolved identifier's last repetition index; the document API states that repetition indices are 1-based in [`PathPart.java`](../../a12-kernel/kernel-md/kernel-md-document-v2/src/main/java/com/mgmtp/a12/kernel/md/document/apiV2/PathPart.java). A formally invalid field is not check-relevant, and [`BedingungsOperatorHelper.java`](../../a12-kernel/kernel-rt/kernel-core-runtime/src/main/java/com/mgmtp/a12/kernel/core/rt/_30_8/internal/core/BedingungsOperatorHelper.java) maps a comparison reading such an operand to the kernel's collapsed false-or-unknown result. [`CodeGenFilterCondition.st`](../../a12-kernel/kernel-tool/kernel-core-codegen-condition/src/main/resources/internal/templates/validation/java/conditionsDir/CodeGenFilterCondition.st), [`CodeGenCondition.java`](../../a12-kernel/kernel-tool/kernel-core-codegen-condition/src/main/java/com/mgmtp/a12/kernel/core/codegen/internal/generator/backingbeans/conditions/CodeGenCondition.java), and [`ConditionSuffix.st`](../../a12-kernel/kernel-tool/kernel-core-codegen-condition/src/main/resources/internal/templates/validation/java/conditionsDir/ConditionSuffix.st) show that a generated filter retains only a known true result, so the resulting candidate is dropped.

Star binding is inseparable from this mechanism. Bindings are retained only above the first star; a same-group star reopens that group and spans all its instantiated rows rather than silently collapsing to the current row; a multi-star path binds at the first star. The initial Lean reference meaning must be a straightforward candidate-row filter, not the interpreter's equality-bucket/hash-join optimization.

The most important edge cases are already exposed by the read-only evidence estate:

- Self-match is real. `[inner] == [$outer]` includes the outer row unless the authored filter explicitly adds `CurrentRepetition(G) != CurrentRepetition($G)`; the evaluator must not infer self-exclusion. See [`CorrelationLawsTest`](../../a12-rulekit/adapter/src/test/java/io/github/mbackschat/a12/dm/adapter/laws/CorrelationLawsTest.java) and [`OuterCorrelationDiffTest`](../../a12-rulekit/adapter/src/test/kotlin/io/github/mbackschat/a12/dm/adapter/laws/OuterCorrelationDiffTest.kt).
- Empty and malformed Number keys separate cleanly. Empty uses comparison-local zero at either origin; a malformed inner key drops only that candidate, while a malformed captured outer key drops every candidate whose comparison reads it. The checked neighboring non-law in [`Conformance/Correlation.lean`](../A12Kernel/Conformance/Correlation.lean) is that numeric `notEqual` is not Boolean negation of numeric `equal` on a malformed operand: both yield Lean `K.unknown` and keep neither row.
- Filtering precedes consumed-cell classification. A malformed consumed cell in a dropped row is not read; one in a kept row may make the consumer unknown. See [`SelfCorrelatedOverlapQuantifierDiffTest`](../../a12-rulekit/adapter/src/test/kotlin/io/github/mbackschat/a12/dm/adapter/laws/SelfCorrelatedOverlapQuantifierDiffTest.kt) and [`MultiStarFilteredCountDiffTest`](../../a12-rulekit/adapter/src/test/kotlin/io/github/mbackschat/a12/dm/adapter/laws/MultiStarFilteredCountDiffTest.kt).
- Validation and computation have different consequences inside the filter. Validation unknown drops a candidate because the filter is not true; computation reads contribute to dependency and poison behavior. See [`HavingOverComputedDependencyDiffTest`](../../a12-rulekit/adapter/src/test/kotlin/io/github/mbackschat/a12/dm/adapter/laws/HavingOverComputedDependencyDiffTest.kt).
- Polarity is not a simple consequence of selected values: filtered quantifiers/comparisons conservatively escalate, while filtered counts have an operand-dependent exception. See [`HavingQuantifierPolarityDiffTest`](../../a12-rulekit/adapter/src/test/kotlin/io/github/mbackschat/a12/dm/adapter/laws/HavingQuantifierPolarityDiffTest.kt), [`HavingStarComparisonPolarityDiffTest`](../../a12-rulekit/adapter/src/test/kotlin/io/github/mbackschat/a12/dm/adapter/laws/HavingStarComparisonPolarityDiffTest.kt), and [`FilteredCountPolarityDiffTest`](../../a12-rulekit/adapter/src/test/kotlin/io/github/mbackschat/a12/dm/adapter/laws/FilteredCountPolarityDiffTest.kt).
- Evidence transport must preserve the raw stored condition because the documented kernel formatter defect can lose a `$`-bearing conjunct; see [`KERNEL-FINDINGS.md`](../../a12-rulekit/docs/KERNEL-FINDINGS.md). Every retained correlation case therefore records and byte-checks the actual stored condition that executed rather than trusting canonical or display output.

Lean consequence: represent selection as an ordered candidate-row filter with explicit inner and captured-outer origins, keep only known-true filters, and apply the selected consumer afterward. Checked lowering must establish group, scope, origin, kind, operator, and scale legality before constructing that core; an optimized join may be added only with a refinement proof against the straightforward selector. [`ARCHITECTURE.md`](ARCHITECTURE.md) owns the concrete current representation, [`IMPLEMENTATION-MAP.md`](IMPLEMENTATION-MAP.md) owns live support and proof status, and [`EVIDENCE.md`](EVIDENCE.md) owns the exact retained witnesses.

Limit: the durable mechanism does not imply general repeatable paths, nested or multiple stars, cross-group execution, general consumers, computation poison, or filtered-result polarity. It also does not expose the hidden kernel truth value behind an observed suppressed message. Those axes require separate capsules and evidence rather than expansion of this finding's current implementation narrative.

## LF4 — Cedar is the primary Lean engineering precedent; Radix is a scoped secondary check

> Date: 2026-07-13. Sections: cross-cutting. Basis: filesystem audit of Cedar specification revision `3977eb4f017b421b7ac0b31ea4635e1dd36ce3ef` and Radix revision `617b67eb09681ca98e19759b48978866dcafeb17`.

Cedar's actual source tree, not merely its project description, is the best default for this repository's Lean engineering shape. `Cedar/Spec` contains ordinary executable semantic definitions with explicit result/error domains and observable short-circuit order; `Cedar/Validation` is separate from execution; `Cedar/Thm` is a recursively aggregated theorem hierarchy; unit tests, symbolic tests, differential interfaces, CLI/FFI, and protobuf transport are separate libraries or executables. Its Lake linter recursively checks that the theorem root imports every proof file. The differential/FFI boundary may use `unsafe` and timing/IO, while the semantics and theorem layers remain pure. Its style guide favors minimal sorted imports, theorem docstrings, explicit types, `simp only` for stable rewrites, and explicit proof steps.

Lean treatment adopted here: keep `Semantics`, `Elaboration`, `Proofs`, and `Conformance` distinct; maintain one mechanically complete proof root; state residual non-success outcomes rather than claiming accepted input cannot fail; keep portable differential transport outside the trusted semantics; and inspect the complete elaborated trusted environment plus its dependency closure for trust escapes. This repository's explicit axiom and compiled-behavior audit is intentionally stronger than Cedar's import-completeness linter alone; LF11 records the mechanism.

The first iteration capsule applies Cedar's primary pattern concretely: the executable `SingleStar.select` stays in `Semantics`, the ordered `SelectRows` relation and exact two-way bridge stay in `Proofs`, and separating fixtures stay in `Conformance`. The `sumSelected_filter_before_consumer` theorem is the small reachability/noninterference result this slice can honestly support: after selection, consumer-cell agreement is required only on selected rows. This is preferable to inventing a full read-footprint framework before computation or partial validation needs one. Radix remains only a secondary confirmation that an evaluator/relation bridge is useful when the relation exposes independent structure.

Radix is useful when a compact technique needs confirmation. Its `BigStep` relation is genuinely independent from its fuel-based interpreter, and it proves fuel monotonicity plus soundness/completeness for successful terminating runs. Its optimization passes pair transformations with forward semantic-preservation theorems. Its top-level `Slides.lean` imports the real Radix library through Verso, so presentation examples elaborate against live code. These are valuable patterns only with their exact limits: the relation omits failure/divergence outcomes, the optimization results are forward preservation rather than unrestricted equivalence, and the slides dependency tracks an unpinned branch in the audited project.

Decision rule: inspect Cedar first for a new Lean architectural or proof-engineering choice. Use Radix to validate a small relational/evaluator/transformation or checked-exposition pattern when Cedar does not provide a compact example. Do not adopt either project's headline theorem counts, dependency choices, or theorem wording without reading the exact source, result domain, and hypotheses.

## LF5 — empty handling is a layered consuming-clause policy, not a field-kind function

> Date: 2026-07-13; amended 2026-07-15. Sections: §2, §3, §4, §6, §7, §8, §9, §11, §12. Basis: a12-dmkits' kernel-verified operator catalog and structural checks, focused differential tests, retained operator-sensitive capture, and read-only kernel source inspection.

The expert rule “empty input depends only on field type” is a useful baseline for direct comparisons, but `only` is false. The catalog has exactly eight kind defaults: Number is `AS_ZERO`, Confirm is `AS_FALSE`, and Boolean, String, Enumeration, Date, Time, and DateTime are `NOT_EVALUATED`. It also has 31 operators carrying empty-operand deviation blocks: 36 separate claims because five operators distinguish operand positions, backed by 39 executable probes. Every default and deviation claim has at least one probe. [`CATALOG-SPEC.md`](../../a12-rulekit/docs/CATALOG-SPEC.md#4-the-emptyoperand-dimension) defines this as a deviation-only layer and explicitly separates empty operands from invalid/not-check-relevant values.

The decisive same-kind counterexamples are independently visible in tests and source. An empty String suppresses a direct comparison, while `Length(empty String)` produces fillable zero and can fire `0 < 5`; the kernel source explicitly says that `Length` returns zero instead of unknown ([`EmptyValueLawsTest.java`](../../a12-rulekit/adapter/src/test/java/io/github/mbackschat/a12/dm/adapter/laws/EmptyValueLawsTest.java), [`BigDecimalUtils.java`](../../a12-kernel/kernel-rt/kernel-core-runtime/src/main/java/com/mgmtp/a12/kernel/core/rt/_30_8/internal/core/BigDecimalUtils.java)). An empty Date suppresses direct comparison, while the date/time extractors and `DifferenceIn*` functions return fillable zero for an empty operand ([`EmptyOperandFiringDiffTest.kt`](../../a12-rulekit/adapter/src/test/kotlin/io/github/mbackschat/a12/dm/adapter/laws/EmptyOperandFiringDiffTest.kt), [`BedingungsOperatorHelper.java`](../../a12-kernel/kernel-rt/kernel-core-runtime/src/main/java/com/mgmtp/a12/kernel/core/rt/_30_8/internal/core/BedingungsOperatorHelper.java)). Value-list membership, overlap predicates, and count/aggregate families further distinguish subject versus list-entry positions, `NOT_EVALUATED` versus `SKIPPED` versus `EVALUATES_FALSE`, and per-element treatment versus the identity of an all-empty selection.

Field kind and operator are still insufficient. An empty string-like result can suppress a direct comparison, contribute `""` inside concatenation, and produce `CLEARED` when stored as a standalone computation; [`FieldValueAsStringComputeDiffTest.kt`](../../a12-rulekit/adapter/src/test/kotlin/io/github/mbackschat/a12/dm/adapter/laws/FieldValueAsStringComputeDiffTest.kt) pins the latter two contexts, while [`CalculationController.java`](../../a12-kernel/kernel-rt/kernel-core-runtime/src/main/java/com/mgmtp/a12/kernel/core/rt/_30_8/internal/core/CalculationController.java) rejects a no-value string at the computation store boundary. An index field is not another empty kind: its model role generates requiredness and uniqueness findings, so an empty or duplicate index cell becomes not-check-relevant before an aggregate or lookup consumes it ([`IndexAggregateDiffTest.kt`](../../a12-rulekit/adapter/src/test/kotlin/io/github/mbackschat/a12/dm/adapter/laws/IndexAggregateDiffTest.kt), [`IndexedReadSuppressionDiffTest.kt`](../../a12-rulekit/adapter/src/test/kotlin/io/github/mbackschat/a12/dm/adapter/laws/IndexedReadSuppressionDiffTest.kt)). Phase then controls whether findings expose validation unknown, computation poison, or—in the required-only carve-out—the underlying empty. A clean empty itself is not phase-normalized; it stays empty until a consumer interprets it. Row eligibility is an additional outer gate: the comparison defaults are probed in a content-bearing row because an entirely empty non-repeatable row may not be evaluated at all.

The evidence qualification matters. [`CatalogFacetProbesTest.java`](../../a12-rulekit/adapter/src/test/java/io/github/mbackschat/a12/dm/adapter/laws/CatalogFacetProbesTest.java) executes every catalog empty probe against the real kernel's Groovy-dynamic runtime and checks model acceptance, firing, and optional polarity. It does not run all 39 probes through the static-Java strategy. Separate differential tests tri-check both strategies for selected high-risk mechanisms, so the defensible claim is complete dynamic-runtime probe coverage of the catalog claims, not universal two-strategy coverage of every catalog cell.

Lean treatment: keep `CellObservation.empty` unresolved at the shared read boundary. Never expose a universal `FieldKind → Value` empty normalization and never make `Phase` an empty-substitution table. Each operator family owns named resolution clauses for the relevant operand positions and preserves the provenance its consumer observes: a symmetric clause may need given/substituted, while numeric polarity needs independent grow/shrink directions. Functions may turn empty into a concrete derived value such as fillable zero; otherwise no-value remains explicit so a comparison, concatenation, or store can make its own decision. Aggregates specify two independent clauses: per-element empty treatment and all-empty selection identity. Model roles add staged findings during elaboration/document passes rather than mutating the scalar kind policy.

The current design now demonstrates this boundary rather than merely preparing for it. [`Observation.lean`](../A12Kernel/Semantics/Observation.lean) leaves a clean empty as `.empty` and normalizes a parsed empty String to that same state; [`formalCheck_empty_observes_empty`](../A12Kernel/Proofs/Observation.lean) locks non-normalization at the phase read. [`FlatValidation.lean`](../A12Kernel/Semantics/FlatValidation.lean) then gives direct String equality and `Length` different resolvers: the former is not evaluated, while the latter becomes grow-only zero. [`StringLength.lean`](../A12Kernel/Proofs/StringLength.lean) proves that same-field operator distinction, and the retained six-case capture separates it externally. Direct Number comparison now delegates to directional fillability rather than treating every substituted firing as omission; LF10 owns that refinement. This remains narrow coverage: String presence/requiredness, String inequality, other Length operators, and the rest of the deviation matrix remain rejected or open. Each future deviation still needs a paired witness, a checked false generalization, and retained kernel observation.

Limit: the catalog deliberately leaves some construction and conversion families uncharacterized. The 31-operator inventory is therefore a verified snapshot of recorded deviations for kernel 30.8.1, not a proof that every other operator follows the kind default. Unknown matrix cells remain open evidence obligations rather than inferred defaults.

## LF6 — bare-name resolution is local-or-global, not an ancestor walk

> Date: 2026-07-13. Sections: §10. Basis: kernel parser source, two new kernel-confirmed rejection probes, retained positive local/model-wide cases, and a12-dmkits parser treatment.

The earlier three-tier account—declaring group, then nearest ancestors, then model-wide unique fallback—was falsified. It came from the read-only [`spec/08-paths-and-references.md`](../spec/08-paths-and-references.md) and made the first Lean elaborator internally consistent but externally wrong. The original positive “bare ancestor” witness was underdetermined: its field name was globally unique, so both an ancestor walk and model-wide fallback selected the same field.

The kernel mechanism has two tiers. [`CheckIdNameRefImpl.java`](../../a12-kernel/kernel-tool/kernel-core-parser/src/main/java/com/mgmtp/a12/kernel/core/parser/internal/check/entity/CheckIdNameRefImpl.java) first constructs exactly `declaringGroup/name` and accepts that entity if present. On a miss, a non-absolute reference may use the flag-gated model-wide short-name index: one match resolves, multiple matches raise `MVK_FIELDNAME_NOT_UNIQUE`, and no match raises `MVK_INVALID_ENTITY`. It never walks ancestor groups. The a12-dmkits parser documents and implements the same local-candidate-then-global rule in [`Parser.kt`](../../a12-rulekit/interpreter/src/commonMain/kotlin/io/github/mbackschat/a12/dm/interpreter/parse/Parser.kt).

The separating evidence holds all other relevant structure fixed. A bare `Limit` in a nested group with `Limit` at two ancestor levels is rejected as [`MVK_FIELDNAME_NOT_UNIQUE`](../evidence/kernel-30.8.1/diagnostics/bare-duplicate-ancestors-rejected.json), instead of choosing the nearest. A second model with one unique ancestor but short-name fallback disabled is rejected as [`MVK_INVALID_ENTITY`](../evidence/kernel-30.8.1/diagnostics/bare-unique-ancestor-flag-disabled-rejected.json), instead of walking upward. Conversely, a same-named declaring-group field wins even when an ancestor has that name, and a unique cross-group name resolves when the flag is enabled; both positive cases are retained runtime observations.

Lean treatment: [`FlatModel.resolveBare`](../A12Kernel/Elaboration/Flat.lean) now checks only `declaringGroup/name` before the flag-gated model-wide uniqueness query. The unused ancestor-search helpers were removed. [`Conformance/Elaboration.lean`](../A12Kernel/Conformance/Elaboration.lean) locks the local win, globally unique fallback, duplicate-name rejection, and flag-disabled rejection. Explicit parent-relative resolution remains unchanged and is covered by matching and mismatching retained observations.

This finding overrides LF2's original path-order wording and records a known discrepancy in the read-only semantic input. Under the repository discipline, `spec/` is not edited here; kernel 30.8.1 remains authoritative, and every writable design/status surface uses the corrected two-tier rule.

## LF7 — a malformed `Having` filter drops its row before consumption

> Date: 2026-07-13. Sections: §3, §9. Basis: retained focused observation [`having-malformed-filter-drops.json`](../evidence/kernel-30.8.1/cases/iteration/having-malformed-filter-drops.json), direct agreement of kernel 30.8.1 Groovy-dynamic and static-Java strategies, and the retained a12-dmkits triangulation split.

For rows `(Weight, Count) = (malformed, 100), (1, 3), (1, 4)`, the rule `Sum(Items*/Count Having Weight == 1) == 7` fires. The malformed Weight still produces its formal-error message, but its row-local filter comparison is not true and the row is dropped before Count is consumed; the remaining selected counts sum to seven. Both kernel strategies produce the authored OMISSION message and the formal error.

The current a12-dmkits interpreter reports only the formal error for this witness. The exact external split is retained in [`triangulation/having-malformed-filter-drops.json`](../evidence/kernel-30.8.1/triangulation/having-malformed-filter-drops.json). This does not weaken the Lean evidence: by LF1 the interpreter is a peer for triangulation, not the oracle, and the two agreeing kernel strategies determine the observation. It is recorded explicitly rather than hidden, dropped from the capsule, or encoded as a kernel-strategy divergence in the standard case format.

Lean treatment: `SingleStar.select` evaluates the filter first and `Verdict.keepsHaving` keeps only `.fired`; validation `.unknown` therefore drops the row. `sumSelected_filter_before_consumer` proves the broader internal boundary that dropped-row consumer cells are irrelevant once selection agrees. The neighboring retained pair changes only the malformed consumer row's Weight from `2` to `1`: dropped means the authored rule fires, kept means the sum is unknown and the authored rule is silent.

Limit: the retained external message boundary establishes firing versus silence and the observed OMISSION signature, not the kernel's hidden internal representation of the malformed filter as Lean `Verdict.unknown`. This finding does not diagnose or fix the a12-dmkits interpreter's internal cause, and this repository must not modify that sibling project.

## LF8 — reusable Lean domains and kernel-backed public domains are different boundaries

> Date: 2026-07-13. Sections: §5, §9, cross-cutting protocol/evidence. Basis: cold-handover audit of the checked correlation API, all 12 retained runtime correlation projections, and the first language-neutral candidate suite.

The reusable correlation core deliberately accepts any positive duplicate-free candidate IDs and scans them in list order. That is a useful total semantic abstraction, but it is not harmless generality at the public kernel-correspondence boundary: `CurrentRepetition` compares the numeric `RowIndex` itself, while selection order comes separately from the list. Gapped or reordered IDs can therefore change comparison truth, not merely rename rows. Every retained kernel correlation observation uses actual contiguous one-based repetition indices `[1..n]`; no retained case establishes arbitrary positive IDs as kernel documents.

The same distinction appears in Number transport. The Lean account resolves exact rational values and applies scale-19 `HALF_UP` comparison, and the normalized decoder accepts canonical exact decimals. The runtime correlation projection schema transports integers, and the retained cases exercise only non-negative integer cells. Static evidence establishes selected declared-scale legality, but it does not establish dynamic negative, fractional, or scale-19 rounding behavior inside correlation.

Lean treatment: keep the reusable semantic and checked-elaboration domains general where that aids proofs and later evidence, but narrow the normalized `singleGroupCorrelation.firingRows` operation to non-empty contiguous candidate sequences. Machine-readable support metadata and the implementer kit state that general exact-decimal and signed correlation inputs are Lean-account behavior with external evidence pending. The handover methodology has two conceptual lanes: the finite candidate suite of retained cases and broader Lean differential/property generation outside that suite; it must never relabel the second lane as new external evidence.

This is not an argument to weaken internal types until they match today’s corpus. It is a rule for public support claims: a representation generalization becomes a released kernel-semantic capability only after its observable consequences are researched and retained. A future probe may close negative/fractional values or show that a different document adapter supplies row identity; until then, downstream implementers receive an exact answer and an exact claim class rather than being asked to repeat kernel research.

## LF9 — the bounded process contract depends on pinned Lean runtime details

> Date: 2026-07-15. Sections: cross-cutting process and differential testing. Basis: the pinned Lean 4.31 process API documentation and native runtime implementation, [`A12Kernel/Process/Bounded.lean`](../A12Kernel/Process/Bounded.lean), [`RelayMain.lean`](../A12Kernel/Process/RelayMain.lean), and the adversarial [`ProcessTestMain.lean`](../A12Kernel/ProcessTestMain.lean) gate.

Lean 4.31 documents that `IO.Process.Child.kill` terminates the whole POSIX process group when the child was spawned with `setsid := true`. The pinned native implementation makes that behavior depend on metadata stored in the Lean child handle. Its `Child.takeStdin` path returns a replacement handle without preserving the `setsid` bit, even though the operating-system process remains its session leader. A parent that spawns a candidate with piped stdin, calls `takeStdin`, and later calls `kill` can therefore kill only the candidate PID while leaving descendants alive. The public API shape alone does not reveal this liveness failure.

Lean treatment: the bounded parent starts a project-owned relay with null stdin, piped output, and `setsid := true`, so it retains the original child handle and its group-kill metadata. A capped request is placed in an automatically removed temporary file; the relay starts the candidate in the same process group, feeds its stdin, and streams candidate stdout and stderr without buffering them itself. Only after candidate stdin, exit, and both output streams complete does the relay overwrite that file with a strict completion record containing a magic value and canonical decimal candidate exit code. The parent bounds and parses that record independently, so candidate status and diagnostics cannot be mistaken for relay status.

Cleanup order is a safety invariant, not an implementation detail. Starting `child.wait` too early can reap the relay leader before the process-group kill; the operating system may then reuse its PID as another process-group ID, making a later group kill capable of targeting an unrelated group. The bounded parent instead treats EOF on both relay output pipes as the completion signal, kills the owned group while the unreaped relay leader still reserves its PID, and only then starts the bounded wait. It applies the same kill-before-wait sequence on timeout, stream failure, output overflow, and success, which also prevents a same-group descendant that closed its inherited streams from surviving an otherwise successful invocation.

The result is a finite resource boundary, not hostile-code containment. Input, stdout, stderr, execution, cleanup, poll, aggregate campaign, and result limits are explicit and checked, but the candidate runs with inherited credentials, working directory, and environment. A deliberately hostile child can inspect or modify anything available to that user, create a new session to escape the process group, or otherwise attack the host. The supported contract is cooperative POSIX execution on macOS and Linux; it is not a Windows contract and must not be described as a sandbox.

Toolchain consequence: every change to [`lean-toolchain`](../lean-toolchain), and every proposed extension beyond the declared macOS/Linux profile, must re-audit the process API and native `spawn`, `takeStdin`, `kill`, and wait implementation before the bounded-runner claim is reused. Then rerun `lake exe checkBoundedProcess` on every supported platform, including timeout, blocked input, exact and overflowing outputs, invalid UTF-8, nonzero status separation, and both success- and failure-path descendant cleanup. A compatible type signature or successful ordinary child execution is not enough to preserve this contract.

## LF10 — numeric polarity needs directional fillability, not a given bit

> Date: 2026-07-15. Sections: §2, §5, §7, §12. Basis: [`spec/10-validation-and-polarity.md`](../spec/10-validation-and-polarity.md), read-only kernel and a12-dmkits polarity mechanisms, the retained six-case operator-sensitive capture, red/green conformance, and trusted directional-polarity proofs.

The earlier flat evaluator attached only `given : Bool` to a comparison operand and classified every true comparison over a substituted value as OMISSION. That loses the reason polarity exists: OMISSION means filling could clear the currently true error condition, so the evaluator must know which directions filling can move the value. The bug is visible while holding the substituted value and operator fixed. An empty unsigned Number reads as zero and can grow but cannot shrink: `0 != -1` is therefore VALUE because filling cannot reach `-1`, whereas `0 != 1` is OMISSION because filling could reach `1`. An empty signed Number can move in either direction, so its `0 != -1` is OMISSION. A filled zero is fixed and makes each true inequality VALUE.

The String deviation exposes the same directional mechanism after a different consumer. Direct equality over an empty String is not evaluated, but `Length(empty)` is numeric zero whose future value can only grow. Consequently `Length(empty) < 5` fires as OMISSION, while `Length(empty) >= 0` fires as VALUE. The six retained cases hold the models and three-rule inventories fixed across empty, filled, and empty-row documents; both kernel strategies and a12-dmkits agree on the complete external message lists. This establishes the named observable outcomes, not every internal branch or every String and Number operator.

Lean treatment: [`A12Kernel/Semantics/NumericComparison.lean`](../A12Kernel/Semantics/NumericComparison.lean) separates exact `Rat` amount from `NumericFillability {canGrow, canShrink}`, owns shared scale-19 normalization, and computes whether an available left-operand direction can falsify a firing against a fixed literal. Filled scalars use `fixed`; empty unsigned Number and empty `Length` use `growOnly`; empty signed Number uses both directions. `NumericComparisonOp` intentionally contains only `equal`, `notEqual`, `less`, and `greaterEqual`, the union needed by the closed consumers. The theorem [`growOnlyNotEqualWhenLeftNotBelow_is_value`](../A12Kernel/Proofs/NumericComparison.lean) states the reusable law; conformance locks both `-1` and the predicted second instance `-2`, so a special-case patch for one literal would fail.

This finding corrects the one-bit Lean modelling note in [`spec/03-empty-and-required.md`](../spec/03-empty-and-required.md): its `given` bit remains a useful simplified account for symmetric Boolean/Confirm clauses, but it is insufficient for numeric polarity and must be read together with the superseding directional account in [`spec/10-validation-and-polarity.md`](../spec/10-validation-and-polarity.md). The public flat operation already admitted Number inequality, so the semantic correction changes the result of an accepted request such as empty unsigned Number `!= -1`; no public String/Length surface or versioned Rust capability was added. The frozen Rust capability does not admit this case and remains unchanged. The compatibility decision therefore freezes reference semantics 0.2.0 and its v1 artifacts, advances the current account to 0.3.0 with v2 suites, and retains the OMISSION-versus-VALUE separating replay instead of silently rewriting history.

Limit: the current polarity helper has one fillable left expression and a fixed literal. It does not establish general two-operand propagation, arithmetic or aggregate fillability, date symmetry, concatenation flags, filtered polarity, partial validation, or the UTF-16 astral boundary externally. Those remain separate consuming-clause and evidence obligations.

## LF11 — trust boundaries must inspect Lean's elaborated environment, not source spellings

> Date: 2026-07-15. Sections: cross-cutting proof and executable trust. Basis: adversarial Lean 4.31 declaration probes, the audited Cedar separation between specification/theorems and compiler/FFI integration, and the project trust-gate review.

A line-oriented ban on `axiom`, `unsafe`, `partial`, or selected attributes is useful only as defense in depth. Lean permits declaration modifiers, same-line and multiline attributes, late `attribute` commands, and command macros. A macro can therefore expand an innocent-looking custom command into an axiom, while `implemented_by` can give compiled execution a different body from the kernel-visible definition and `extern` can route it to foreign code. A theorem-root `#print axioms` list also cannot expose an unused project axiom. These are mechanism-level gaps, not reasons to add another list of spelling exceptions.

Lean treatment: [`A12Kernel/Trust/Environment.lean`](../A12Kernel/Trust/Environment.lean) audits the `ConstantInfo` values owned by every imported trusted project module. It rejects any project `axiomInfo`, `isUnsafe` declaration, unclassified `opaqueInfo`, non-generated partial declaration, `implemented_by` mapping, `extern` entry, or dependency outside `propext`, `Classical.choice`, and `Quot.sound`. The opaque rule deliberately rejects source `partial` wrappers and bodyless inhabited stubs; a future legitimate opaque abstraction needs an explicit policy decision rather than passing accidentally. Lean also creates partial `_unsafe_rec` compiler helpers for total recursive definitions, so the audit admits that exact shape only when its parent is a kernel-visible safe `defnInfo` with the identical type. Treating every `isPartial` constant as authored `partial` would reject ordinary total recursion and would misunderstand Lean's compiled representation; accepting the suffix alone would let a forged helper borrow an unsafe or unrelated parent.

[`scripts/check-lean-trust.sh`](../scripts/check-lean-trust.sh) still verifies proof-root imports, the named human-readable theorem registry, source closure, and forbidden dependency directions. Its adversarial probes elaborate attributed and macro-generated axioms, unsafe and partial declarations, late compiler/foreign attributes, `sorry`, `native_decide`, and opaque stubs, then require the environment audit to reject each for the predicted reason. An accepted control contains those words only as String data and uses an ordinary non-substituting attribute plus the reviewed logical axioms. Conformance remains outside the trusted environment audit because its explicit `native_decide` examples are regression locks whose compiler/runtime trust is already classified.

Limit: this audit establishes the declared Lean project trust policy for the pinned toolchain; it is not a proof of Lean itself or of native compiler correctness. Toolchain upgrades must revalidate the environment APIs, generated-recursion classification, and adversarial probes. Source scanning remains deliberately non-authoritative, and the trust driver itself stays outside the trusted roots it inspects.

## LF12 — String computation has expression, store, and delta boundaries

> Date: 2026-07-15. Sections: §2, §7, §11. Basis: [`spec/09-computations.md`](../spec/09-computations.md), read-only kernel and a12-dmkits computation mechanisms, the retained String-computation packet inventoried in [`EVIDENCE.md`](EVIDENCE.md), red/green conformance, and trusted evaluator/store/delta proofs.

An empty String does not have one context-free computation result. A bare field copy reads as clean no-value. Concatenation consumes the same missing operand as the empty contribution `""`, so an empty source plus `"-X"` evaluates and stores `"-X"`. But the fact that concatenation evaluated does not make an all-empty final text storable: `[Source] + [Other]` over two empty fields reaches evaluated text `""`, then the root String store treats that final empty text as no calculation. A stale target is therefore `CLEARED`; an absent target remains silent. This refines the broader table in [`spec/09-computations.md` §2](../spec/09-computations.md#2-separate-operations-and-the-three-outcomes), whose “STRING in a concatenation → VALUE” entry is correct at the operand-consuming expression layer but not a guarantee that an all-empty final root result becomes a stored `VALUE`.

The 13 retained cases separate direct copy, suffix concatenation, and two-field concatenation while varying filled versus empty source, stale versus absent target, and content-bearing versus independently empty row. Groovy-dynamic and static-Java kernel strategies agree on every complete target-delta list. At the retained 699e-era revision the a12-dmkits interpreter agrees in ten cases but reports `VALUE|` for the three all-empty variants where the kernel clears or stays silent. That exact raw result is mechanically projected to kernel delta granularity and retained as triangulation, not hidden or used as the oracle. The mismatch identifies one reusable implementation rule: final computed empty String must be rejected at the root store boundary even though empty remains a legal intermediate concatenation contribution. a12-dmkits later confirmed and fixed that rule as IF123; the correction strengthens current triangulation but never rewrites the historical packet.

Lean treatment: [`Semantics/StringComputation.lean`](../A12Kernel/Semantics/StringComputation.lean) separates `StringTerm` expression results, `StringStore` root decisions, and `StringDelta` reporting against a typed prior target. `StoredString` makes an empty stored target unrepresentable. Computation-phase field reads preserve clean absence versus poison, concatenation is left-to-right and short-circuits after a left poison, while an empty left contribution still reads a poisoned right operand; root storage alone collapses evaluated `text ""` to no-value. The trusted proof spine connects `CheckedCell` observation to the library evaluator, proves empty contribution on both sides, term-level associativity with poison order, store and delta preservation under a missing contribution, final-empty rejection, typed change reporting, and the nearest stronger non-law: missing is not an identity at the intermediate term layer. This makes root-store or delta preservation—not raw expression equality—the safe boundary for a String refactoring. Term-level associativity alone does not authorize reassociating `StringExpr` trees across effectful reads; that future transformation needs an evaluator-level preservation theorem with explicit context assumptions.

Limit: the external packet contains only unconditional, non-repeatable, clean String inputs and unconstrained String targets. It directly establishes final `VALUE`/`CLEARED`/silence deltas, not Lean's intermediate term/store states. Poison and its immediate clear projection are source-grounded internal account behavior but are not distinguished by this packet. LF13 subsequently closes only the positive one-bound String target-length and value-only application slice; general target constraints, alternatives, dependency overlays and cascades, repeatable scope, implicit validation, non-ASCII strings, and general concatenation formatting remain separate capsules.

## LF13 — String target outcome, delta, and application are separate boundaries

> Date: 2026-07-15. Sections: §3, §7, §11. Basis: [`spec/09-computations.md`](../spec/09-computations.md), read-only kernel and a12-dmkits computed-result mechanisms, the retained nine-case [String target-validation projection](../evidence/kernel-30.8.1/string-target-validation-projection.json) and [capture receipt](../evidence/kernel-30.8.1/captures/string-target-validation-2026-07-15.json), red/green conformance, and trusted target-check/delta/application proofs.

A nonempty root String write attempt is not yet a successful computation result. The target's reduced computed-result check classifies that attempt as accepted or errored before reporting compares it with the prior target. An accepted value equal to the prior typed value remains a clean accepted outcome but produces no change delta. A target error is different from both clean no-value and computation poison: it retains the attempted stored String and a cause, and `ERRORED` is reported unconditionally whether the prior target was absent, stale, or already equal to the invalid attempt. Equality suppression therefore belongs only to accepted `VALUE`, never to `ERRORED`.

The retained packet holds the copy expression and target address fixed across positive `maxLength` 4 versus 3 models, with absent, stale, and equal prior-target cases; adds exact positive `minLength` 5 boundary and short controls; and adds a padded `" " + [Source] + " "` result accepted exactly at `maxLength` 5. Groovy-dynamic and Java-static kernel strategies agree on all nine complete rich computation results, change-only deltas, and exact applied-target observations. The a12-dmkits interpreter agrees on all nine projected deltas and stored-value application views. The equal accepted case is still present in the kernel's clean-result collection with `changed = false`, while the equal invalid case remains payloadful `ERRORED`; the padded control establishes for this admitted ASCII result that leading and trailing blanks survive the reduced target check rather than receiving the baseline input-field blank rejection. a12-dmkits remains triangulation, not evidence authority. F11 records the old compact-signature gap; the maintained `compute-observation-v1` boundary now transports rich per-runner observations with explicit unavailable fidelity instead of pretending the compact route is lossless.

Lean treatment: [`Semantics/StringComputation.lean`](../A12Kernel/Semantics/StringComputation.lean) inserts `StringTargetLengthPolicy`, fail-closed `StringTargetCheckResult`, and `StringTargetOutcome` after `StringStore` and before delta projection. The admitted policy is unconstrained, one positive minimum, or one positive maximum. Produced text containing CR or LF is `unsupported` because this capsule does not yet model the preceding line-break-permission and CRLF-normalization clause; admitted no-line-break text is accepted or errored. `StringTargetOutcome.errored` and `StringDelta.errored` both carry the attempted nonempty `StoredString` and the language-neutral `tooShort`/`tooLong` cause. Accepted results retain the established change-sensitive delta rule, while errors bypass prior-target equality. `appliedValue` deliberately exposes only whether a stored value remains: accepted yields that value, while no-value, poison, and error yield none. The proof spine establishes strict violations, inclusive acceptance on both permitted sides, exact boundaries as corollaries of those inclusive laws, universal no-value and poison bypass, accepted-outcome preservation of the unconstrained delta, attempt/cause preservation, unconditional error reporting, and the nearest stronger non-law: equal value-only application results do not imply equal deltas because no-value can be silent where `ERRORED` remains visible. The retained packet externally exercises exact boundaries and violations but not strict permitted-side values, no-value/poison bypass, or line-break routing, so those remain internal laws or fail-closed boundaries rather than kernel-correspondence claims.

Limit: the kernel-correspondence packet is a positive, one-bound-only String-length capsule over nonempty conservative ASCII values. The internal length theory is broader over nonempty text with no CR or LF, but that broader domain is not externally established; CR/LF fails closed until the earlier line-break clause is modeled. The capsule does not admit a zero bound, simultaneous minimum and maximum, patterns, enumerations, charset/newline checks, the general ordering or interaction of target constraints, `noValueValidation`, other target kinds, target rendering beyond the retained String forms, or the supplementary-plane UTF-16 boundary externally. The kernel packet retains that applying an error to an absent target leaves it absent while applying one to a filled target leaves an empty target instance, but Lean currently models only the common value-only result and makes no exact document-shape claim. No-value and poison bypass are proved and conformance-locked internally but are not exercised by this target packet. Initially present-empty targets, poison and dependency cascades, preconditions, alternatives, computation ordering, repeatable targets, implicit validation, checked model/path lowering, and public protocol exposure remain open.

## LF14 — CRLF normalization is an ingestion boundary, not merely a target-length helper

> Date: 2026-07-17. Sections: §2, §7, §11. Basis: the a12-dmkits IF124 kernel-source and dual-route probe handback, its separate IF125 review discriminator, and the current fail-closed Lean String target boundary. This is upstream-confirmed knowledge; no local retained CR/LF packet yet establishes Lean correspondence.

Kernel 30.8.1 normalizes CRLF to LF when an input value enters evaluation, before the normalized value populates the evaluation caches. The effect therefore reaches every later consumer of that evaluated String, including computations and length checks; it is not a local cleanup performed only by one target validator. Lone CR remains unnormalized. The same investigation fixed a distinct clause-order issue in which line-break handling precedes enumeration-domain validation. A separate IF125 discriminator found that a raw-type `Length(nvv) > c` rule can be eliminated during code generation rather than evaluated with an ordinary empty or normalized operand. These are different mechanisms and must not be collapsed into one “String length” branch.

Lean treatment: the existing target-length capsule continues to reject produced text containing CR or LF as `unsupported`. That conservative boundary is intentional until retained portable observations bind pre-normalized input, evaluation-ingestion output, target attempt, structured cause, and applied state. A future line-break capsule should model normalization as an explicit earlier phase and let target checks consume the normalized value; it should pair CRLF with LF and lone CR, permitted with forbidden line breaks, enum membership with newline rejection, and ordinary fields with the raw-type code-generation discriminator. The current ASCII proofs remain valid because their domain excludes all three line-break forms.

Limit: IF124 and IF125 are confirmed and guarded in current a12-dmkits, but that project is still knowledge and transport rather than the oracle. Until their complete kernel observations are returned and replayed here, they justify the fail-closed design and future scenario matrix, not a new Lean correspondence claim or expansion of the supported String domain.
