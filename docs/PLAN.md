## Current spiral

The project now follows the controlled-interleaving and risk-first method in [`PROJECT-DESIGN.md`](PROJECT-DESIGN.md#controlled-interleaving-and-coverage-horizons). This is interleaving between completed vertical capsules, not parallel partial implementation, and baseline/risk pairs are a default heuristic rather than a quota. Reuse an existing baseline, keep a selected pair together unless blocked, and reassess frontier order after the pair. Every retained semantic unit closes and commits independently.

1. **Date and DateTime sequence:** the pure full-Date value, Gregorian validity, value floor, and strict chronological comparison baseline is closed. Next, as a separate baseline capsule, add the typed distinction for local DateTime under UTC and one instant-preserving `AddHours` path. Then spike the Europe/Berlin autumn-ambiguity/instant-identity family, including its chained-`AddHours` separator; exclude the spring gap, `Today`, other arithmetic, and broader timezone history. Model configuration owns the selected zone; the injected world supplies external time and any explicitly assumed timezone-rule version.
2. **Message and whole-rule language boundary:** first add one narrow post-fire interpolation capsule over the closed emitted-message boundary. Next add one parser-independent authored whole-rule/model lowering that consumes the closed condition and resolved-message semantics, reusing rather than rebuilding the numeric and correlation elaborators. Then select one syntax-shape legality discriminator. Each is a separate semantic capsule; do not build a general parser, generic lowering framework, publication surface, or custom-oracle runtime.

The separate power-fillability risk remains parked: the zero-base/empty-exponent discriminator is known, the required kernel observation is unavailable, and no experimental code remains. Do not repeat it or build a harness; resume only when the observation arrives or a new route is explicitly approved.

Before writing Lean for an ambiguous risk, require enough source or retained-observation support to choose the account. If the discriminator is unavailable, record the competing accounts, smallest required observation, and downstream consequence once, add no code, and continue with the next ready frontier. If two consecutive risks park this way, choose one evidence-ready ordinary capsule or propose one batched evidence request; any new harness or governance mechanism still requires an explanation of need, gap, payoff, cost, and alternatives plus explicit approval.

Consumer probes are capability-specific, may fail usefully, and never imply support from overall Lean coverage or implementation language. A named task is internally usable only when every required semantic branch is closed or rejected; shipment-ready only when the task profile is research-closed with its decision procedure or preservation relation, examples, evidence limits, laws/non-laws, support declaration, and applicable task interface aligned; and independently qualified only after a cold consumer completes the task without kernel, a12-dmkits, or undocumented context. Product and release support remain later gates. The historical Rust result validates this transport shape for one finite retired evaluator capability, not current or general interpreter support.

At each probe checkpoint, consult the full catalog in [`USE-CASES.md`](USE-CASES.md) and select only the one or two task profiles whose dependency cuts were materially improved; do not turn ten categories into parallel workstreams or a quota.
