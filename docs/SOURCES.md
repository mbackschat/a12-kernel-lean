# Authoritative sources & how to drill

This is the repository's provenance and drill hub from semantic topics to the sources that verify or deepen them. It owns project-local source navigation and the historical trail behind the project-owned [`../spec/`](../spec/) body, but it is not a second semantic specification or corrections ledger. [`../spec/SEMANTICS-MAP.md`](../spec/SEMANTICS-MAP.md) is the starting taxonomy; outbound reconciliation of spec changes is owned by [`A12-DMKITS-SPEC-SYNC.md`](A12-DMKITS-SPEC-SYNC.md). Authority order: **[`../../a12-kernel`](../../a12-kernel) (the engine — ground truth) → [`../spec/`](../spec/) → [a12-dmkits (local `a12-rulekit/` checkout)](../../a12-rulekit) (peer clean-room + verified knowledge)**. Everything here is pinned to kernel **30.8.1**. Read to learn behaviour; never link, call, or transcribe the engine (see [`../CLAUDE.md`](../CLAUDE.md)).

**The drill chain**, per semantic area (`§n` = the shared 14-section taxonomy): our `spec/NN-*.md` (understand) → a12-dmkits [`KERNEL-SEMANTICS.md §n`](../../a12-rulekit/docs/KERNEL-SEMANTICS.md) (the peer project's detailed semantic account) → [`KERNEL-FINDINGS.md §n`](../../a12-rulekit/docs/KERNEL-FINDINGS.md) (edge cases + the oracle that established each) → the catalog `semantics` facet if one exists (machine-readable + runnable probes) → the `corpus/` family if one exists (replayable `expected` signatures) → the `interpreter` `commonTest` classes (portable property/unit tests to mirror as Lean theorems) → if you need the mechanism, the kernel runtime class. a12-dmkits' own [`docs/SEMANTICS-MAP.md`](../../a12-rulekit/docs/SEMANTICS-MAP.md) is the **guard-checked hub** for the middle of that chain — follow it for the exhaustive per-`§n` test list rather than duplicating it here.

## `../../a12-kernel` — the engine (ground truth)

A codegen + runtime: the DSL compiles to Java/TS/Groovy that **calls runtime helper classes**, so the evaluation semantics live in the helpers, not the generated code.

**Where the semantics are** — `kernel-rt/kernel-core-runtime/src/main/java/com/mgmtp/a12/kernel/core/rt/_30_8/internal/`:

- `core/BedingungsOperatorHelper.java` — operator evaluation: numeric/date/string comparison, the four fixed tolerance bands with independent scale-19 operand normalization and directional inequality polarity, date arithmetic (`addiereJahre`/`addiereMonate` with Feb-28 / leap corrections), date difference, date-part extraction.
- `core/ValidierungsErgebnis.java` — the **3-valued truth model**: `TRUE_WF` (value error), `TRUE_AF` (omission error), `FALSE_OR_UNKNOWN`, with `combineUND`/`combineODER` (the Kleene AND/OR tables).
- `core/{Number,Date,FirstValue}Combiner.java` — aggregation over iteration + the empty→0 machinery.
- `util/VkBigDecimal.java` — decimals: `DEFAULT_SCALE=19`; one `MathContext(50)` used by `+`, `−`, `×`, `÷`, and `^`; div-by-zero→`INVALID_NUMBER`; negative power computes the precision-50 reciprocal before positive power; exact directional fillability formulas for ordinary arithmetic; conservative `powCanChange*` dispatch rather than exact reachability; empty sentinels. The positive-power operation order comes from the OpenJDK 21 [`BigDecimal.pow(int, MathContext)` contract](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/math/BigDecimal.html#pow(int,java.math.MathContext)): X3.274 binary exponentiation at enlarged working precision followed by the destination round.
- Decimal rounding: `BigDecimalUtils.java` owns scale-19 `HALF_UP` pre-rounding followed by target-scale `FLOOR`/`CEILING`/`HALF_UP`; `CheckOpUtils.java` accepts authored places `0..14`; `RundenOpCreator.java` and `RundenOperOpCreator.java` lower omission to zero for the field-value and expression forms.
- `util/DreiWertBool.java` (three-valued boolean), `util/DateUtil.java` (`clearTime` = Now/Today truncation).
- Full Date admission and direct comparison: [`ValidationDateParser.java`](../../a12-kernel/kernel-rt/kernel-core-runtime/src/main/java/com/mgmtp/a12/kernel/core/rt/_30_8/internal/util/ValidationDateParser.java) strictly parses one declared full-Date format in the model zone; [`FormatDefinitionDatum.java`](../../a12-kernel/kernel-rt/kernel-core-runtime/src/main/java/com/mgmtp/a12/kernel/core/rt/_30_8/internal/formatdef/FormatDefinitionDatum.java) separates syntax/civil validity, optional 1900 policy, and the universal floor from [`Constants.java`](../../a12-kernel/kernel-rt/kernel-core-runtime/src/main/java/com/mgmtp/a12/kernel/core/rt/_30_8/internal/util/Constants.java); [`BedingungsOperatorHelper.java`](../../a12-kernel/kernel-rt/kernel-core-runtime/src/main/java/com/mgmtp/a12/kernel/core/rt/_30_8/internal/core/BedingungsOperatorHelper.java) compares two non-null decoded dates chronologically. The focused 2026-07-19 audit at kernel revision `cb66e51fa7ab90b650698f861bf670754e2e1e66` is the source basis for [`LF41`](LEAN-FINDINGS.md#lf41--calendar-reality-and-a12-date-value-admission-are-different-types); absent/formal operands, formats, and zones remain outside that first Lean capsule.
- Decoded UTC DateTime and the `AddHours` runtime core: [`InternalDefaultDataFormats.java`](../../a12-kernel/kernel-conversion-java/src/main/java/com/mgmtp/a12/utils/localization/internal/InternalDefaultDataFormats.java) fixes canonical internal whole-second `yyyy-MM-dd'T'HH:mm:ss`; the strict [`ValidationDateParser.java`](../../a12-kernel/kernel-rt/kernel-core-runtime/src/main/java/com/mgmtp/a12/kernel/core/rt/_30_8/internal/util/ValidationDateParser.java) and [`FormatDefinitionDatum.java`](../../a12-kernel/kernel-rt/kernel-core-runtime/src/main/java/com/mgmtp/a12/kernel/core/rt/_30_8/internal/formatdef/FormatDefinitionDatum.java) apply model-zone decoding and the value floor; [`RuntimeController.java`](../../a12-kernel/kernel-rt/kernel-core-runtime/src/main/java/com/mgmtp/a12/kernel/core/rt/_30_8/internal/core/RuntimeController.java) routes decoded DateTime and hour addition; [`BedingungsOperatorHelper.java`](../../a12-kernel/kernel-rt/kernel-core-runtime/src/main/java/com/mgmtp/a12/kernel/core/rt/_30_8/internal/core/BedingungsOperatorHelper.java) converts the numeric amount, applies `Calendar.HOUR_OF_DAY` in the model zone, and returns the resulting Date instant; [`VkBigDecimal.java`](../../a12-kernel/kernel-rt/kernel-core-runtime/src/main/java/com/mgmtp/a12/kernel/core/rt/_30_8/internal/util/VkBigDecimal.java) supplies truncation through `BigDecimal.intValue`. The focused 2026-07-19 audit at kernel revision `cb66e51fa7ab90b650698f861bf670754e2e1e66` grounds [`LF42`](LEAN-FINDINGS.md#lf42--datetime-arithmetic-needs-scalar-instant-identity-apart-from-wall-labels). The first Lean capsule starts after numeric conversion and models UTC whole-second resolution plus the instant shift only; large-offset cross-runtime behavior, operand provenance, rendering, and result admission remain outside.
- `core/{MainValidatorController,CalculationController,CalculationCommand}.java` — validation + computation orchestration; caches under `core/`. `CalculationController.handleBerechnetenWert(VkBigDecimal, id)` is the numeric target-store root: scale-19 pre-round, uncapped ordinary fit rendering, warning-suppressed no-fit rendering, reduced target check, and calculation invalidity. `CalculationCommand` strips calculated fields' stored inputs before the pass; `CalculationCache` records accepted target-rendered values but throws before value or presence access for an invalid computed target; the generated `CodeGenCalculationAlternative` catch marks the reading computation invalid. Together these are the source basis for clean-empty versus poisoned dependency observation without attempted-value or error-subclass propagation.
- `DocumentComputationResultImpl` → `FieldInstanceImpl.fieldValuesEqual` — change-only Number deltas compare the stored decimal objects with scale-sensitive `BigDecimal.equals`; numerically equal `7` and `7.00` are therefore distinct stored forms.
- Modern V2 application: [`DocumentComputationResultImpl.applyTo`](../../a12-kernel/kernel-md/kernel-md-runtime-service/src/main/java/com/mgmtp/a12/kernel/md/rt/a12internal/service/DocumentComputationResultImpl.java) dispatches cleared, errored, and changed instances through [`ComputedFieldInstanceImpl.applyTo`](../../a12-kernel/kernel-md/kernel-md-runtime-service/src/main/java/com/mgmtp/a12/kernel/md/rt/internal/service/ComputedFieldInstanceImpl.java) and [`GroupDocCommonImpl`](../../a12-kernel/kernel-md/kernel-md-document-v2/src/main/java/com/mgmtp/a12/kernel/md/document/internal/service/implV2/immutable/GroupDocCommonImpl.java). The shared mechanism yields present-value for VALUE, empties a present target for CLEARED/ERRORED, and never creates an absent target solely to clear it; the deprecated mutable V1 route is outside this project's account.
- Iteration and correlated `Having`: [`IterationState.java`](../../a12-kernel/kernel-rt/kernel-core-runtime/src/main/java/com/mgmtp/a12/kernel/core/rt/_30_8/internal/util/IterationState.java) retains the complete repetition-index vector and captured outer state, [`EbenenIterator.java`](../../a12-kernel/kernel-rt/kernel-core-runtime/src/main/java/com/mgmtp/a12/kernel/core/rt/_30_8/internal/util/EbenenIterator.java) evaluates each filter candidate with that capture attached, and [`RtInternalIdentifier.java`](../../a12-kernel/kernel-rt/kernel-core-runtime/src/main/java/com/mgmtp/a12/kernel/core/rt/_30_8/internal/util/RtInternalIdentifier.java) switches an outer-marked reference to the captured state before resolving every named path level. The surrounding iteration machinery is `util/{EntityIterator,KontextIterator,EbenenIterator}.java`; [`RuntimeController.java`](../../a12-kernel/kernel-rt/kernel-core-runtime/src/main/java/com/mgmtp/a12/kernel/core/rt/_30_8/internal/core/RuntimeController.java) owns Number reads and `CurrentRepetition`. This full-vector route is the source basis for the diagonal/off-diagonal cross-level correction in [`SPEC-2026-07-19-16`](A12-DMKITS-SPEC-SYNC.md#spec-2026-07-19-16---correlation-captures-every-named-outer-repetition-level).
- Flat partial validation: `IDocumentRtService.validatePart` and `IValidator.validatePartlyWith3ValueLogic` state the public error-field gate and out-of-set three-valued contract; `ValidationCommand` constructs the route; `CodeGenTeilValidierung` dispatches by relevant error field; [`RuntimeController.java`](../../a12-kernel/kernel-rt/kernel-core-runtime/src/main/java/com/mgmtp/a12/kernel/core/rt/_30_8/internal/core/RuntimeController.java) and `EntityUtil` own non-relevant reads and membership.
- Computation alternatives and generated validation: [`ValidationModelConverter.java`](../../a12-kernel/kernel-md/kernel-md-model/src/main/java/com/mgmtp/a12/kernel/md/model/internal/service/ValidationModelConverter.java) creates both the calculation and ordinary ERROR rule at the target; [`ComputationAlternativesJoiner.java`](../../a12-kernel/kernel-md/kernel-md-model/src/main/java/com/mgmtp/a12/kernel/md/model/internal/service/utils/ComputationAlternativesJoiner.java) retains every guarded strict-or-tolerance mismatch; the Java, Groovy, and JavaScript [`CodeGenCalculationAlternative.st`](../../a12-kernel/kernel-tool/kernel-core-codegen/src/main/resources/internal/templates/validation/java/calcDir/CodeGenCalculationAlternative.st) variants instead stop computation after the first selected operation. This is the source route for the first-match/all-alternatives non-law.

TS mirror (identical, German-named files): `kernel-rt/kernel-core-runtime-ts/src/main/js/internal/`. String↔typed conversion (date formats, decimal parsing, partially-known dates): `kernel-conversion-java/.../a12internal/util/`.

**Codegen (names every operator, emits calls — not semantics):** `kernel-tool/kernel-core-codegen-condition/.../generator/backingbeans/{conditions,fieldoperations}/` + `internal/utils/RuntimeMethodNameFactory.java`. For the narrow filter boundary, [`CodeGenFilterCondition.st`](../../a12-kernel/kernel-tool/kernel-core-codegen-condition/src/main/resources/internal/templates/validation/java/conditionsDir/CodeGenFilterCondition.st), [`CodeGenCondition.java`](../../a12-kernel/kernel-tool/kernel-core-codegen-condition/src/main/java/com/mgmtp/a12/kernel/core/codegen/internal/generator/backingbeans/conditions/CodeGenCondition.java), and [`ConditionSuffix.st`](../../a12-kernel/kernel-tool/kernel-core-codegen-condition/src/main/resources/internal/templates/validation/java/conditionsDir/ConditionSuffix.st) show that a generated `Having` predicate keeps only a known true condition result. **Parser/grammar:** `kernel-tool/kernel-core-parser` (ANTLR `.g4` + bilingual `LexerTerminals_{en,de}.java`); `CheckKonstanteImpl`, `CheckOperationImpl`, and `CheckVergleichsBedingungImpl` own numeric scale/constant-expandability checking. `CheckOperationImpl` also establishes the plain multiplication/division-region and direct-left-power checks; its listener's exception-based reset makes traversal through operation-valued wrappers non-compositional, so that larger surface remains observation-gated. `parsebaum/util/DivisionTransformer.java`, `ICompositeOperation.traverse`, and codegen `CompositeOperation` together establish one order-sensitive post-order pass over original multiplication nodes and left-folded evaluation: ordinary factors precede extracted numerators, newly constructed products are not revisited, and braces do not block a root-division factor. **Model/document API:** `kernel-md/*`.

**Docs:** [`documentation/_merged/kernel-ba.md`](../../a12-kernel/documentation/_merged/kernel-ba.md) — the definitive evaluation-behaviour spec (**read first**); [`documentation/_merged/kernel-dev.md`](../../a12-kernel/documentation/_merged/kernel-dev.md) — custom conditions/types, Document API; [`ANALYSIS.md`](../../a12-kernel/ANALYSIS.md) — architecture; [`KERNEL-GRAMMAR.md`](../../a12-kernel/KERNEL-GRAMMAR.md), [`KERNEL-USE.md`](../../a12-kernel/KERNEL-USE.md).

### `adapter/` — the kernel-as-oracle differential (JVM, kernel-linked)

- `adapter/src/test/java/…/laws/RuntimeLaws.java` — **the oracle**: builds a model, injects a rule, runs the real kernel over a document, reports fired `code|type|pointer`.
- `adapter/src/test/kotlin/…/laws/*DiffTest.kt` — ~180 interpreter↔kernel differentials, one semantic case each; canonical pattern in `InterpreterDiffTest.kt`.

- `adapter/src/test/kotlin/…/laws/{CorpusEngines,CorpusReplay}.kt` — the corpus reference runner.

How it works: the kernel is a **black box**; the same document is materialized into every engine and outcomes are compared as signature multisets. For one observed case, empirical kernel correspondence for the a12-dmkits interpreter or this project means agreement with that oracle at the explicitly retained projection. It does not define internal Lean correctness or establish universal kernel equivalence.

## What the coverage tells us

- **Machine-readable facts exist for only 5 areas** (§2, §5, §6, §9, §12). For those, lean on the catalog `semantics` facets and their runnable probes; for the other nine, the available drill path is prose, findings, focused tests, and the relevant kernel class. - **The sibling replayable corpus covers only 7 areas** (§2, §5, §6, §7, §9, §11, §12) and is thin. The corpus is useful kernel-derived source evidence, not a live dependency or substitute oracle; broader focused differentials live in `adapter/laws` (JVM, kernel-linked, not portable to Lean). Expanding local coverage means externally capturing the relevant case, retaining a portable own-repository observation, and adding only the typed projection needed by the Lean capsule.
- **The hardest, test-densest areas are §6, §9, §11, §12** — the same ones our [`spec/SEMANTICS-MAP.md`](../spec/SEMANTICS-MAP.md) rates ★★★★★. Encoding risk and proof payoff concentrate there.
